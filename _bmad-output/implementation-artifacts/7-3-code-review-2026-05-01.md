## Top-Line Verdict: **APPROVE-WITH-CHANGES**

**Findings:** 0 BLOCKING / 4 SHOULD-FIX / 6 NICE-TO-HAVE

The single-switch wiring is correct at the textual level: `BrokerMode` is inspected exactly once outside the dispatch boundary (the log-label use in `emit_startup_banner`), and the binary's `--paper` path constructs a paper-specific orchestrator without leaking the enum further. The TOML loader is small, well-tested, and includes a fail-closed default (`Live`) that an operator who omits the `mode` key cannot silently slide into paper. Test coverage of the deserialisation path is solid (typo rejection, snake_case enforcement, shipped configs round-trip).

Two structural concerns outweigh the wiring win:

1. **The `MockBrokerAdapter` is decorative on the order path** (S-1). Tasks 2.2 and 3.4 explicitly say "orders submitted to MockBrokerAdapter via the same `BrokerAdapter::submit_order()` interface" and AC1 says "orders routed to MockBrokerAdapter instead of real OrderPlant." The implementation routes orders straight from the engine→broker SPSC into `MockFillSimulator::simulate` and skips the `BrokerAdapter` trait entirely on the order/fill path. The adapter is held by the orchestrator and exercised only for `subscribe`/`cancel`/`query` (the read-side surface). This is the same shortcut the replay orchestrator takes (so paper inherits the precedent) but it is a real deviation from the story's task wording AND it shifts the "live vs paper differs only at the BrokerAdapter" claim from "single trait-object swap" to "two parallel orchestrators with no shared order plumbing."

2. **AC2's "all circuit breakers / risk / regime function normally" is unverified by the orchestrator** (S-2). The `PaperTradingOrchestrator` does NOT own a `CircuitBreakers`, a `SignalPipeline`, a `RegimeDetector`, an `EventWindowManager`, or a `DataQualityGate`. It calls `apply_market_event` directly on a bare `OrderBook`, no signals fired, no regime detected, no fee-gate evaluated. The integration test `circuit_breakers_trip_identically_in_paper_mode` constructs `CircuitBreakers` manually and feeds it manually — proving the breaker code works in isolation, NOT proving the orchestrator integrates it. The dev's claim "the orchestrator does not branch on `BrokerMode`, so the regime detector receives the same SPSC stream regardless of mode" is sleight of hand: the orchestrator never CONSTRUCTS or RUNS a regime detector at all — not branching on a value you never look at is trivially true. The AC reads "function normally"; in reality the orchestrator runs none of these subsystems, so no AC verification is possible against the orchestrator as shipped.

The remaining issues are softer:

- **Live data wiring is genuinely deferrable to Epic 8 only at the connection layer** (S-3). The `MarketDataFeed` trait is the right shape for an orchestrator-level seam, but its docstring conflates "end of stream" and "no event right now" — for a real Rithmic stream this means the FIRST momentary quiet period terminates `pump_until_idle`. The dev's claim that "the only change required [for Epic 8] is providing a `RithmicMarketDataFeed: MarketDataFeed` impl and swapping the `let feed = ` line" is therefore false: the trait semantics also need to grow a separate "EOF vs idle" signal.
- **The shipped `engine --paper --config config/paper.toml` binary subscribes to no symbols, attaches no journal, and consumes zero events** (S-4). It is a smoke runner, not a deployable paper-trading process. The dev's "Engine binary smoke test runs cleanly" is true; an operator reading that as "paper trading is wired in production" would be misled.

Probe answers (per review brief):

- **Live data wiring deferral.** PARTIALLY DEFENSIBLE. The Rithmic socket itself genuinely belongs to Epic 8 (config schema, credentials, connection lifecycle). But the bridge (`crates/broker/src/market_data.rs::MarketDataStream` already exists from Story 2.1) could ship now as a sync-side adapter implementing `MarketDataFeed`. That ~30-line bridge would let the binary at least *attempt* live data and surface the missing-config error helpfully, instead of running a zero-event smoke loop. The `MarketDataFeed` trait's "EOF vs idle" conflation (S-3) is the bigger blocker — that needs to be resolved before any production data feed can be wired in cleanly.
- **`BrokerMode` branches scattered around.** NO. `grep -r BrokerMode crates/` shows the enum at 6 sites: definition, two re-exports, one `BrokerMode != Paper` guard in `main.rs::run_paper`, one struct field + check in `startup.rs`, and one `BrokerMode::Paper.as_str()` log-label use in `paper/orchestrator.rs`. All non-text-label uses are at the startup boundary. AC3's "single config parameter" is honored at the textual level. The architectural concern is the parallel-orchestrator pattern, not the enum surface (covered by S-1/S-2).
- **`PaperTradingOrchestrator` reuses `event_loop.rs`?** NO. It has its own `pump_until_idle` / `tick` loop, structurally identical to `ReplayOrchestrator::run`. Both call `apply_market_event` directly. The production `EventLoop` (`crates/engine/src/event_loop.rs`, 1174 LoC) does substantially more (sequence-gap detection, stale-data detection, data-quality gate, circuit breakers, anomaly detection, event windows). None of that runs in paper mode either. This is the SAME architectural concern flagged in Story 7.1's review for replay; the verdict here should be consistent with the verdict there. If 7.1 accepted the parallel pattern, 7.3 inherits that acceptance — but the AC2 implications (no risk/regime wired) are MORE severe in paper mode than in replay because paper is the dress rehearsal for live.
- **Story 7.4 hook cleanly extensible?** PARTIALLY. The journal channel IS exposed (`attach_journal` accepts a `JournalSender`, and the integration test demonstrates `TradeEventRecord` round-trips). But the orchestrator's `drain_fills` only INCREMENTS A COUNTER — it does NOT emit any `TradeEventRecord` to the journal today. For 7.4 to record paper fills with `source: paper`, it WILL need to modify the orchestrator (add the journal-emit call into `drain_fills` or `drain_orders_and_simulate_fills`). The dev's claim "no orchestrator changes required" is wrong; the more accurate claim is "the channel scaffolding is in place; 7.4 layers the emission on top." That's still a clean extension surface, but the dev's framing oversells it.

### SHOULD-FIX

#### S-1 — `MockBrokerAdapter::submit_order` is never invoked on the paper-mode order path; AC1 + Task 3.4 specifically require this

**Source:** Acceptance Auditor + Edge Case Hunter
**Location:** `crates/engine/src/paper/orchestrator.rs:408-415` (`drain_orders_and_simulate_fills`), Task 3.4 in `_bmad-output/implementation-artifacts/7-3-paper-trading-mode.md`

The orchestrator drains the engine→broker SPSC and feeds each order DIRECTLY into `MockFillSimulator::simulate(...)`:

```rust
pub fn drain_orders_and_simulate_fills(&mut self) {
    while let Some(order) = self.order_consumer.try_pop() {
        let order: OrderEvent = order;
        self.orders_processed += 1;
        self.fill_sim
            .simulate(&order, &self.book, &mut self.fill_producer);
    }
}
```

The `MockBrokerAdapter` field (`self.broker`) is held by the orchestrator for `subscribe`/`cancel`/`query` (the read-side surface), but `BrokerAdapter::submit_order` is never called on the order path. AC1 says "orders routed to MockBrokerAdapter instead of real OrderPlant" and Task 3.4 says "the order is submitted to MockBrokerAdapter via the same `BrokerAdapter::submit_order()` interface."

Verification: `grep -n "submit_order" crates/engine/src/paper/orchestrator.rs` returns NOTHING; `grep -n "submit_order" crates/engine/tests/paper_trading.rs` also returns NOTHING. The integration test `paper_mode_end_to_end_order_to_fill` injects an `OrderEvent` via `order_producer_mut().try_push(order)` and then calls `drain_orders_and_simulate_fills` — bypassing the BrokerAdapter trait entirely. The same test's assertion (`fill.fill_price.raw() == 18_005`) verifies the FILL SIMULATOR's behaviour, not the BROKER ADAPTER's.

**Impact:**
- Tasks 2.2 / 3.4 / 7.4 / 7.5 / 8.3 are all about "orders go through `MockBrokerAdapter`" — the ACTUAL behavior is "orders go through `MockFillSimulator`." The `MockBrokerAdapter` is ornamental on the order path.
- Production wiring in Epic 8 will swap `MockBrokerAdapter` for `RithmicAdapter`, but the paper orchestrator NEVER uses the trait surface that `RithmicAdapter` implements for orders. So the "swap a trait object at startup" architectural promise (AC3) does NOT actually prove that the paper code-path matches the live code-path for orders — it proves the ADAPTER FIELD is interchangeable, but the field is unused on the hot path.
- The replay orchestrator takes the same shortcut (also routes order→fill via `MockFillSimulator`, also holds an unused `MockBrokerAdapter`). So paper inherits the precedent — but in paper this is more visible because AC1 specifically names the BrokerAdapter trait.

**Fix options:**
1. **Honor the AC literally:** route the SPSC order drain through `BrokerAdapter::submit_order` (which is async — would require a tokio runtime in the orchestrator OR a sync wrapper). The `MockBrokerAdapter::submit_order` writes to `submitted_orders` and returns the order id; `MockFillSimulator::simulate` would then be called to synthesize the fill that the real Rithmic OrderPlant would have returned. This makes the BrokerAdapter the load-bearing seam.
2. **Update the AC text:** acknowledge the actual design — "orders routed to `MockFillSimulator` (which represents the broker-side fill behavior); the `BrokerAdapter` trait surface is exercised for connect / subscribe / cancel / query but not for order submission in the paper-mode loop." This needs explicit alignment with the architecture document and Story 7.4's source-tagging plan.
3. **Acknowledge the inherited replay precedent:** if 7.1's review accepted the same shortcut for replay, document the consistency in 7.3's dev notes and the architecture document. The honest framing is "in V1, fill production happens in `MockFillSimulator` for both replay and paper; the `BrokerAdapter` trait surface for `submit_order` is exercised only by Epic 8's live wiring."

Option 3 is probably the right V1 answer; Option 1 is the long-term shape. Either way, **the dev notes need to be honest about the gap** — currently they imply the BrokerAdapter trait is the load-bearing seam.

#### S-3 — `MarketDataFeed::next_event` returning `None` conflates "end of stream" with "no event right now"; `pump_until_idle` exits on the first quiet moment of any production stream

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/paper/data_feed.rs:23-37` (trait + docstring), `crates/engine/src/paper/orchestrator.rs:340-373` (`pump_until_idle`)

The `MarketDataFeed::next_event` docstring:

> Pull the next event, or `None` if no more events are available right now. End-of-stream and "currently empty" share this signal — callers decide their own termination policy.

`pump_until_idle` consumes this trait verbatim:

```rust
while let Some(event) = self.feed.next_event() {
    ...
}
// Final drain and return summary.
```

For a `VecMarketDataFeed`, "no more events available" IS terminal. For a real Rithmic stream (Epic 8's `RithmicMarketDataFeed`), `next_event() == None` would mean "no message has arrived since the last poll" — a TRANSIENT state that `pump_until_idle` would treat as terminal, exiting the pump on the first quiet microsecond.

The dev notes claim: "When Epic 8's startup sequence lands the Rithmic socket, the only change required is providing a `RithmicMarketDataFeed: MarketDataFeed` impl and swapping the `let feed = ` line in `main.rs::run_paper`." This is FALSE — the trait semantics also need to grow a separate "EOF vs idle" signal, OR the orchestrator needs a separate termination policy (e.g., a shutdown signal, an idle timeout, an explicit `is_terminated()` method on the feed).

**Impact:**
- Epic 8 cannot wire Rithmic into this trait without a change. The dev's "one-line swap" claim is wrong.
- `tick()` has the same issue: `if let Some(event) = self.feed.next_event()` — fine for `tick` because callers loop externally, but the IDLE signal still terminates `pump_until_idle`.

**Fix options:**
1. **Split the trait:** `next_event(&mut self) -> FeedEvent` where `FeedEvent` is `enum { Event(MarketEvent), Idle, Terminated }`. `pump_until_idle` exits on `Terminated`, spins / yields on `Idle`.
2. **Add a separate `is_terminated(&self) -> bool` method.** `pump_until_idle` checks it after each `None`.
3. **Document the actual contract:** rename the trait method to `try_next_event` and rename `pump_until_idle` to `pump_until_feed_exhausted_or_idle`. This is honest about the V1 semantics but doesn't fix Epic 8's wiring problem.

Option 1 is the cleanest. Option 2 is the smallest delta. Option 3 is a docstring fix that preserves the wiring problem.

#### S-4 — `engine --paper --config config/paper.toml` ships as a smoke runner: zero events, no journal, no subscriptions

**Source:** Edge Case Hunter + Acceptance Auditor
**Location:** `crates/engine/src/main.rs:154-211` (`run_paper`)

The shipped binary's `run_paper` function does the following:
1. Loads the config and verifies `mode == Paper`.
2. Calls `sanity_check_paper_credentials`.
3. Constructs `let feed = VecMarketDataFeed::new(Vec::new())` — **zero events.**
4. Constructs `PaperTradingOrchestrator` with **`PaperTradingConfig::default()`** — empty subscriptions list, no journal.
5. Calls `orch.emit_startup_banner()` and `orch.pump_until_idle()`.
6. Logs a summary that will report 0/0/0/0.
7. The terminal `if summary.events_consumed == 0` warn block fires every single time, surfacing "V1 ships without a live data feed wired in."

So the shipped binary CANNOT be used for paper trading. It is a smoke runner that proves the dispatch path works. The dev notes claim "Engine binary smoke test runs cleanly" — true. An operator reading "Story 7.3: Paper Trading Mode — Status: done" would be misled into thinking they can run the binary against live markets; they cannot.

**Impact:**
- Story 7.4's plan ("paper trade results recorded with same fidelity as live trades") cannot be exercised end-to-end via the binary because the binary produces no trades.
- Any external validation of paper mode requires a custom test harness; the shipped binary is unusable for production paper trading.

**Fix options:**
1. **Wire `subscribe_all` into `run_paper`:** read symbols from config (a Task-1.1 follow-up — symbols would also need to be added to the `[trading]` section of the TOML), call `subscribe_all().await`. Requires a tokio runtime in the binary (currently it has `tokio` in dev-deps only — would need to promote).
2. **Wire a journal into `run_paper`:** construct an `EventJournal::channel()`, attach to orchestrator, spawn a worker. Story 7.4 will need this anyway.
3. **At minimum, document the scope:** rename the warn message to "PAPER TRADING MODE BINARY IS SCAFFOLDING — NO LIVE DATA, NO JOURNAL, NO SUBSCRIPTIONS. DO NOT USE IN PRODUCTION." Or refuse to start unless either a non-empty feed is provided or an explicit `--smoke` flag is passed.

Option 3 is the honest V1 path. Option 1 + 2 are real Epic-8 prerequisites — defer with explicit tracking.

#### N-3 — `paper_mode_uses_system_clock_replay_uses_sim_clock` does not actually exercise `ReplayOrchestrator`'s clock selection

**Source:** Edge Case Hunter
**Location:** `crates/engine/tests/paper_trading.rs:159-182`

The test name promises a comparison between paper and replay clock semantics. The implementation:

```rust
// Replay: SimClock starts at 0 and only advances when set explicitly.
// We don't construct a full ReplayOrchestrator here ...
let sim = futures_bmad_testkit::SimClock::new(0);
let s1 = sim.now().as_nanos();
```

2. Rename the test to `paper_mode_uses_system_clock_and_sim_clock_does_not_advance` so the name matches what's asserted.

Option 2 is the cheap honesty fix.

#### N-4 — `emit_startup_banner` is `pub` and stateful (emits a journal record); calling twice produces duplicate `paper_mode_start` records

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/paper/orchestrator.rs:230-253`

The function emits a `JournalEvent::SystemEvent(rec)` to the attached journal each time it's called. Idempotency is not documented; calling it twice (e.g. an over-eager test or a restart-after-error path) writes two `paper_mode_start` records, which Story 7.4's source-tag query would see as two session boundaries.

Also, the function reads `self.clock.now()` even when `self.journal.is_none()` — a wasted system call. Trivially cheap (`SystemClock::now` is nanos), but the early-return-when-no-journal is missing.

**Fix:** Either (a) document the idempotency contract ("call once at startup; calling twice is a bug") and add a `debug_assert!(!self.banner_emitted)` guard, or (b) make the function idempotent with a `banner_emitted: bool` field. Move the `clock.now()` call inside the `if let Some(journal)` block.

#### N-5 — `sanity_check_paper_credentials` reads `RITHMIC_ACCOUNT_ID` from process env; no test covers the warning branch, and any test that did would race in parallel with sibling tests

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/startup.rs:84-103`

```rust
if std::env::var("RITHMIC_ACCOUNT_ID")
    .ok()
    .filter(|v| !v.is_empty())
    .is_some()
{
    warn!(target: "paper", ...);
}
```

The function reads process env directly. Tests that set/unset env vars are racy when run in parallel with other tests in the same process (and `cargo test` runs tests in parallel by default). So the warning path is effectively untestable without serializing all env-touching tests via a global mutex.

The function only WARNs (doesn't error), so this is low-risk — but a future change that escalates the warning to an error (sensible if the operator clearly intended live but flipped to paper) would silently regress without a test gate.

**Fix options:**
1. **Take env-var name as a parameter** (`sanity_check_paper_credentials(mode: BrokerMode, env_var_name: &str)`) so tests can pass a unique var per test. Production callers pass `"RITHMIC_ACCOUNT_ID"`.
2. **Take a `Fn() -> Option<String>` env reader** so tests can inject a mock without touching process env.
3. **Document the untested-warning-branch** explicitly and accept the gap.

Option 2 is the cleanest. Option 3 is the cheapest.

#### N-6 — Duplicate `shipped_*_config_*` tests in `startup.rs::tests` and `tests/paper_trading.rs`; both rely on workspace-relative paths via `CARGO_MANIFEST_DIR`

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/startup.rs:175-198` and `crates/engine/tests/paper_trading.rs:218-227`

Two tests cover the same property: "the shipped `config/paper.toml` parses to `BrokerMode::Paper`." The unit test in `startup.rs::tests::shipped_paper_config_parses_to_paper_mode` does it via `manifest.parent().unwrap().parent().unwrap()`; the integration test in `paper_trading.rs::shipped_paper_config_routes_to_paper_mode` does it via `format!("{}/../../config/paper.toml", env!("CARGO_MANIFEST_DIR"))`. Both fail if the workspace layout changes.

The integration-test version is also weirdly worded ("routes to paper mode") but only asserts `cfg.broker.mode == BrokerMode::Paper` — same property as the unit test. It does not verify any actual ROUTING.

**Fix:** Drop the duplicate from `tests/paper_trading.rs` (the unit test in `startup.rs::tests` covers the property) OR rename the integration test to `shipped_paper_config_round_trips_through_loader` so the name matches what's asserted.

## Scope Decision Verdicts

### 1. Live data wiring deferred to Epic 8 (`MarketDataFeed` trait + `VecMarketDataFeed` test double, no Rithmic bridge)

**Dev claim (Completion Notes):** "When Epic 8's startup sequence lands the Rithmic socket, the only change required is providing a `RithmicMarketDataFeed: MarketDataFeed` impl and swapping the `let feed = VecMarketDataFeed::new(Vec::new())` line in `main.rs::run_paper`."

**Verdict: PARTIALLY DEFENSIBLE.** The seam at the orchestrator level is the right shape (a trait the orchestrator owns, with an in-memory test double for V1). But:
1. The trait semantics conflate EOF and IDLE (S-3) — a real Rithmic stream would terminate `pump_until_idle` on the first quiet moment. The trait MUST grow a separate termination signal before Epic 8 can wire in cleanly.
2. The `MarketDataStream` adapter from Story 2.1 (`crates/broker/src/market_data.rs`) already exists; a sync wrapper implementing `MarketDataFeed` could ship now (~30 LoC) to demonstrate the seam. Deferring the bridge entirely to Epic 8 is a defensible scope call but it leaves the binary as a smoke runner (S-4).

The "one-line swap" claim is wrong. Update the dev notes accordingly.

### 2. `MockBrokerAdapter` held by orchestrator but `submit_order` never called on the order path

**Dev claim:** Implicit in "the same SPSC market / order / fill queues live trading uses" — the dev presents the architecture as "swap a BrokerAdapter at startup, downstream code is mode-agnostic."

**Verdict: VIOLATES THE STORY'S TASK WORDING — see S-1.** Tasks 2.2 / 3.4 / 7.4 / 7.5 / 8.3 all reference "MockBrokerAdapter" or "the same `BrokerAdapter::submit_order()` interface." The implementation routes orders through `MockFillSimulator` and the BrokerAdapter trait surface is exercised only for `subscribe`/`cancel`/`query`. This is the same shortcut the replay orchestrator takes (so paper inherits the precedent), but in paper this is more visible because AC1 specifically names the BrokerAdapter trait.

The honest framing: "in V1, the BrokerAdapter trait is the seam for connect / subscribe / cancel / query but NOT for order submission; order submission uses the SPSC + `MockFillSimulator` directly. Epic 8 will wire the BrokerAdapter onto the order path for live mode, and 7.4 will source-tag fills there." Update dev notes / task annotations to reflect this.

### 3. `BrokerMode::default() == Live` (fail-closed for the loader)

**Dev claim (Completion Notes):** "a `Default` of `Live` so a config that omits `[broker] mode` fails closed (the `--live` CLI flag itself remains the second gate)."

**Verdict: CORRECT and well-motivated.** A user who writes `[broker]\n` (no `mode` key) and runs `--paper --config thatfile` gets:
- `cfg.broker.mode == BrokerMode::Live` (default)
- `cfg.broker.mode != BrokerMode::Paper` → `--paper` exits with code 2 and an error message.

So the default cannot route real orders by accident. The combination of "default Live + explicit `--live` CLI flag for live + cfg-must-match-CLI guard for paper" is solid defense-in-depth.

### 4. `PaperTradingOrchestrator` does NOT use the production `EventLoop` — own pump loop instead

**Dev claim:** Implicit (the orchestrator just has `pump_until_idle` / `tick`).

**Verdict: INHERITED FROM REPLAY (7.1's review verdict).** The replay orchestrator took the same shortcut. If 7.1's review accepted the parallel pattern, 7.3 is consistent. But the AC2 implications (no risk/regime wired) are MORE severe in paper mode because paper is the dress rehearsal for live, not a historical-data analysis run. **See S-2** — at minimum the dev notes should be honest about which production subsystems are NOT integrated.

### 5. Orchestrator never emits `TradeEvent`s for fills — only `paper_mode_start` system event at startup

**Dev claim (Completion Notes):** "`attach_journal` accepts a `JournalSender`; the integration test demonstrates that a `TradeEventRecord` already round-trips through the same channel. 7.4 will add a `source` field (paper/live) on the record itself — no orchestrator changes required."

**Verdict: PARTIALLY MISLEADING.** The channel IS exposed and IS clean to extend. But the orchestrator never emits any `TradeEventRecord` today — `drain_fills` only increments a counter. Story 7.4 will need to MODIFY the orchestrator (add the journal-emit call into `drain_fills` or `drain_orders_and_simulate_fills`). The honest claim is "the channel scaffolding is in place; 7.4 layers the emission on top in `drain_fills`." That's still a clean extension; the dev's framing oversells the readiness.

Also note: `TradeEventRecord` does not currently carry a `source` field; 7.4's plan (Task 2.1 of `7-4-paper-trade-result-recording.md`) is to add a `TradeSource` enum to `EngineEvent` variants — that change is in the journal crate, not the paper orchestrator. Both changes will need to land together for 7.4 to function.

### 6. `attach_journal` takes `JournalSender` (consuming) and is one-shot

**Verdict: ACCEPTABLE FOR V1.** The orchestrator owns the sender after attach. No method to detach or replace. For V1 this is fine (a paper session has one journal). Future restart / failover scenarios may want a `set_journal(Option<JournalSender>)` or builder pattern.

### 7. `pump_until_idle` is single-threaded; SPSC is used in a single-threaded loop with push-then-drain

**Verdict: ARCHITECTURALLY SUSPICIOUS BUT FUNCTIONAL.** The SPSC's purpose is multi-threaded decoupling. Using it in a single-threaded loop adds zero value (could be a `Vec` or a function call) and PREVENTS multi-threaded use later — if a production `EventLoop` is ever wired onto a separate consumer thread, the orchestrator's own `drain_market_consumer` would race with it. The dev's intent is presumably "match the live trading SPSC topology so the paper code path mirrors live" — fine, but the orchestrator should NOT also drain the consumer side. Either it's a producer (matches live) or it's a single-threaded smoke runner; trying to be both is incoherent.

For V1, the smoke-runner shape is acceptable; for real paper trading, the orchestrator should be a producer and the engine event loop should be the consumer (on a separate thread). This is Epic 8's wiring, not a 7.3 blocker.

### 8. `event` is `MarketEvent` (Copy) so the double-`try_push` after a failed first push is sound

**Verdict: CORRECT.** Verified via `crates/core/src/events/market.rs:11`: `MarketEvent` derives `Copy`. The pattern `try_push(event)` followed by `try_push(event)` works — `event` is copied each time. Not a bug.
