## Challenges & Growth Areas

### Live-Trading Exit Gates (3 SHOULD-FIX items, all explicit ship-stoppers)

These three items are the live-trading gate. **Epic 8.2 startup sequence MUST verify all three closed before enabling live trading.**

1. **4-5 S-2 — `trading_halted` unwired.** `PositionTracker` sets `trading_halted` on reconciliation mismatch, but `OrderManager::submit_order` never consults it. AC3's "trading halted" effect is structurally absent. **Resolution:** wire into Story 5.1 CircuitBreaker framework as the canonical halt source.
2. **4-5 S-3 (carryforward from 4-3 S-2 / 4-4 S-2) — partial-entry SL/TP oversizing.** `BracketManager::on_entry_fill` warn-logs and proceeds with original SL/TP quantity on partial entries. Buy-5 partial-filled-3 → SL fires at 5 → flatten 3 + naked short 2 = NFR16 violation. **Resolution:** panic-mode escalation in Story 5.3.
3. **4-5 S-1 — in-flight-fill reconciliation race.** `reconcile(&[BrokerPosition])` has no as-of-T0 sequence number or quiesce hook. Periodic reconciliation during live trading WILL trip on this race during fill bursts. **Resolution:** quiesce/as-of-T0 implementation in Story 8.2 alongside periodic-reconciliation wire-up.

### Recurring Review Themes

- **Journal completeness.** Three stories (4-1, 4-2, 4-4) had findings about state transitions not being journaled (B-1 ate events; auto-upgrade arcs missed). All resolved in `366fcf7`. Lesson: every state transition must be journaled or forensic replay is broken.
- **Backpressure handling.** Three stories (4-2, 4-3, 4-4) had findings on non-blocking I/O paths swallowing failures (`try_push` return ignored, channel-full on flatten retries). Lesson: non-blocking I/O must have explicit error handlers and observability hooks.
- **Spec ambiguity → implementation variance.** 4-2 (`OrderSubmitter` vs `BrokerAdapter::submit_order`), 4-3 (`BracketOrder.entry: OrderEvent` vs `OrderParams`), 4-5 (bracket P&L vs position P&L). Architecture doc was authoritative; conflicting story-spec text caused reviewable variance. Lesson: flag spec-vs-architecture conflicts upfront in dev notes, not at review.

## Key Insights & Learnings

1. **Carryover-fix passes are a virtuous design pattern.** Batching SHOULD-FIX items from prior stories into the next story's PR (when scope-aligned) keeps quality debt from accumulating without spinning dedicated cleanup stories. 4-5's `366fcf7` cleared six items in one commit. Make this an expected behavior, not an accidental one.

2. **Seam-first deferral keeps epics shippable.** When the live-broker dependency was unresolvable mid-epic, the team installed clean traits and stub implementations rather than blocking the epic. The OrderPlant integration story will be a swap, not a rewrite. Architecture stays intact.

3. **Exhaustive matching beats wildcard `_`.** 4-4's state machine compile-time-rejects unknown state/trigger pairs. The cost (verbose match arms) was worth it: 4-5's reconciliation breaker integration just plugged in without new defects.

4. **Plateau ≠ regression.** Findings counts didn't drop to zero like Epic 3's tail. Reasonable: persistence + async + broker IO + state machines is a richer surface than pure signal computation. Quality is acceptable at ~10 findings/story when the carryover-fix cadence is tight.

5. **Live-trading exit gates need to be one explicit checklist, not scattered SHOULD-FIX items.** The Epic 4 conversation surfaced 3 items that read as "shippable per code review" but are "do-not-enable-live-trading" items. Story 8.2 must own this checklist as a startup gate.
