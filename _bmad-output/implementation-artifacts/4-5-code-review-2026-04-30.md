## Findings

### SHOULD-FIX

#### S-1: In-flight-fill race during periodic reconciliation produces false-positive mismatches (architectural)
**Where:** `crates/engine/src/order_manager/tracker.rs:255-316` (`reconcile`)
**What:** The reconciliation algorithm compares the local view at time T against a broker view fetched at some earlier T0 (`query_positions` is async). Between T0 and T, new fills can arrive at the engine and update local state. Result: the local view shows +1 fill the broker view doesn't yet reflect → reconcile reports a `SideOrQuantity` mismatch → `trading_halted = true`, breaker tripped.

This is the "false positive from in-flight fills" failure mode the architecture spec already enumerates. The current API has no way to express "as-of T0" or to quiesce in-flight fills, so consumers in 8-2/8-4 cannot mitigate this without adding a sequence-number / timestamp seam back into the tracker.

**Why SHOULD-FIX:** The startup case is safe (no fills before trading opens). The reconnection case is mostly safe (broker reconnect FSM can quiesce). The 60s **periodic** case during live trading WILL trip on this race during any active fill burst.

**Recommendation:** Either (a) tag broker queries with the local last-fill timestamp; on reconcile, if any fill applied since T0 touches a mismatched symbol, **retry** rather than trip the breaker; or (b) provide a `quiesce()` hook on the tracker that the lifecycle module drives before each periodic reconciliation. Either way the API surface needs to grow before 8-2 ships periodic reconciliation against this. (Tracking note: the deferred-work entry for periodic-reconciliation triggers SHOULD include this race handling as part of the same story.)

#### S-2: `trading_halted` is set but no submission gate consults it
**Where:** `crates/engine/src/order_manager/tracker.rs:380-381`; `crates/engine/src/order_manager/mod.rs:306-399` (`OrderManager::submit_order`)
**What:** On mismatch, `tracker.trading_halted = true`. AC3 says "trading halted" — but `OrderManager::submit_order` checks only `self.submissions_paused` (set by Uncertain-order timeouts, NOT by reconciliation mismatch). A grep for `trading_halted` in `crates/engine/src/` returns one file: `tracker.rs`. There is no plumbing from the tracker's halt-flag into the submission gate.

**Why SHOULD-FIX:** A mismatch trips the circuit breaker callback (registered externally) and sets `trading_halted` — but if the OrderManager doesn't consult either signal, new orders submitted on the same engine tick after the mismatch will still go to the broker. The breaker callback is the operator's escape hatch, not a structural gate.

**Recommendation:** Either (a) wire `submit_order` to consult `tracker.trading_halted()` (adds a `&PositionTracker` parameter or a shared `Arc<AtomicBool>`), or (b) route the mismatch through the same `submissions_paused` gate the Uncertain-state path uses. Document the gating contract clearly so 8-2 doesn't have to re-derive it. The tracker's `with_circuit_breaker` is a callback, not a gate — they are not the same thing.

#### S-3: Partial-entry SL/TP oversizing remains a position-safety hole (deferral contested)
**Where:** `crates/engine/src/order_manager/bracket.rs:325-334`
**What:** On `FillType::Partial` for an entry leg, `BracketManager::on_entry_fill` warn-logs and submits SL/TP at the **original** quantity, not the partial-fill quantity. If a Buy 5 entry partial-fills 3 then SL fires at 5 contracts, the result is a flatten of the 3 long + a naked **short 2** at the SL trigger. That is exactly the position-safety failure NFR16 forbids.

**Why SHOULD-FIX (not BLOCKING):** This branch does add a `PositionTracker` and a `CircuitBreakerCallback`, so a panic-mode escalation on partial-entry detection is a small fix (~20-30 LoC: in the `FillType::Partial` arm, fire the breaker and engage flatten on the partial position rather than submit SL at original size). The dev's defense — "lives inside BracketManager, not the tracker" — is technically correct on file boundary, but evades the spec wording: 4-5 was earmarked as the live-trading gate.

**Recommendation:** Add a panic-mode trigger in the partial-entry branch BEFORE 8-2 wires the engine event loop to a live broker. Alternative: clearly mark in the engine init code that "live trading is disabled until partial-entry handling lands" so an operator cannot accidentally enable it.

#### S-4: Reconciliation `avg_entry_price` equality is strict — 1-q-tick rounding drift will trip false mismatches
**Where:** `crates/engine/src/order_manager/tracker.rs:296-306`
**What:** The mismatch check uses `l.avg_entry_price != b.avg_entry_price` — strict integer equality. Local weighted-avg computation uses `total_cost.raw() / new_qty as i64` (truncating integer division, position.rs:80). A broker may use a different rounding policy (banker's, round-half-up, last-fill-replaces, etc.), and after several additions to a position the two avg-entry values can drift by 1 quarter-tick without any actual error. Result: PhantomLocal-style mismatch → trading halted, breaker tripped.

**Why SHOULD-FIX:** This is independent of the in-flight-fill race; it manifests under purely static reconciliation. The `reconciliation_avg_entry_disagreement_detected` test verifies the strict-equality semantic — i.e., it codifies a behavior that will produce false positives in production.

**Recommendation:** Allow a configurable tolerance band (default: 1 quarter-tick per fill, capped at some sane max) on `avg_entry_price` comparison, OR document that the local weighted-avg algorithm matches the broker's exactly (which would require sourcing the broker's algorithm — Rithmic OrderPlant). Side and quantity must remain strict-equality.

### NICE-TO-HAVE

#### N-1: `last_mid_prices` map is never pruned
**Where:** `tracker.rs:163, 233`
**What:** `last_mid_prices.insert(symbol_id, current_price)` is called on every mid-price update; a flat-and-pruned position leaves its mid-price entry behind. Across a long session of many symbols traded once each, the map grows. Trivial leak.
**Recommendation:** Optional — call `last_mid_prices.remove(&symbol_id)` when pruning a flat-with-no-history position, or do a periodic compaction.

#### N-2: Synthetic `InvalidTransition` to fire the circuit breaker is awkward
**Where:** `tracker.rs:384-388`
**What:** The breaker is fired with a fabricated `InvalidTransition { from: OrderState::Idle, trigger: OrderTransition::Reject }` purely to satisfy the callback signature. The `from`/`trigger` values have nothing to do with the actual reconciliation mismatch — operators reading the breaker's diagnostic context will see misleading info.
**Recommendation:** Either widen the breaker callback signature to accept a typed reason (`enum BreakerCause { InvalidTransition(...), PositionMismatch(...) }`), or accept the awkwardness with a comment. Today's comment-free fabrication is the worst of both.

#### N-3: Consistent-path log/journal does not enumerate the local positions
**Where:** `tracker.rs:333-348`
**What:** AC4's "success logged with both states" is met as `open_positions=N`. A diagnostic operator reading the journal sees `reconciliation passed (periodic); open_positions=3` but cannot tell which 3 symbols and at what sizes — they have to grep elsewhere.
**Recommendation:** On `Consistent`, append a per-symbol summary into the journal message (or a separate journal record per symbol). Bounded by the open-position count, so cost is fine.

#### N-4: `BrokerPosition::flat(symbol_id)` and broker-omitted-symbol are indistinguishable
**Where:** `tracker.rs:268-270`
**What:** `broker_active = broker_pos.is_some_and(|p| !p.is_flat())` — both "broker explicitly reports symbol N as flat" and "broker doesn't include symbol N at all" produce `false`. That's intentional today, but loses a piece of information: if a broker omits symbol N entirely, we don't know if the broker has never seen it or has flushed it. Not blocking.
**Recommendation:** No code change needed; document the equivalence in the `reconcile` doc comment so 8-2 doesn't second-guess.

#### N-5: Fees / commissions are not modeled in realized-P&L
**Where:** `position.rs:94-100`
**What:** `realized_pnl = (exit - entry) * closed_qty * direction`. No fee subtracted. The broker's reported P&L will include fees; local will not. Reconciliation does not currently compare P&L (only side/qty/avg_entry), so this is dormant — but if a future story adds P&L reconciliation, fees will be a divergence vector.
**Recommendation:** Document the no-fees assumption in `Position::apply_fill` doc comment so a future reviewer doesn't get surprised.

#### N-6: `BracketManager` computes `realized_pnl_quarter_ticks` independently of `Position::apply_fill`
**Where:** `bracket.rs:106-118` (`FlattenOutcome::TakeProfit/StopLoss`); `position.rs:94-100`
**What:** Two implementations of "realized P&L on close." When story 8-2 wires the event loop to forward fills to BOTH `BracketManager::on_bracket_fill` AND `PositionTracker::apply_fill`, each will compute its own P&L. If both are surfaced, double-counting is possible; if only one is surfaced, the other's value is unused/stale.
**Recommendation:** Document at the seam: which side is canonical? My read: the `PositionTracker` is the source of truth (it persists across a session, accumulates across multiple brackets); the `BracketManager`'s `realized_pnl_quarter_ticks` is bracket-scoped diagnostic. A doc comment on `FlattenOutcome::TakeProfit::realized_pnl_quarter_ticks` would prevent confusion.

## Adversarial Review Notes

### Blind Hunter

- **P&L arithmetic** — saturating_i64 throughout. Round-trip Buy/Sell at same price yields zero realized P&L (exact close path resets and restores). Flip path (qty < self.quantity, then flip > self.quantity) tested. No silent wrap; saturation on extreme values masks errors but never corrupts.
- **i64 overflow on accumulated realized_pnl** — `total_realized_pnl_quarter_ticks` uses `saturating_add` over the per-symbol values. Realistic max session size: ES tick value $12.50, quarter-tick $3.125, i64 max ~9.2e18. Even at $1B P&L (3.2e8 quarter-ticks) we're 10 orders of magnitude under saturation. Practical concern: zero. Saturation guard: present.
- **Sign convention on long/short close** — `direction = +1` for Buy-side position, `-1` for Sell-side. (exit - entry) * closed_qty * direction. Verified: long buys low/sells high → positive. Short sells high/buys low → positive. Tests `sell_fill_close_computes_realized_pnl`, `short_partial_close_accumulates_realized_pnl` cover this.
- **`BrokerPosition` field validation** — `quantity: u32` (unsigned), `side: Option<Side>`. Shorts are NOT represented as negative qty (they are `side = Some(Sell), quantity > 0`). This contradicts a common broker-API convention (negative qty for shorts) — the wire-up in OrderPlant integration must explicitly normalize. Worth a doc comment on `BrokerPosition::side`.
- **Mismatch detection — false positives from in-flight fills** — see S-1 above. Real, architectural, requires API extension to fix correctly.
- **Reconciliation race during a fill being applied** — single-threaded engine hot path, so within the engine no race. The race is between the broker's snapshot time T0 and the engine's reconcile call time T (covered by S-1).

### Edge Case Hunter

- Tracker reconciles when broker returns empty positions but engine has open: PhantomLocal — `reconciliation_phantom_local_detected` covers.
- Engine flat but broker shows phantom: MissedFill — `reconciliation_missed_fill_detected` covers.
- Multiple symbols simultaneously diverged: `multi_symbol_reconciliation_reports_only_offenders` covers two-symbol case (one consistent, one mismatch). N-symbol generalization is implicit in the loop.
- Reconciliation triggered while a fill is being applied: not possible on the single-threaded hot path; race is between async broker query and synchronous fill application (S-1).
- Realized P&L overflow: saturating_i64; practical impossibility (S-1 calculations above).
- `query_positions` partial failure: trait signature is `Result<Vec<_>, BrokerError>`. Partial-success is not representable. If a broker can return positions for some symbols and fail for others, the wire-up code must turn that into either Ok(partial) (which silently omits symbols and triggers PhantomLocal!) or Err. **Recommendation: trait signature should evolve to `Result<HashMap<u32, Result<BrokerPosition, BrokerError>>, ...>` or document the all-or-nothing contract explicitly.** Today's contract is implicitly all-or-nothing — a single-symbol failure must surface as Err. Worth a doc comment.

## Deviations Review

### Deviation 1 — Reconciliation triggers (Tasks 6.1/6.2/6.3) deferred to 8-2/8-4
**Verdict: ACCEPTED with caveat (S-1 above).**

API surface is `reconcile_and_handle(broker_view: &[BrokerPosition], trigger: ReconciliationTrigger, now: UnixNanos)`. For startup: caller fetches once, passes the slice — clean. For reconnection: same shape, caller drives from FSM Reconciling state. For periodic: caller fires every 60s.

Adequate for startup (no fills in flight). Marginal for reconnection (fills could arrive during reconnect). **Inadequate for periodic during live trading** — the API has no quiesce hook or as-of-T0 sequence number, so periodic will trip on the in-flight-fill race (S-1). The Deviation itself is fine; the API should grow a quiesce/sequence seam before 8-2 ships periodic.

### Deviation 2 — `RithmicAdapter::query_positions` returns `BrokerError::PositionQueryFailed`
**Verdict: ACCEPTED.**

TickerPlant doesn't carry positions; the live position query goes through OrderPlant or PnlPlant. Wiring this in 4-5 would require building OrderPlant connectivity, which is its own multi-story effort. Surfacing as a typed error (rather than panicking or returning empty) is the right call: when reconciliation actually queries, it gets a clear "not yet implemented" signal rather than silently treating the account as flat.

This DOES prevent live trading TODAY — but it's gated behind OrderPlant integration anyway, so the gap is structurally unavoidable in 4-5 scope.

### Deviation 3 — `BracketManager::on_bracket_fill` → `Position::apply_fill` wiring deferred to event loop (8-2)
**Verdict: ACCEPTED with N-6 caveat.**

The seam `FlattenOutcome::TakeProfit { realized_pnl_quarter_ticks, .. }` carries the bracket-scoped P&L. The event loop in 8-2 will forward fills to both `BracketManager::on_bracket_fill` (for bracket lifecycle) and `PositionTracker::apply_fill` (for the position view). Two independent P&L computations; both should agree on a closed bracket. A doc comment on the seam (N-6) would prevent ambiguity about which is canonical.
