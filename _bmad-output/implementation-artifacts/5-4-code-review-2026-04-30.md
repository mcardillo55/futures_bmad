## Findings

### [Decision-Needed] D1 — Call-site contract for `permits_trading` vs `permits_trade_evaluation` is unenforceable by the compiler

**Source:** auditor + edge
**Location:** `crates/engine/src/risk/circuit_breakers.rs:241` (`permits_trading`), `:1077` (`permits_trade_evaluation`)
**AC reference:** Task 3.3 ("CRITICAL: fee staleness gate ONLY blocks trade evaluation"), Task 3.4 ("Implement `permits_trade_evaluation` as a separate check from `permits_trading` to enforce this distinction").

The two functions have identical signatures — `&self -> Result<(), TradingDenied>` — and differ only in semantics by docstring. There is no current call-site. When story 5.X eventually wires entry-decision and safety-flatten paths, picking the wrong one will silently violate the AC: a future developer using `permits_trade_evaluation()` for a stop-loss path would block stops on a stale fee schedule (the exact failure the AC names "the worst possible state"), and using `permits_trading()` for an entry path would let the bot enter trades with stale fees.

The current safety net is "the docstring on `permits_trade_evaluation` shouts CRITICAL." That is fragile.

**Suggested approaches** (pick one; this is the decision the user must make):

a. **Type-tag the result.** Two distinct return types:
   ```rust
   pub fn permits_entry(&self) -> Result<(), TradingDenied<EntryContext>>
   pub fn permits_safety(&self) -> Result<(), TradingDenied<SafetyContext>>
   ```
   Plus a `TradeIntent` enum the caller passes in. Forces callers to declare intent at the type level.

b. **Single function, `intent: TradeIntent` parameter.**
   ```rust
   pub fn permits(&self, intent: TradeIntent) -> Result<(), TradingDenied>
   ```
   `TradeIntent::{EntryEvaluation, SafetyFlatten}`. One function, callers cannot accidentally pick the wrong variant — they must pass intent explicitly.

c. **Rename for clarity** (keep two functions): rename `permits_trading()` → `permits_safety_or_entry()` (or `permits_safety_orders()` plus a separate `permits_entry()` that wraps both). The current name `permits_trading()` reads as "is trading permitted?" — which intuitively suggests the *strict* interpretation, the opposite of what the function actually does. This is the cheapest fix but doesn’t catch misuse at compile time.

d. **Accept the risk and rely on review.** Add a `#[doc = include_str!("CALLSITE_RULES.md")]` link, expand the docstring with examples, and rely on code review to catch misuse.

**Recommendation:** option (b) — single `permits(intent)` function with a callsite-mandatory `TradeIntent` enum. Smallest API surface, impossible to misuse, no breaking the existing `permits_trading()` callers (only tests today, easy migration).

### [Patch] P1 — Task 3.6 not executed: `fee_gate.rs` not wired into the new breaker

**Source:** auditor
**Location:** `crates/engine/src/risk/fee_gate.rs` (unchanged in this commit)
**AC reference:** Task 3.6: "Wire into fee gate module (`engine/src/risk/fee_gate.rs`) for integration with trade evaluation pipeline."

The story marked Task 3.6 as `[x]` complete, but `fee_gate.rs` is byte-for-byte unchanged. `FeeGate::permits_trade` (lines 71-104) still performs its own `days_since > 60` check and returns `Err(FeeGateReason::StaleSchedule)`. Nothing in this commit:

- Calls `CircuitBreakers::check_fee_staleness` from `FeeGate`,
- Replaces `FeeGate`’s internal staleness check with a delegation to the breaker,
- Or even references `CircuitBreakers` from `fee_gate.rs`.

Result: when story 5.X wires the trade-evaluation pipeline, there will be two independent fee-staleness paths with the same threshold (60 days) but different state machines. They could diverge — e.g. operator manually resets the breaker but `FeeGate` still has its own date-based check returning Err. The story’s own Dev Notes (`Project Structure Notes`) call out that `fee_gate.rs` should host fee staleness gating distinct from safety orders — that wiring did not happen.

**Suggested fix:** Either (a) remove the `>60 days` block from `FeeGate::permits_trade` and have callers consult `CircuitBreakers::permits_trade_evaluation()` first; or (b) explicitly delegate from `FeeGate::permits_trade` into `breakers.check_fee_staleness` + `permits_trade_evaluation` and remove the duplicate logic. The story spec implies (a). Tests should be updated to drive both implementations consistently.

### [Patch] P2 — `update_buffer_occupancy` re-logs `error!` every tick at ≥100% occupancy

**Source:** edge + blind
**Location:** `crates/engine/src/risk/circuit_breakers.rs:236-245`

```rust
if pct >= BUFFER_FULL_PCT {
    error!(target: "circuit_breakers", occupancy_pct = pct, current_len, capacity,
        "SPSC buffer 100% full — events will be dropped, tripping BufferOverflow");
}
```

There is no transition guard. If `tick()` runs at hundreds of kHz on a hot path while the buffer is wedged at 100%, this emits an `error!` line on **every tick** until the breaker is reset. The architecture spec (line 506) says "log the drop explicitly, trigger immediate circuit break" — the *drop* (the producer-side push failure) is per-event, but this is the *consumer-side observation* of saturation. Once the BufferOverflow breaker is `Tripped`, repeating the error log adds nothing and floods the journal. (Compare to `trip_breaker`, which is idempotent and silent on subsequent calls.)

**Suggested fix:** gate the `error!` on `matches!(self.buffer_overflow_state, BreakerState::Active)` (i.e. only log on the trip transition), or move the log inside the trip-breaker block at line 247 so it fires once.

### [Patch] P3 — `reset_breaker(MalformedMessages)` does not drain `malformed_window`

**Source:** edge
**Location:** `crates/engine/src/risk/circuit_breakers.rs:500-510`

`reset_breaker` flips state to `Active` but never touches `self.malformed_window`. If the operator manual-resets while the deque still holds 11+ recent timestamps within the 60s window, the *next* arrival immediately re-trips because the size check at line 419 (`> MALFORMED_THRESHOLD`) sees the carryover plus the new arrival.

The doc comment on `reset_breaker` says "V1: only invoked on process restart" — which would re-create the whole struct and avoid the issue. But the public API allows mid-process reset, and an existing test (`malformed_breaker_requires_manual_reset`) actually exercises that path. The test happens to use a 120s gap so pruning empties the deque on the next arrival, masking the bug.

**Suggested fix:** in `reset_breaker`, when `breaker_type == MalformedMessages`, also clear the deque. Same defensive treatment is appropriate for `BufferOverflow` though buffer occupancy is naturally re-measured each tick, so it self-heals.

### [Patch] P4 — `update_buffer_occupancy` has a stuck-state path if `DataQuality` is externally `reset_breaker`’d while buffer pressure is sustained

**Source:** edge
**Location:** `crates/engine/src/risk/circuit_breakers.rs:262-272`

The 80% block:
```rust
if pct >= BUFFER_GATE_PCT {
    if !self.buffer_high_condition {
        self.buffer_high_condition = true;
        if matches!(self.data_quality_state, BreakerState::Active) {
            self.trip_breaker(BreakerType::DataQuality, ...);
        }
    }
}
```

Sequence to stuck state:
1. Buffer crosses 80%. `buffer_high_condition = true`, DataQuality trips.
2. Operator (or future automation) calls `reset_breaker(BreakerType::DataQuality)` directly. State becomes `Active`. `buffer_high_condition` stays `true`.
3. Buffer remains ≥80%. Next tick: `if !self.buffer_high_condition` is false — the inner trip block is skipped. State stays `Active` despite buffer pressure being unresolved. The 95% breaker still fires, but the 80% gate is silently disabled until buffer drops below 80% (clearing the condition flag) and crosses 80% again.

This is admittedly a corner case (no current call-site invokes `reset_breaker(DataQuality)`), but the invariant `buffer_high_condition = true ⇒ DataQuality is Tripped` is broken by `reset_breaker`. Either `reset_breaker` should also clear the `buffer_high_condition` / `data_quality_condition` flags, or the 80% block should re-trip whenever `pct >= 80 && state == Active` regardless of the condition flag.

**Suggested fix:** the simpler correction — drop the `!self.buffer_high_condition` guard on the trip side; idempotency is already handled by `trip_breaker` (line 374-384 short-circuits when `previous == Tripped`). Keep the flag for the auto-clear bookkeeping but stop using it as a "have we seen this transition" memo on the trip side.

### [Defer] DF1 — `EventLoop::tick` measures buffer occupancy *before* `try_pop`

**Source:** edge
**Location:** `crates/engine/src/event_loop.rs:182-186`

`update_buffer_occupancy` is called with `consumer.available()` before the per-tick `try_pop`. Reading is not wrong (it’s the live count at the start of tick), but it means the breaker measures one slot higher than the post-pop view. At sustained saturation the difference is irrelevant; at the boundary (e.g. exactly 95%) it could cause a one-tick early trip. Defer — pre-existing tick ordering, would need design work to decide if the framework should observe pre- or post-pop occupancy.

### [Defer] DF2 — `event_processed_this_tick` gating means data-quality only refreshes on event-bearing ticks

**Source:** edge
**Location:** `crates/engine/src/event_loop.rs:231-251`

`update_data_quality` only runs in the per-tick branch when an event was popped (the periodic 1000-tick stale block also wires it). This is a deliberate trade-off — the comment says "otherwise we’d flap the gate based on a stale view of the book" — but it means a transient quality issue resolves only on the next event arrival. In a quiet market, the gate could stay tripped for many ticks after recovery. Defer — design choice with a stated rationale.

### [Defer] DF4 — `ConnectionFsm` accepts any state transition without validation

**Source:** auditor
**Location:** `crates/engine/src/connection/fsm.rs:56-65`

`transition()` blindly assigns the new state and notifies the breaker. The story Dev Notes admit "richer back-off logic is deferred to future stories." The unused `TransitionError::Illegal` variant is a placeholder. This is fine for story 5.4 (the AC only requires `CircuitBreak` detection), but a future bug where two threads race transitions through `Reconnecting → Reconciling → CircuitBreak` could miss intermediate states and the breaker still trips correctly only because the *terminal* state is the trigger. Defer — explicitly out of scope per Dev Notes.

### [Dismiss] R1 — `chrono::NaiveDate::checked_sub_days` not used in `check_fee_staleness`

The naive subtraction `(current_date - fee_schedule_date).num_days()` is correct for the intended use (operator config dates and trading dates within the supported chrono range). The `chrono::Duration` arithmetic does not panic for normal date ranges. Not worth changing.

### [Dismiss] R2 — Float comparison `pct >= 95.0` could be off by an FP ULP

`f32` precision at 95.0 for the input domain (current_len up to ~131K, capacity up to ~131K) is well below 0.0001 deviation. Boundary tests verify expected behavior. Dismiss.

### [Dismiss] R3 — `ConnectionFsm::transition` notifies the breaker on every state change including no-op transitions to the same state

`on_connection_state_change` only acts on `CircuitBreak` and is itself idempotent (`trip_breaker` short-circuits on already-tripped). No real concern.
