### 4-3 S-1 (rejected entry → no naked stop) — **VERIFIED**

### 4-3 S-3 (engage_flatten returns AlreadyFlattening) — **VERIFIED**
`bracket.rs:653-675` — `engage_flatten` now branches on `entry.bracket.transition(Flattening)`. On `Err`, it warns, re-inserts the entry to keep the bracket alive, and returns `Some(FlattenContext::AlreadyFlattening { bracket_id })`. Test: `engage_flatten_twice_returns_already_flattening` confirms the second call returns `AlreadyFlattening` rather than a fresh `EntryFlatten`.

### 4-3 S-2 (warn-log only — judgment call) — **VERIFIED-AS-DEFERRED**
`bracket.rs:266-275` — method-level docstring explicitly notes the assumption (atomic Market fills on CME) and the deferral (story 4.5 owns true partial handling). `bracket.rs:325-334` adds a `warn!` log when `FillType::Partial` arrives on the entry leg.

**Important caveat:** the warn-log does NOT change the SL sizing — the SL is built from `entry.bracket.stop_loss` which was constructed at bracket-creation time with the FULL quantity. So in live trading a 1-of-3 partial entry would result in a 3-lot SL on a 1-lot position; on stop-out, the SL would over-fill by 2, creating a 2-lot opposite position. **This is a real correctness hole for live trading.** The dev is honest about the deferral and the architectural decision is reasonable (CME Market orders typically fill atomically), but **production live-trading must NOT proceed until 4.5 lands**. See SHOULD-FIX S-2 below.

## Verdict on Dev Deviations

### Deviation 1 — `BrokerAdapter::query_order_status` not added; `BrokerOrderStatus` + `resolve_uncertain` instead
**JUSTIFIED.** The current `BrokerAdapter` trait already has `query_open_orders() -> Vec<(u64, OrderState)>` (`crates/core/src/traits/broker.rs:32`). The dev correctly observed that adding a new trait method would expand `RithmicAdapter` and `MockBroker` simultaneously, and that the bare *enum + entry-point* pattern lets 4-5 wire the live query into the existing `query_open_orders` (mapping `OrderState` to `BrokerOrderStatus`) without redesign. The seam is complete: caller iterates `state_machines.values()` for orders in Uncertain/PendingRecon, calls `query_open_orders` (or 4-5 may add `query_order_status` if per-order semantics are needed), maps the result, calls `begin_recon` then `resolve_uncertain`. **Plug-in for 4-5 is straightforward.**

### Deviation 2 — `apply_fill` keeps Submitted→Confirmed auto-upgrade
**JUSTIFIED but WITH AUDIT TRAIL GAP.** `mod.rs:497-505` flips `tracked.state = Confirmed` and calls `sm.transition(Confirm, ...)` on first non-rejection fill. The state machine is updated correctly. However, **no journal record is emitted for the Submitted→Confirmed arc** — only the subsequent PartialFill/Filled transition is journaled, with `from_state = "Confirmed"`. A forensic operator replaying the audit trail will see Confirmed appear out of nowhere with no Submitted→Confirmed entry. This is the same defect as 4-2 N-4 (deferred). It is NOT addressed in 4-4 despite the dev now writing every other transition (Submit, Timeout, BeginRecon, Resolve) to the journal explicitly. **Recommendation:** add a one-line `journal.send(JournalEvent::OrderStateChange(...))` for the implicit Submitted→Confirmed transition. Currently filed as SHOULD-FIX S-3.

### Deviation 3 — 4-3 S-2 (partial entry over-sizes SL/TP) addressed only with doc + warn
**JUSTIFIED FOR PAPER-TRADING ONLY.** As discussed in carryover verification above. The dev is explicit about the gap, and the production-readiness gate (story 4.5) is correctly identified. The warn-log is sufficient for the testbed; it is NOT sufficient for live trading. Carryover S-2 remains in deferred-work tracked against 4-5.

## Findings

### SHOULD-FIX (4)

#### S-1 — Terminal `mark_resolved` failure leaves WAL stale (warn-only)
`crates/engine/src/order_manager/mod.rs:618-628`. When a fill drives the order to a terminal state, the in-memory SM and `TrackedOrder` advance to Filled/Rejected, but if `wal.mark_resolved()` fails, the row is left at the previous non-terminal state and only a `warn!` is emitted. On crash, `recover_pending` would re-include this order. Recovery would then query the broker, find Filled, and resolve correctly — but the recovery path is an implicit fallback rather than an explicit guarantee. **Recommend:** either (a) propagate the WAL failure as a fatal engine error (the spec says circuit-breaker-on-WAL-fail is the right posture), or (b) at minimum, journal a `SystemEvent` so an operator sees the durability lapse without grepping logs.

#### S-2 — Partial entry SL over-sizing (carryover 4-3 S-2)
`crates/engine/src/order_manager/bracket.rs:325-334` (warn-log only). The SL is built from `entry.bracket.stop_loss.quantity` (the original full size) regardless of how much of the entry leg actually filled. A partial entry produces an over-sized SL on a smaller position. The dev's docstring explicitly defers true handling to story 4.5, but the spec note doesn't mention the cascading risk: on stop-out, the over-sized SL would over-flatten and create a 2-lot opposite position. **Recommendation for live-trading gating:** until story 4.5 lands, either (a) reject Market entries that fill partially (treat any partial entry as a panic-mode trigger for that bracket), or (b) document explicitly in `4-5` spec/story that paper-trading is the only supported mode for 4.4. Currently filed in deferred-work.

#### S-3 — Auto-upgrade Submitted→Confirmed has no journal record
`crates/engine/src/order_manager/mod.rs:497-505`. The implicit transition is applied to the SM but is NOT journaled. Every other transition in the manager (`submit_order`, `check_timeouts`, `begin_recon`, `transition_to_resolved`) emits an `OrderStateChange` journal record. The auto-upgrade is the only audit-trail gap. **Recommend:** emit a journal record for this transition. ~5 lines of code.

#### S-4 — `flatten_side_for` returns None silently when WAL is absent or row missing
`crates/engine/src/order_manager/mod.rs:821-833`. If `self.wal` is `None` (the no-WAL constructor — used for the 4-2 fill-consumer path), `flatten_side_for` returns None. In `resolve_uncertain` line 786-788, this collapses the `FilledFlattenRequired` branch into `ResolveOutcome::NotApplicable` — silently dropping the flatten request despite the broker confirming the position was filled. **This is a silent failure for a critical safety path.** In production the `with_wal` constructor will always be used, but the failure mode is not asserted. **Recommend:** add a debug-assert that the WAL is present whenever `bracket_ids` lookup found nothing AND broker reports filled, OR change `flatten_side_for` to surface a distinct error variant so the caller sees "we cannot determine the flatten side — escalate to circuit breaker" rather than `NotApplicable`.

### NICE-TO-HAVE (6)

#### N-1 — `wal.rs:168` `let _ = order.order_type;` workaround for unused-field lint
The OrderType is intentionally not stored in the WAL row (Market-only for now). The dummy `let _ =` and the `_silence_unused_order_type` fn at line 348-351 are awkward. Cleaner: drop the unused import and drop the `_ =` line — the compiler doesn't warn about unused struct fields when the struct is only borrowed.

#### N-2 — `WalError::BadRow` has no `order_id` context
A corrupt row that fails `parse_state` returns just `unparseable WAL row: invalid state string: ...` with no `order_id`. Add the offending row's `order_id` to the error message for forensic diagnosis.

#### N-3 — Fill arriving while order is in PendingRecon is silently dropped
`crates/engine/src/order_manager/mod.rs:550-566`. If a real fill races the recon round-trip (broker confirms via `query_open_orders` but a `FillEvent` had already been pushed onto the queue), the SM is in PendingRecon and `Fill`/`PartiallyFill` triggers from PendingRecon are invalid — `apply_fill` returns `FillOutcome::InvalidTransition`, the fill is dropped, and the circuit-breaker callback fires (which conflates "data race" with "state machine corruption"). The order will still resolve correctly via `resolve_uncertain`, so position safety is preserved. But the `InvalidTransition` callback firing for a benign race is noise. **Recommend:** in `apply_fill`, if SM is in `Uncertain | PendingRecon`, log at `info` and absorb the fill silently (treat the eventual `resolve_uncertain` as authoritative) rather than tripping the breaker.

#### N-4 — `tick` does not invoke any clock — caller must pass `now`
This is intentional and good for determinism (matches the rest of the engine), but worth documenting in the module-level docs that the caller (event loop) is the source of monotonic time and must not pass a non-monotonic `now`.

#### N-5 — `state_to_str` and `parse_state` duplicate the `OrderState` Debug derives
The two functions are open-coded match arms that risk drifting from `core::OrderState`. A `serde` derive on `OrderState` (with kebab-case rename), or a single `as_str(&self) -> &'static str` method on `OrderState` itself, would centralize the mapping. Low priority — there are only 11 states.

#### N-6 — `OrderManager::with_wal` constructor pattern feels brittle
`mod.rs:245-250` uses `..Self::new(journal)` to merge fields. This means future field additions to `OrderManager` get wired correctly through `new` but need to be considered carefully for `with_wal`. Not a bug, just a maintenance trap. A builder pattern (`OrderManager::builder().with_journal(j).with_wal(w).build()`) would be more robust as the field set grows.

## Edge Cases Audited

- **WAL corruption / db-locked / disk-full on order persist** — `submit_order` propagates `WalError` as `SubmitError::WalWriteFailed(...)`; queue push and SM transition are skipped. Order is not submitted. **Correct.**
- **`tick` called before `submit_order`** — Empty `state_machines` map, `to_uncertain` empty, no-op. **Safe.**
- **`resolve_uncertain` for an order that completed during recon round-trip** — SM is in PendingRecon; if the broker reports Filled and the bracket exists, returns `FilledBracketProtects` and tears down. If the order had ALREADY been removed by an apply_fill that succeeded (impossible because SM is in PendingRecon, not Confirmed), returns `NotApplicable`. **No double-flatten.**
- **Partial fill, then network drops, then recon** — In flight: SM is in PartialFill (cumulative remaining < original), order is NOT eligible for the timeout watchdog (only Submitted is). If the network drops while in PartialFill, the order is not detected as Uncertain — this is correct because the broker has already partially filled (the position is real). **Safe but worth a follow-up:** is there a watchdog for PartialFill orders that never receive their final Fill? Not in 4-4 scope.
- **Over-fill and zero-size rejections journaled?** — They are NOT journaled (`apply_fill` returns `InvalidFillSize` early, before the journal block at line 600-615). The `warn!` log is the only durable trace. The original tracked state is preserved (correct, no corruption), but a forensic replay will see no record of the malformed fill. **Mild audit-trail gap** — could be promoted to a `SystemEvent` journal entry. Filed under deferred-work.
- **Submitted state with `submitted_at == None`** — Shouldn't happen (the SM captures the timestamp on Submit transition), but `check_timeouts` line 665 guards with `if let Some(submitted_at) = sm.submitted_at()`. **Defensive coding correct.**
- **PendingRecon escalation re-fires on every tick** — `recon_started_at.remove(&oid)` after escalation prevents this (line 901). **Correct.**
- **`Send`/`Sync`** — `OrderWal` holds `rusqlite::Connection`, which is `!Send`. `OrderManager` is therefore `!Send`. The `CircuitBreakerCallback` is `Arc<dyn Fn + Send + Sync>`, so the callback can safely be observed from other threads. The manager itself runs on the engine main thread per the architecture spec. **No threading defect.**
- **Duplicate order_id resubmit** — `submit_order` rejects with `SubmitError::DuplicateOrderId(_)` (line 309-313). Test: `duplicate_order_id_is_rejected`. **Correct.**

## Bottom Line

The state machine is one of the cleanest pieces of code in the engine — exhaustive matching, comprehensive Cartesian-product testing, and a clean trigger-based API that insulates downstream callers from `core::OrderState` evolution. The WAL is small, focused, and shares the journal database for a single fsync barrier (architecturally correct). The reconciliation skeleton is testable, complete, and ready for 4-5 to plug the live broker query in.

The four SHOULD-FIX items concern silent-failure modes around secondary writes (WAL on terminal, journal on auto-upgrade) and known carryover gaps (partial entries) — none of which prevent the merge of this skeleton, but all of which should be closed before the engine event loop drives this onto a live fill stream.

**Verdict: APPROVE-WITH-CHANGES.** Address S-1, S-3, and S-4 before story 4-5 wires the live query; S-2 (partial entry sizing) is a paper-only-until-4.5 gating issue and is correctly tracked in deferred-work.
