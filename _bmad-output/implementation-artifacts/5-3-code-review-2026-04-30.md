## Top-Line Verdict: **CHANGES-REQUIRED**

**Findings:** 2 BLOCKING / 1 SHOULD-FIX / 4 NICE-TO-HAVE

The component-level pieces are well constructed: `flatten() -> Result<(), FlattenError>` is a clean Result-shape sibling to the legacy `flatten_with_retry` (no behaviour drift, identical retry mechanics, captures per-attempt rejection reasons), `AnomalyCheckOutcome` is a tidy enum, and `handle_anomaly` correctly composes flatten -> panic-mode escalation with a `compare_exchange` guard against double-activation and an `AlreadyInPanic` short-circuit on re-entry. The sound rebase adaptation onto 5.6's `PanicContext`/`PositionSnapshot`/`FlattenAttemptDetail` types is verified end-to-end in `flatten_failure_activates_panic_mode_with_context` (3 attempts -> panic -> cancel-once -> journal SystemEvent).

But two of the six tasks the story claims as "complete" are not actually done in main:

1. **Task 4 wiring** (anomaly check / handle_anomaly invoked from the event loop) is absent — neither `check_position_anomaly` nor `handle_anomaly` has any non-test caller. The acceptance criterion *"Given position outside active strategy context When detected Then anomalous position breaker activates, automated flatten submitted"* is not satisfied at integration level. The Dev Notes correctly disclose that Task 4 is implemented as "a pure async helper... intended to be `tokio::spawn`-ed onto the broker runtime", but a helper that never runs is not wiring.
2. **Task 5** (CircuitBreakers integration with panic mode) is structurally absent — there is no `panic_mode: bool` field on `CircuitBreakers`, `permits_trading()` does not consult panic mode, and there is no `DenialReason::PanicModeActive` variant. Two parallel "can we trade?" gates now exist (`CircuitBreakers::permits_trading` and `PanicMode::is_trading_enabled`) and every order-submission path must remember to check both. The story checked all three 5.x boxes but the framework was never extended.

These are BLOCKING for a `done` status: Task 4.x and Task 5.x literal AC text is not satisfied. They are NOT BLOCKING for a `review`-status snapshot of the component-level pieces — those work as designed.

## Acceptance Criteria Audit

The spec lists 3 BDD ACs:

- **AC1** — *Given position outside active strategy context When detected Then anomalous position breaker activates, automated flatten submitted*: **PARTIAL.**
  - `check_position_anomaly` correctly identifies the divergence and trips the `AnomalousPosition` breaker via `trip_breaker` with a structured `delta=` reason. Verified by reading; **NOT verified by any unit test in main** — see B-3 below.
  - "automated flatten submitted" requires `handle_anomaly` to fire on the `Anomalous` outcome. There is no production caller. **NOT MET at integration.**

- **AC2** — *Given `broker/src/position_flatten.rs` When flatten attempted Then market order submitted, if rejected wait 1s retry (3 attempts total), returns success/failure*: **MET.**
  - `PositionFlattener::flatten` (alias of `FlattenRetry::flatten`) submits a Market `OrderEvent`, sleeps `FLATTEN_RETRY_INTERVAL` between attempts, returns `Ok(())` on first success or `Err(FlattenError::AllAttemptsFailed { attempts })` after `FLATTEN_MAX_ATTEMPTS` rejections. Tests `flatten_result_ok_on_first_attempt_success`, `flatten_result_returns_all_attempts_failed_after_three_rejections`, `flatten_result_ok_after_retries` cover the three paths.
  - Pre-allocates `Vec::with_capacity(self.max_attempts)` so the retry loop never grows the failure-reason vec — appropriate.

- **AC3** — *Given `engine/src/risk/panic_mode.rs` When all 3 attempts fail Then panic mode: all trading disabled, all entry/limit cancelled (stops preserved), operator alerted, requires manual restart, full context logged*: **MET (via 5.6's framework).**
  - `PanicMode::activate_with_context` flips an `AtomicBool` (so `is_trading_enabled()` returns false on every thread without a Mutex), invokes `OrderCancellation::cancel_entries_and_limits` (whose contract explicitly excludes stops), emits a `Critical`-severity `Alert` with `flatten_attempts` populated, journals a `CircuitBreakerEvent` and a `SystemEvent`, and exposes no `deactivate()` API. `compare_exchange` ensures idempotent activation. Test `flatten_failure_activates_panic_mode_with_context` end-to-ends the path; `activate_emits_critical_alert_with_flatten_context` (5.6 test) confirms per-attempt detail flows through the alert channel.
  - "full context logged" — the journal `SystemEvent.message` carries activation reason + cancel count + "stops preserved" but NOT per-attempt detail. Per-attempt detail is on the alert channel only. The story's relaxed test assertion (only "stops preserved") matches this split. Acceptable post-rebase: per-attempt detail IS logged, just on a different channel.

**AC summary: 1 met / 1 partial (Task 4 wiring missing) / 1 met-via-rebase-adaptation.**

## Adversarial Findings

### BLOCKING

#### B-1 — Task 5 not implemented: `permits_trading()` does not consult panic mode

**Source:** Acceptance Auditor + Edge Case Hunter
**Location:** `crates/engine/src/risk/circuit_breakers.rs:105-171, 240-273`

The story explicitly checks Tasks 5.1, 5.2, 5.3:
- 5.1: Add `panic_mode: bool` field to CircuitBreakers struct
- 5.2: In `permits_trading()`, check panic_mode first — if active, return denial with `PanicModeActive` reason
- 5.3: Panic mode overrides all other breaker states — it is the highest severity

None of these are present in main. The `CircuitBreakers` struct has no panic-mode field; `permits_trading()` reduces over the ten breaker/gate states only (`all_active_excluding_eval_only`); `DenialReason` (in `risk/mod.rs`) has no `PanicModeActive` variant. Two parallel gates now exist:
- `CircuitBreakers::permits_trading()` — used by every order-submission path that's been wired so far.
- `PanicMode::is_trading_enabled()` — held in a separate `PanicMode` struct (Story 4.3 origin), exposed via `Arc<PanicMode>` in `handle_anomaly`.

Consequence: a panic-mode activation does NOT cause `permits_trading()` to deny. Any caller that checks only `permits_trading()` (which is most of them — see story 5.1's wiring) will continue to permit trades after panic activates. The only way panic mode currently blocks a trade is if the caller independently consults `PanicMode::is_trading_enabled()`, which the engine wiring does not do.

The 5.3 worktree's original 414-line `circuit_breakers.rs` scaffold DID have `panic_mode_active()` and `permits_trading()` with panic-mode short-circuit (visible in the dropped commit `4412206`). The rebase port of `check_position_anomaly` left this behind.

**Fix:** Either add `panic_mode: Arc<PanicMode>` to `CircuitBreakers::new`, prepend a `if self.panic_mode.is_active() { return Err(...PanicModeActive...) }` in `permits_trading()`, and add the `PanicModeActive` variant to `DenialReason` — OR explicitly defer this to a follow-up story and update the AC checkboxes to reflect that 5-3 only ships the components, not the `permits_trading` integration. The current state (boxes checked, integration absent) is misleading.

#### B-2 — Task 4 wiring absent: nothing calls `check_position_anomaly` or `handle_anomaly`

**Source:** Acceptance Auditor + Blind Hunter
**Location:** `crates/engine/src/event_loop.rs` (no references); `crates/engine/src/risk/anomaly_handler.rs:70` (no production callers)

Tasks 4.1 ("After position reconciliation or fill processing, call `check_position_anomaly()`") and 4.2 ("When anomalous position detected, spawn flatten attempt via `PositionFlattener`") are checked but unimplemented:

```
$ rg "handle_anomaly|check_position_anomaly" crates/engine/src/event_loop.rs
(no matches)

$ rg "handle_anomaly|check_position_anomaly" --type rust crates/
(matches only in anomaly_handler.rs and circuit_breakers.rs themselves and their own test modules)
```

The Dev Notes call out this gap: *"Task 4's 'wire into event loop' is implemented as a pure async helper (`risk::anomaly_handler::handle_anomaly`) rather than embedded inside the synchronous `EventLoop`. The helper is intended to be `tokio::spawn`-ed onto the broker runtime"*. Defining a helper does not satisfy the AC — the AC says the breaker must activate "When detected" and the flatten must be "automated submitted", which requires a caller in the event loop or fill-processing path that detects the anomaly and dispatches the helper.

In the absence of this wiring, the anomalous-position detection path is dead code: `check_position_anomaly` and `handle_anomaly` exist, are unit-tested in isolation, and never run in production. Combined with B-1, this means an anomalous position will not be detected, will not trigger a flatten, will not escalate to panic mode, and will not block subsequent trades. The complete safety chain is non-functional.

**Fix:** Add an integration point in `event_loop.rs` (or `order_manager`'s reconciliation handler) that calls `circuit_breakers.check_position_anomaly(symbol_id, current, expected, ts)` after each fill or reconciliation event, and on `AnomalyCheckOutcome::Anomalous` dispatches a `tokio::spawn(handle_anomaly(...))` onto the broker runtime. Otherwise, downgrade Tasks 4.1-4.2 to deferred and capture in a follow-up story before marking 5-3 done.

### NICE-TO-HAVE

#### N-1 — `check_position_anomaly` allocates the reason string before the state check

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/risk/circuit_breakers.rs:861-867`

```rust
let delta = current_position.saturating_sub(strategy_expected_position);
let reason = format!(
    "anomalous position on symbol {symbol_id}: current={current_position} \
     strategy_expected={strategy_expected_position} delta={delta}"
);
if matches!(self.anomalous_position_state, BreakerState::Active) {
    self.trip_breaker(BreakerType::AnomalousPosition, reason, timestamp);
}
```

When the breaker is already tripped (the second and subsequent anomaly arrivals), the `format!` allocation is wasted — we build a `String`, never pass it anywhere, and drop it. The detection path is on the broker runtime, not the engine hot path, but the function may be called per-fill on a busy reconciliation loop. Hoist the `format!` inside the `if`. Cosmetic.

#### N-2 — `anomalous_position_detail` not refreshed on subsequent calls

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/risk/circuit_breakers.rs:850-872, 297-301`

Once `AnomalousPosition` is tripped, subsequent `check_position_anomaly` calls return `Anomalous { ... }` but never re-enter `trip_breaker` (it's idempotent — `previous == Tripped` returns early). Consequence: `anomalous_position_detail` (set inside `trip_breaker_with_context` on the first trip) is frozen at the FIRST anomaly's delta string. If a later, larger divergence arrives, the operator's `permits_trading()` denial-reason text still shows the first one. By that point the breaker is already in manual-reset mode and panic mode is the relevant escalation, so this is mostly cosmetic — but operators reading the denial reason will see stale detail. Update the cached detail on every Anomalous outcome regardless of state transition.

#### N-3 — Misleading dead arm in `handle_anomaly`

**Source:** Blind Hunter
**Location:** `crates/engine/src/risk/anomaly_handler.rs:147-157`

```rust
Err(FlattenError::OrderRejected { .. }) => {
    // OrderRejected is a per-attempt variant — `flatten()` itself
    // never returns it (it is wrapped into AllAttemptsFailed when
    // every attempt fails). This arm exists for future-proofing.
    warn!(...);
    AnomalyResolution::Flattened
}
```

If `flatten()`'s contract ever changes and this arm fires, it returns `Flattened` (the success outcome). Operators and tests would see "flatten succeeded" when the order was actually rejected. The comment correctly notes the arm is unreachable today, but the chosen fallback is semantically wrong for the future. Better: `unreachable!("flatten() never returns OrderRejected directly")`, or `AnomalyResolution::PanicActivated { attempts: vec![reason] }` to fail safely toward the panic path.

## Rebase Adaptation Soundness — Targeted Verification

Per the review brief, three rebase adaptations needed verification:

1. **414-line scaffold dropped, `check_position_anomaly` + `AnomalyCheckOutcome` ported onto Story 5.1's framework.**
   Verified. The ported function uses `BreakerType::AnomalousPosition` (5.1's enum), `BreakerState::Active`, and `trip_breaker(...)` (5.1's method) — all correctly. `AnomalyCheckOutcome` is independent. The state-check predicate `matches!(self.anomalous_position_state, BreakerState::Active)` is correct (only trip on transition; idempotent at the `trip_breaker` level too). **Sound, but the 9 dropped tests left a coverage gap — see S-1.**

2. **Bespoke `PanicContext`/`PanicPositionDetails`/`FlattenAttemptRecord` replaced with 5.6's `PanicContext`/`PositionSnapshot`/`FlattenAttemptDetail`.**
   Verified. `anomaly_handler.rs` constructs a `PanicContext { position: PositionSnapshot { ... }, current_pnl: 0, flatten_attempts: Vec<FlattenAttemptDetail> }` matching the live struct shape, populates `attempt_number = (i as u8).saturating_add(1)` (correct for `FLATTEN_MAX_ATTEMPTS = 3`), reuses `request.symbol_id`/`side`/`quantity` for `order_details`, and threads the per-attempt `reason` String into `rejection_reason`. The `current_pnl: 0` and `unrealized_pnl: 0` defaults are honest placeholders — the helper doesn't have a P&L source — but they will need real values once Task 4 is wired (the position-tracker is the canonical source). The size derivation correctly maps flatten-side -> position-side (Sell flatten -> Buy position -> positive size; Buy flatten -> Sell position -> negative size). **Sound.**

3. **Test assertion relaxed: journal `SystemEvent` no longer asserts "attempt 1/2/3" — just "stops preserved".**
   Verified. The assertion change is correct given that 5.6 routes per-attempt detail through the alert channel (covered by `panic_mode::tests::activate_emits_critical_alert_with_flatten_context`), not through the journal. The journal's `SystemEvent.message` only carries the activation reason + cancellation count + "stops preserved" — the relaxed assertion matches that contract. **Sound.**

4. **Skipped clippy-fix commit `7b4d170` targeting tests that no longer exist.**
   Verified. `cargo clippy --workspace --all-targets -- -D warnings` is clean on main. The skipped commit fixed `clippy::manual_contains` in a test that the rebase never landed on main. No remediation needed.
