## Adversarial Findings

### SHOULD-FIX

#### S-1 — `naive_to_utc` silently treats every TOML naive timestamp as UTC, but nothing in the type system or the parser enforces this

**Source:** Edge Case Hunter + Acceptance Auditor
**Location:** `crates/engine/src/risk/event_windows.rs:138-141`, `crates/core/src/config/trading.rs:117`

`EventWindowConfig.start` and `EventWindowConfig.end` are `NaiveDateTime`. TOML's RFC 3339 partial form (`"2026-04-16T14:00:00"` — no zone designator) deserialises to `NaiveDateTime` cleanly, but the *interpretation* of "what wall-clock zone is this in" happens silently in `naive_to_utc`:

```rust
fn naive_to_utc(naive: NaiveDateTime) -> DateTime<Utc> {
    DateTime::<Utc>::from_naive_utc_and_offset(naive, Utc)
}
```

The doc-comment on `EventWindowConfig.start` (`trading.rs:117`) says *"keep TOML strings in UTC to avoid surprises"* — that's documentation, not enforcement. An operator in `America/New_York` who reads "FOMC at 14:00" (which is 14:00 ET = 18:00 UTC) and writes `start = "2026-04-16T14:00:00"` will get a window that activates four hours early. There is no validation, no warning, and the failure mode is silent — the window will activate at the wrong wall-clock and trip nothing because nothing else is wrong.

**Fix options** (pick one):
1. Change the deserialiser to require RFC 3339 with explicit zone (`DateTime<Utc>` instead of `NaiveDateTime`). TOML supports it (`"2026-04-16T14:00:00Z"` or `"…+00:00"`). Forces operators to write the zone, fail-loud if missing.
2. Keep `NaiveDateTime` but add a TOML-level `time_zone: String` field to `TradingConfig` (e.g., `"UTC"`, `"America/New_York"`) and have `naive_to_utc` consult it via `chrono-tz`. More flexible, more code.
3. Keep the current behaviour but add a startup-time `warn!` log when `events` is non-empty saying *"event windows interpreted as UTC; configured times: FOMC=2026-04-16T14:00:00 UTC, …"*. Soft fix, makes the silent assumption visible at boot.

I'd take (1) — `DateTime<Utc>` in the config struct directly. It's a one-line type change, the test fixtures already construct via `Utc.with_ymd_and_hms`, and the only downside is that operators must learn to suffix `Z`. (3) is the cheapest if (1) breaks too many things.

#### S-2 — `EventAction::severity` and `TradingRestriction::severity` are duplicated and will drift

**Source:** Blind Hunter + Edge Case Hunter
**Location:** `crates/core/src/config/trading.rs:85-91`, `crates/engine/src/risk/event_windows.rs:71-78`

Two separate `severity() -> u8` methods with identical match arms exist on two enums that mirror each other one-for-one:

```rust
// trading.rs (core)
impl EventAction {
    pub const fn severity(self) -> u8 {
        match self {
            EventAction::DisableStrategies => 1,
            EventAction::ReduceExposure => 2,
            EventAction::SitOut => 3,
        }
    }
}

// event_windows.rs (engine)
impl TradingRestriction {
    pub const fn severity(self) -> u8 {
        match self {
            TradingRestriction::DisableStrategies => 1,
            TradingRestriction::ReduceExposure => 2,
            TradingRestriction::SitOut => 3,
        }
    }
}
```

`get_trading_restriction` only uses `TradingRestriction::severity` (`event_windows.rs:245`). `EventAction::severity` is **never called** in this branch. It's dead code today but live API surface — anyone could call it.

If a future story adds a fourth variant (e.g., `EventAction::CancelStops`) and forgets to update one side, the comparison silently misranks. There is no test that asserts the two tables agree.

**Fix options** (pick one):
1. Drop `EventAction::severity` (it's unused) and have `TradingRestriction::severity` be the single source of truth. Most-restrictive comparison goes through the engine-side enum.
2. Keep both but add a parity test in `event_windows.rs` that iterates every `EventAction` variant and asserts `EventAction::X.severity() == TradingRestriction::from_action(EventAction::X).severity()`.

I'd take (1) — it's two lines of deletion and removes the trap.

#### S-3 — `EventWindowManager::new` silently drops malformed entries; no `debug_assert!` matches the validate-first contract

**Source:** Blind Hunter + Edge Case Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:159-162`, `ResolvedWindow::from_config:115-134`

`EventWindowManager::new` filter-maps with `ResolvedWindow::from_config`:

```rust
pub fn new(configs: &[EventWindowConfig]) -> Self {
    let windows = configs.iter().filter_map(ResolvedWindow::from_config).collect();
    Self { windows }
}
```

`from_config` returns `None` for any config that has both end+duration, neither end+duration, `duration_minutes == 0`, end before start, or arithmetic overflow. The doc-comment correctly identifies this as defence-in-depth ("validation lives in `core::config::validation`") and there's a test `manager_drops_malformed_entries`.

The problem: silent drops mean a config like

```toml
[[events]]
name = "FOMC"
start = "2026-04-16T14:00:00"
# (operator forgot both end and duration_minutes)
action = "sit_out"
```

— if it slips past `validate_trading_config` (e.g., a code path constructs `TradingConfig` directly without running validation, like the test fixture in `circuit_breakers.rs:497-528` or `validation.rs:300-329`) — produces an `EventWindowManager` with `len() == 0`. The operator ships, expects FOMC blackout, gets nothing. The audit trail is nothing.

This is the same class of issue as S-4 in the 5.1 review (`CircuitBreakers::new` not enforcing the validation invariants).

**Fix:** add `debug_assert!` in `EventWindowManager::new` matching `validate_event_window_config` rules, so debug builds and tests catch the mistake immediately. Release builds still silently drop (preserving the defence-in-depth posture). One test that constructs an invalid config and panics in debug-mode would pin the contract.

```rust
pub fn new(configs: &[EventWindowConfig]) -> Self {
    debug_assert!(
        configs.iter().all(|c| validate_event_window_config(c).is_ok()),
        "EventWindowManager::new received malformed configs; \
         caller must run validate_event_window_config first"
    );
    let windows = configs.iter().filter_map(ResolvedWindow::from_config).collect();
    Self { windows }
}
```

This pulls a `core::config::validation` import into `engine`, which is fine — the dependency already flows that way.

### NICE-TO-HAVE

#### N-1 — `ActiveEvent::name` heap-allocates on every active tick

**Source:** Blind Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:223-227`

```rust
if is_active_now {
    active.push(ActiveEvent {
        name: window.name.clone(),       // heap copy per tick per active window
        action: window.action,
        remaining: window.end - now,
    });
}
```

For a 2-hour FOMC window with the event loop calling `current_trading_restriction()` once per tick (typical: hundreds of millions of ticks across the window), that's hundreds of millions of `String` clones on a path the spec explicitly identifies as "called by the consumer side AFTER the circuit-breaker `permits_trading` gate" — i.e. on the trade-evaluation hot path.

The hot-path budget for the breaker gate is documented as <1µs/call; this clone is a `malloc` per active window per call.

**Fix options** (pick one):
1. Change `ResolvedWindow::name` and `ActiveEvent::name` to `Arc<str>`. One-time cost at construction, zero-cost per tick.
2. Have `check_active_events` return references (`Vec<&ResolvedWindow>` or a custom `ActiveEventRef<'_>` borrowing from the manager) and let consumers extract the name only when they need to log/format.
3. Drop `name` from `ActiveEvent` entirely; the `get_trading_restriction` API returns only the restriction, not the name. Callers who want the name can iterate `check_active_events` themselves (which they don't, today). Story spec listed `name` as a field of `ActiveEvent` (Task 2.5) but never required the hot-path consumer to consume it.

I'd take (1). `Arc<str>` is the cheapest fix and matches the "shared identifier" semantics.

#### N-2 — `current_trading_restriction()` calls `check_active_events` (which mutates `is_active`) every call; idempotent in theory, but there's no rate-limit and it drives logging

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:240-246`, `crates/engine/src/event_loop.rs:97-99`

The doc-comment on `current_trading_restriction` warns *"call once per evaluation pass, not once per signal"* (`event_loop.rs:94-96`) — but nothing enforces this. Two calls per tick won't double-log (the `is_active` state machine handles it: state transitions only flip on actual transitions). But two calls per tick WILL execute the hot-path twice, and they WILL allocate twice (per N-1).

The risk: a future signal evaluator with N signals could plausibly call `current_trading_restriction` once per signal "to check before evaluating" — that's N calls per tick. The state machine still works, but the cost is N×.

**Fix options** (pick one):
1. Cache the last-tick restriction on `EventLoop`, recompute only when `clock.now()` (or a tick counter) changes. Adds one `Option<TradingRestriction>` field and one comparison. Eliminates redundant calls.
2. Accept the contract is "consumer's responsibility to call once" and add a `debug_assert!` (or panic-in-debug counter) that tracks the call count per tick.
3. Promote the doc-comment to a crate-level module comment so the contract is more visible.

(1) is the safest and matches the "compute once per tick, fan out the result" pattern already used by `permits_trading`.

#### N-3 — Resumption log reports configured `end - start`, not actual observed-active duration

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:207-217`

```rust
(true, false) => {
    let duration_secs = (window.end - window.start).num_seconds().max(0);
    info!(
        target: "event_windows",
        event_name = %window.name,
        action = ?window.action,
        duration_seconds = duration_secs,
        "event window expired — normal trading resumed"
    );
}
```

`duration_seconds` here is the *configured* duration of the window, not how long the manager observed it as active. If the manager started up *after* a window had already begun (e.g., a process restart 30 minutes into a 120-minute FOMC window), the activation transition will fire on the first tick at 30:00, the resumption transition will fire at 120:00, and the log will say `duration_seconds = 7200` even though the manager only observed 90 minutes.

For the in-window-startup case, the activation log is also misleading: it says `start = 14:00:00, expected_end = 16:00:00` but the actual first observation was at 14:30:00.

This is a story-level decision — the spec language says "When end time passes Then normal trading resumes, resumption logged" without specifying whether the duration is "configured" or "observed". The current code reports configured duration, which is consistent and reproducible across replay/restart but doesn't tell the operator how long *this particular run* was restricted.

**Fix options** (pick one):
1. Track `activated_at: Option<DateTime<Utc>>` on `ResolvedWindow`, populate on the inactive→active transition, compute `now - activated_at` on the active→inactive transition. Reports observed duration. Slightly more state.
2. Document the current semantics in the doc-comment ("`duration_seconds` is the configured window length, not the observed-active duration; in mid-window-startup cases this will overstate") and leave the implementation alone. Cheapest.
3. Log both: `configured_duration_seconds` and `observed_active_seconds`. Most informative, slightly more verbose.

I'd take (1) — observed duration is what an operator wants when reading the resumption line, and the extra `Option<DateTime<Utc>>` is one field per window.

#### N-4 — `info!` log target `"event_windows"` is inconsistent with `circuit_breakers.rs` (which doesn't set a target)

**Source:** Blind Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:200, 211`

```rust
info!(
    target: "event_windows",
    event_name = %window.name,
    …
);
```

Compared with `crates/engine/src/risk/circuit_breakers.rs` which uses default `tracing::info!` without an explicit target. Operators filtering tracing output by target would have to know about both `"event_windows"` and `"futures_bmad_engine::risk::circuit_breakers"` (the default). Minor consistency issue — pick one convention and apply it across `risk/`.

**Fix:** either drop the `target: "event_windows"` (use module-default) or add `target: "circuit_breakers"` to the breaker logs. Trivial.

#### N-5 — `max_by_key` ties: when two `SitOut` events overlap, the later-inserted window wins arbitrarily

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:242-246`

```rust
active
    .iter()
    .map(|e| TradingRestriction::from_action(e.action))
    .max_by_key(|r| r.severity())
```

`max_by_key` returns the *last* element when keys tie. Two overlapping `SitOut` windows produce the same severity, so the second one in the input order wins. Today this is invisible because `get_trading_restriction` only returns the `TradingRestriction` (not the name) — but if a future story exposes the active window name in the API (say, for a metrics endpoint or operator UI), the choice of "which name represents the current restriction?" becomes order-dependent on `EventWindowConfig` declaration order in TOML.

**Fix:** none required for 5.5 (it's a `TradingRestriction` value comparison, not a name comparison). Note for the day this API grows.

#### N-6 — `ResolvedWindow.is_active` mutation requires `&mut self` on `check_active_events`, foreclosing zero-friction concurrent reads

**Source:** Blind Hunter
**Location:** `crates/engine/src/risk/event_windows.rs:189`, struct at 96-104

The `is_active` field is the edge-detect state. Every call mutates it, which is why `check_active_events` and `get_trading_restriction` take `&mut self`. The doc-comment correctly identifies this as single-thread usage from the event loop — fine for V1.

If a future story wants to expose "what's the current restriction?" to a metrics endpoint or an HTTP health-check on a separate thread, the answer requires `&mut self` access to the manager, which means a `Mutex`/`RwLock` wrap or a separate read-only snapshot mechanism. Worth flagging for the metrics/observability epic.

**Fix:** none required for 5.5. Architectural note: if this field were `AtomicBool` instead of `bool` (cheap on x86), the manager could expose `&self` reads and amortise the edge-detect mutation. Not worth the complexity today, but a 30-minute refactor when the read-from-elsewhere requirement lands.

## Recommendation

**APPROVE-WITH-CHANGES.** Mark story `done` after capturing the three SHOULD-FIX items as carryovers. The implementation is correct against AC1-4 and the integration with `event_loop.rs` cleanly preserves the orthogonality from circuit breakers. The fixes are about hardening the surface against future misuse and removing duplicated state (the two `severity` tables) before more code grows on top of either.

**Carryover candidates for `deferred-work.md`:**
1. (S-1) Tighten TOML time-zone semantics: switch `EventWindowConfig.start`/`end` to `DateTime<Utc>` so operators must write the zone, OR add a `time_zone` field to `TradingConfig`. Either way, fail-loud on ambiguous inputs.
2. (S-2) Drop `EventAction::severity` and have `TradingRestriction::severity` be the single source of truth for most-restrictive comparisons. (Or add a parity test if both are kept.)
3. (S-3) Add `debug_assert!` in `EventWindowManager::new` matching `validate_event_window_config`, mirroring the 5.1-S-4 carryover for `CircuitBreakers::new`.
4. (N-1) `ActiveEvent::name` -> `Arc<str>` to eliminate per-tick heap allocations on the hot path.
5. (N-2) Cache `current_trading_restriction()` per tick on `EventLoop` so repeated calls in one evaluation pass don't re-execute `check_active_events`.
6. (N-3) Track `activated_at` on `ResolvedWindow` and report observed-active duration on the resumption log (instead of configured `end - start`). Improves replay diagnostics.
7. (N-4) Pick a single `tracing` `target` convention across `risk/` (either everything explicit or everything default).
8. (N-6) When metrics/observability epic lands, consider `AtomicBool` for `is_active` to allow `&self` reads from another thread.
