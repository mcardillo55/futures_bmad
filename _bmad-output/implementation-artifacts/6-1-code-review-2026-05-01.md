### NICE-TO-HAVE

#### N-1 — No test for `above_volatile && strong_persistence → Rotational` branch

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/threshold.rs:232-238` (classify), test module

The `classify()` decision tree has three outcomes. The test suite covers `Volatile` (high ATR + weak persistence), `Trending` (strong persistence + low ATR), and `Rotational` via the sideways test (low ATR + weak persistence). The fourth quadrant — high ATR AND strong persistence simultaneously — also resolves to `Rotational` (falls through the first two `if`/`else if` branches). This case represents a "trending-but-extremely-volatile" market (e.g., breakout day in momentum) and the spec correctly assigns it to `Rotational` via the `Otherwise` clause. However, no test exercises this branch explicitly. A passing test reader cannot confirm the intent is intentional vs. an accidental fall-through.

The implementation is correct per the spec; the gap is test documentation.

**Fix:** Add a test `volatile_trending_simultaneous_classifies_as_rotational` that feeds bars with ATR > `atr_volatile_threshold` AND persistence >= `directional_persistence_threshold` and asserts `Rotational`.

#### N-2 — `compute_atr` does not guard `r1 = high - low` against inverted bars

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/threshold.rs:155-158`

```rust
let r1 = high - low;  // <-- no abs()
let r2 = (high - prev_close).abs();
let r3 = (low - prev_close).abs();
let tr = r1.max(r2).max(r3);
```

If a malformed bar arrives with `high < low` (impossible for well-formed market data but not enforced by `Bar`'s type invariants), `r1` is negative. `tr` then falls back to `max(r2, r3)`, which underestimates the true range (`|high - low|` is larger than either component when the gap is extreme). In normal operation, every real exchange bar has `high >= low`, so this is not a live-trading risk. The latency is that if `Bar` is ever constructed with inverted fields (e.g., in a fuzz context or a malformed replay message), ATR is silently understated rather than flagged.

**Fix:** Change `r1 = high - low` to `r1 = (high - low).abs()`. One-character change; no runtime cost.

#### N-3 — `capacity() > 0` guard in `push_bar` and `record_atr_sample` is redundant

**Source:** Blind Hunter
**Location:** `crates/engine/src/regime/threshold.rs:242, 249`

```rust
if self.bar_buffer.len() == self.bar_buffer.capacity() && self.bar_buffer.capacity() > 0 {
```

`buffer_capacity` is computed as `atr_period.saturating_add(1).max(warmup_period).max(1)`, so it is always ≥ 1. `VecDeque::with_capacity(buffer_capacity)` will therefore produce a deque whose `capacity()` is ≥ 1 at construction and never decreases. The `&& self.bar_buffer.capacity() > 0` guard can never be false. The same applies to `record_atr_sample` (with `atr_capacity = atr_period.max(1)`).

This is harmless but misleads readers into thinking there is a zero-capacity edge case to worry about.

**Fix:** Remove both `&& ... .capacity() > 0` sub-expressions. Optionally add a `debug_assert!(self.bar_buffer.capacity() >= 1, ...)` in `new()` instead to make the invariant explicit.

#### N-4 — `atr_trending_threshold` is accepted via TOML/config but never consulted; no doc-comment warning for operators

**Source:** Acceptance Auditor
**Location:** `crates/engine/src/regime/threshold.rs:54-58`, `classify()` at line 219-238`

`atr_trending_threshold` has a brief doc comment reading "Currently informational; the V1 classifier uses `atr_volatile_threshold` for the upper bound and treats anything below it as non-volatile." An operator loading a TOML config and setting `atr_trending_threshold = 2.0` (expecting the system to use it as a lower bound for trending confirmation) will observe no effect. The comment explains the design intent but could be misread as the field being active.

The dev's decision to carry the field forward (rather than leave it out of V1) is correct — it avoids a config-schema break when V2 uses it. The risk is silent misconfiguration.

**Fix (minimal):** Add a `// NOTE: not consumed by V1 classifier; reserved for V2` inline comment directly on the field use-site inside `classify()` or on the struct field doc, explicitly warning operators that changes to this value have no effect in the current version.

## Other Observations

- **`#![deny(unsafe_code)]` on `regime/mod.rs`** propagates to `threshold.rs` via the module hierarchy. Satisfies the architecture constraint "no unsafe in engine modules."
