## Verdict

**REQUEST-CHANGES.** The implementation lands the schema, WAL setup, channel wrapper, batching, and acceptance tests cleanly and passes its own test suite. However, the run-loop termination logic in `JournalReceiver::is_disconnected()` is structurally broken and will cause silent event loss and/or premature worker exit in real production scenarios. This must be fixed before merge.

Once the run-loop bug is addressed, the remaining items are minor — the story is otherwise close to merge-ready.

## BLOCKING

### B-1 — `JournalReceiver::is_disconnected()` drops events and exits prematurely

**Location:** `crates/engine/src/persistence/journal.rs:300-306` and the run loop at `:382-440`.

**Problem.** Two bugs compounded:

```rust
fn is_disconnected(&self) -> bool {
    // crossbeam reports Disconnected on recv when senders are gone AND queue empty.
    // We approximate by checking sender count via try_recv on next loop turn.
    self.inner.is_empty() && self.inner.try_recv().is_err()
}
```

1. **Data loss.** `try_recv()` *consumes* a message if one is available. When a producer races with the worker so that an event slips into the channel between the previous `recv_timeout` returning `None` and this check, that event is pulled and immediately discarded. The `Result` is dropped. For a journal whose explicit purpose is durable audit, this is the single worst possible behavior — silent event loss, not surfaced even by a log line.
2. **Premature termination.** The check `self.inner.try_recv().is_err()` returns `true` whenever `try_recv` yields *either* `Empty` *or* `Disconnected`. crossbeam-channel returns `Empty` while senders are alive but the queue is momentarily drained — i.e., it returns `Empty` constantly during normal operation. As soon as the run loop sees one quiet tick, `is_disconnected()` returns `true` and the worker exits even though producers are still healthy.

**Why the test suite did not catch it.** `channel_run_loop_persists_events` has the producer enqueue 20 events synchronously and then immediately drop the sender. The producer is fast enough that the channel rarely empties before the sender is dropped, and even when the worker exits "early" it has already drained all 20 events. The test happens to pass; the bug remains.

**Proposed fix.** Track sender liveness explicitly. The cleanest pattern is to switch the worker to a single `recv_timeout` per loop and treat `RecvTimeoutError::Disconnected` as the terminal signal:

```rust
loop {
    match self.inner.recv_timeout(BATCH_INTERVAL.saturating_sub(last_flush.elapsed())) {
        Ok(event) => batch.push(event),
        Err(RecvTimeoutError::Timeout) => { /* fall through to flush */ }
        Err(RecvTimeoutError::Disconnected) => break,
    }
    // drain any other immediately-available events into the batch
    while batch.len() < BATCH_SIZE {
        match self.inner.try_recv() {
            Ok(more) => batch.push(more),
            Err(TryRecvError::Empty) => break,
            Err(TryRecvError::Disconnected) => { /* defer to outer break after final flush */ ; break }
        }
    }
    // flush + checkpoint logic …
}
```

This eliminates the silent-discard behavior, the premature-exit behavior, and the tight-spin behavior (B-1c below) in one change. The current `JournalReceiver` wrapper can either expose `recv_timeout` returning `Result<EngineEvent, RecvTimeoutError>` directly, or carry the boolean state internally.

**Subsidiary issue (B-1c).** Even ignoring the receiver bug, when `last_flush.elapsed() >= BATCH_INTERVAL` and the channel is empty, the run loop spins: `try_recv` is non-blocking, the empty-batch + interval-elapsed branch resets the timer, and the loop iterates immediately. With B-1 fixed, the natural single-`recv_timeout` shape eliminates the spin. With B-1 unfixed, a separately-spinning worker on a hot CPU is one of the symptoms the parent will see in profiling.

## SHOULD-FIX

### S-1 — `decision_id = 0` sentinel collides with valid id 0

**Location:** `journal.rs:501`

```rust
r.decision_id.unwrap_or(0) as i64
```

The schema declares `trade_events.decision_id INTEGER NOT NULL`. The implementation persists missing decision_ids as `0`. The composite evaluator in story 3.5 generates decision ids by incrementing a counter; the spec doesn't pin the start value, so id=0 is plausibly the very first decision. A reader cannot then distinguish "missing decision_id (NFR17 violation)" from "decision id was 0".

**Proposed fix.** Make the column nullable: `decision_id INTEGER` (drop `NOT NULL`), and persist `r.decision_id.map(|id| id as i64)`. The error log already surfaces the NFR17 violation. Update the relevant tests.

### S-2 — Spec contradiction between AC2 and Task 4.5 around missing decision_id

**Location:** spec `_bmad-output/implementation-artifacts/4-1-sqlite-event-journal.md` lines 14 vs 44.

AC2 says "every trade event includes decision_id" (hard requirement). Task 4.5 says trade events without decision_id "are persisted but emit `tracing::error!`" (soft requirement). The implementation follows Task 4.5. This is internally consistent if the team's intent is "log+persist rather than drop," but the AC text needs updating to match. **Decision needed:** confirm the soft behavior is intended; if so, reword AC2 to "every trade event SHOULD include decision_id; missing values are logged as NFR17 violations and persisted with `decision_id = NULL`."

### S-3 — `JournalError::Sqlite(rusqlite::Error::InvalidQuery)` is misleading when WAL is refused

**Location:** `journal.rs:336-338`

```rust
if !mode.eq_ignore_ascii_case("wal") {
    return Err(JournalError::Sqlite(rusqlite::Error::InvalidQuery));
}
```

The error variant `InvalidQuery` will appear in operator logs as "sqlite error: query is not read-only" or similar — confusing when the actual problem is "the SQLite engine refused WAL mode." Add a dedicated variant:

```rust
#[error("WAL mode unavailable; SQLite reported journal_mode = {actual}")]
WalUnavailable { actual: String },
```

### S-4 — No graceful-shutdown API on `EventJournal` / `JournalSender`

**Location:** `journal.rs:317-440`

Termination relies on every `JournalSender` clone being dropped. A panic in any producer that doesn't unwind cleanly (e.g., a holder inside an `Arc` whose Drop is bypassed) will keep the worker alive forever. There is no explicit `shutdown()` API on the journal.

**Proposed fix or follow-up.** Add either an explicit `Shutdown` event variant or a separate "shutdown" oneshot. Acceptable to defer to the graceful-shutdown story (Epic 8.3) if the dependency is recorded in `deferred-work.md`.

### S-5 — Single-writer-per-file constraint is undocumented

**Location:** `journal.rs:317` (struct doc) — and missing.

SQLite WAL allows concurrent readers but only one writer. Two `EventJournal` instances opened against the same path would compete on busy timeouts. Add a one-line doc constraint: "Exactly one `EventJournal` per database path; multiple writers will produce `SQLITE_BUSY` even with WAL." This becomes important when an operator tools script (e.g., a debug query) opens the same DB in parallel.

## NICE-TO-HAVE

### N-1 — `u64` → `i64` casts at `as_nanos() as i64`, `decision_id as i64`, `order_id as i64`

**Location:** `journal.rs:499-507, 517, 519, 531, 544, 556`

For nanosecond timestamps the wraparound boundary is year 2262 — a non-issue today but worth a single-comment acknowledgment. For `u64::MAX` decision_id values from a counter that lives long enough, the wrap to negative breaks numeric SQL queries. Add a `// NOTE: SQLite stores INTEGER as i64; values > i64::MAX wrap` near the casts.

### N-2 — `unchecked_transaction()` skips the nested-transaction safety check

**Location:** `journal.rs:349, 444, 473`

Currently safe because no caller holds a transaction across these calls. Future maintainers won't notice they've broken the invariant until a runtime error fires. Either prefer `conn.transaction()` (which returns a `Transaction` wrapper enforcing single-level use) or document the invariant in a struct comment.

### N-3 — `wal_checkpoint(PASSIVE)` busy=1 is silently ignored

**Location:** `journal.rs:451-465`

When the checkpoint can't make progress because a reader is holding the WAL open, SQLite returns `busy=1, log=N, ckpt=0`. The current code logs `wal_pages` and `checkpointed` but does not surface the busy state. A `warn!` (or `debug!`) when `busy=1` and `ckpt < log` would help diagnose unbounded WAL growth in production.

### N-4 — `flush_batch` error mid-loop loses subsequent batch contents

**Location:** `journal.rs:436-440`

If `flush_batch` returns `Err`, `run` returns early. Any events drained from the channel into `batch` after that point are lost (well, none are drained because we returned, but events drained *into* the failed batch and not yet flushed are also lost). Acceptable in v1; document the trade-off as a comment so a future operator-alerting story knows to wire a supervisor restart on journal-error events.

### N-5 — `DEFAULT_DB_PATH = "data/journal.db"` is `cwd`-relative

**Location:** `journal.rs:58`

If the engine is launched by systemd / launchd from a different working directory than dev runs, the journal will land somewhere unexpected. Either resolve to an absolute path via env var (`FUTURES_BMAD_DATA_DIR`) at `open_default()` time, or document the cwd dependency.

### N-6 — `channel_backpressure_drops_when_full` 50ms threshold may flake on CI

**Location:** `journal.rs:715-718`

```rust
assert!(elapsed < Duration::from_millis(50), ...);
```

Shared CI runners under load can pause a thread > 50 ms. Consider 250 ms — the test still proves "did not block" without flake risk.

### N-7 — No real-capacity stress test (8192 events through the actual channel)

**Location:** test module.

The backpressure test uses a 2-slot channel; useful but doesn't exercise the real production path. A test that pushes 9000 events through `EventJournal::channel()` against a slow worker (e.g., one with a sleep injected) would prove the AC for `JOURNAL_CHANNEL_CAPACITY = 8192`. Reasonable to defer.
