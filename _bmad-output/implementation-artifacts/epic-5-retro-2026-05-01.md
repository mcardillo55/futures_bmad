## Successes & Strengths

1. **Hot-path discipline is now reflexive.** Every story passed its allocation budget. `permits_trading()` is `#[inline]` and collapses to a boolean reduction with zero allocation on the happy path. `try_send` (non-blocking) became the canonical pattern for ALL async producer→consumer plumbing — journal events (5.1), alerts (5.6), stop-loss-preserving cancellation lists. No hot-path blocking introduced anywhere in the epic. Benchmarks (`permits_trading` / `permits_trade_evaluation`) clear 100k iterations in <100ms.

2. **Type-enforced state machines.** Gate-vs-breaker distinction in 5.1 is enforced by the compiler via `clear_gate -> Result<_, GateClearError::NotAGate>`. Manual-reset breakers cannot be gate-cleared by accident. 5.5 layered the same idea on event-window severity (`SitOut > ReduceExposure > DisableStrategies`). The Epic 4 lesson "exhaustive matching beats wildcard `_`" stuck — `BreakerType::category()` is exhaustive over 10 variants and adding an 11th would fail to compile until all consumers are updated.

3. **Carryover-fix discipline is institutionalized.** Story 5.2's four entry points (`check_position_size`, `update_daily_loss`, `record_trade_result`, `record_trade`) update the `DenialReason` numeric-context fields BEFORE the trip check, organically resolving Story 5.1's S-1 finding. The carryover-fix pattern from Epic 4 (`366fcf7`) has matured into "the next story's surface naturally absorbs the prior story's open SHOULD-FIX items, no separate cleanup commit needed."

4. **Test discipline at component boundaries is tight.** ~110 new tests, boundary cases named explicitly (`start_instant_is_inclusive`, `daily_loss_trips_at_exact_threshold`, `manual_reset_does_not_zero_consecutive_counter`, `panic_mode_cancellation_excludes_stop_loss_orders`). Idempotency tested systematically — re-trips don't double-journal. Determinism verified for replay (`with_simclock_event_activation_is_deterministic`).

5. **Story velocity.** Six stories shipped in a single day. Framework (5.1) + four integrators (5.2–5.4, 5.6) + orthogonal addition (5.5). Pattern of parallel-then-merge worked at the velocity layer; the breakdown was at the merge integrity layer (see Challenges).

6. **Clippy clean, `unsafe` ban respected.** Six for six. `#![deny(unsafe_code)]` propagates through `risk/mod.rs` to all children. No new `unsafe` despite working with threads (5.6 alert spawning), file I/O (5.6 alert log), and cross-thread channels.

7. **5.5 was a particularly clean shipment.** Right-open `[start, end)` interval design prevents back-to-back-window double-counting; edge-triggered logging emits once per state transition (no per-tick spam); `clock.wall_clock()` used everywhere so `SimClock`-driven replays are deterministic; the orthogonality from circuit breakers is enforced by code path AND integration test (`active_window_does_not_affect_data_quality_or_buffer`). Three SHOULD-FIX, no BLOCKING.

## Challenges & Growth Areas

### Worktree-merge-at-end produced partially-stale integration code (root cause of 5.3 BLOCKING)

This is the load-bearing finding of the retro.

**The pattern:** Stories 5.2–5.6 were developed in parallel worktrees, several of which were rooted on pre-5.1 main. Story 5.5's dev notes acknowledge this directly: *"Rebasing mid-story: initial worktree base was on 4.5; had to rebase onto main to pick up story 5.1 before starting implementation."* Story 5.3's code review is more explicit: *"This story was developed against a stale main (pre-5.1 merge) and surgically rebased into the post-5.1+5.2+5.4+5.6 framework on integration. The 5.3 worktree's original 414-line `circuit_breakers.rs` scaffold was DROPPED in favour of porting `check_position_anomaly` + `AnomalyCheckOutcome` onto Story 5.1's full 10-breaker framework. The 5.3 worktree's bespoke `PanicContext` was REPLACED with Story 5.6's `PanicContext`."*

**The consequence:** When 5.1's framework landed, the integrators' wiring shape changed, but the rebase was a code-port — not a redo of the integration. Tasks 4 and 5 of 5.3 were checked ✅ in the original worktree (where the wiring against the stub framework was complete), but the rebase didn't carry the wiring forward against the new framework. Result: two BLOCKING items (`check_position_anomaly` / `handle_anomaly` have no production caller; `CircuitBreakers::permits_trading()` doesn't consult panic mode) shipped.

**The implication for "task ✅ but no call site" pattern.** This pattern appeared in five of six stories (5.1 `trip_breaker` no `orders_to_cancel` call site; 5.2 Task 5 reframed as "API callable"; 5.3 Tasks 4/5 BLOCKING; 5.4 Task 3.6 unexecuted; 5.6 P1/P2/P3 patches). I (Amelia) initially attributed this to checkbox discipline. Michael's input reframed it correctly: **the checkbox was honest in the worktree. The merge silently invalidated it.** No human spot-check caught it because there's no merge-time integration test that says "this newly-shipped API has a non-test caller somewhere in the binary."

**Resolution:** Adopt a **framework-first merge order**. When stories within an epic share a framework (5.1 here), framework merges first; integrators rebase onto its merged main BEFORE declaring their tasks done. Parallelism is preserved during framework dev — but the integrator's "task ✅" assertion only counts post-rebase. (See Action Item P-1.)

### Live-Trading Exit Gates checklist was Epic 4's biggest commitment and we did not implement it

Epic 4 retro identified 3 critical-path items (4-5 S-1, S-2, S-3) and committed to "maintain a live-trading exit gate checklist" so future epics couldn't silently regress on the gate. Epic 5 added **at least 3 more** (5.3 B-1, 5.3 B-2, 5.4 P1) plus carryover SHOULD-FIX items (5.1 S-2, S-3, S-4). Without a consolidated checklist, the regression was invisible until this retro pulled it together manually from six review files.

**Net live-trading readiness has decreased, not increased.** Epic 5's framework is a structural advance (5.1's circuit-breaker plumbing IS the canonical halt source we said we needed for 4-5 S-2), but the integration wiring slipped. Story 8.2 inherits more pre-work than the Epic 4 retro projected.

**Resolution:** Implement the checklist this retro. Section in `deferred-work.md` lists ALL open live-trading blockers in one place. Updated at every story review. Reviewed at every retro. (See Action Item P-2.)

### Docstring-as-contract was the load-bearing safety mechanism in three places

- **5.4 D1.** `permits_trading()` vs `permits_trade_evaluation()` have identical signatures, distinguished only by docstring severity ("CRITICAL: fee staleness gate ONLY blocks trade evaluation"). A future caller picks the wrong one with no compiler help. The split-API contract lives entirely in tests and docstrings — outside the risk module there is exactly zero invocation of either, so the choice is deferred to whoever wires it up.
- **5.2 S-2.** `update_daily_loss(session_cumulative)` doc says "session-cumulative, NOT delta", but the API will silently overwrite if deltas arrive.
- **5.5 N-1.** `current_trading_restriction()` doc says "call once per evaluation pass"; API can't enforce this — repeated calls allocate.

This is fragile. Tomorrow's consumers have only the docstring as a guide. (Mitigations are debt items, not action items — see Tech Debt section.)

### Acceptance criteria vs task semantics drift

5.2's AC says "would exceed" but the four breaker entry points use **three different comparison flavors** (`>=` vs `>` vs `<= -limit`), matching explicit task requirements (1.3, 3.3, 4.3) but contradicting the synced AC text. The implementation is faithful to the tasks; the tasks are faithful to the architecture; the AC is loose. This came out at code review (5.2 S-1), not at story-creation review.

**Resolution:** AC semantic-precision pass at story creation. Threshold tasks must explicitly state comparison semantics with examples. (See Action Item P-4.)

### Claude usage-limit hit mid-sprint

Operational pain: the model usage limit was hit during the Epic 5 burst, costing time and forcing context re-establishment. Not a fundamental finding about Epic 5's content, but a real practical drag.

**Resolution:** Operational pacing pattern documented as an action item owned by Michael. (See Action Item P-5.)

## Action Items

### Process Improvements

**P-1. Framework-first merge order (parallelism preserved).**
Within an epic, identify the framework story (e.g., 5.1) up front. Framework merges first; integrators rebase onto its merged main BEFORE declaring tasks done. Parallel dev still allowed during framework development — but the integrator's "task ✅" assertion only counts post-rebase against merged-framework main.
**Why:** The 5.3 BLOCKING items trace directly to a worktree built against pre-5.1 main; the rebase was a code-port, not an integration redo.
**Owner:** Amelia (workflow discipline).
**Success:** Epic 6 retro shows zero "BLOCKING from worktree-stale-main" findings.
**Deadline:** In effect for Epic 6 onward.

**P-2. Live-Trading Exit Gates checklist.**
Implement Epic 4's unfulfilled commitment. New `## Live-Trading Exit Gates` section in `deferred-work.md`, consolidating all open live-trading blockers in one place. Updated at every story review. Reviewed at every retro.
**Why:** Epic 4 closed with 3 gates; Epic 5 silently added 3+ more. We can't see regression unless they're collected.
**Owner:** Charlie (review discipline). v1 drafted as part of this retro.
**Success:** Section exists, lists all current items, Epic 6 retro references it.
**Deadline:** Drafted with this retro (2026-05-01); maintained ongoing.

**P-3. Task checkbox contract.**
A task may be checked ✅ ONLY IF (a) the implementation has a non-test caller, OR (b) the task line begins with `[DEFERRED to Story X.Y — reason]`. No reframing AC text to make a deferral look like completion.
**Why:** Five of six Epic 5 stories had "Task N: wire into event loop ✅" with no call site. With P-1 in place this should largely self-correct, but P-3 is the explicit norm.
**Owner:** All developers.
**Success:** Spot-check 3 checked tasks per epic at retro, verify call site or `[DEFERRED]` tag.
**Deadline:** In effect immediately.

**P-4. AC semantic-precision pass at story creation.**
Threshold tasks must explicitly state comparison semantics (`>=` vs `>`, inclusive vs exclusive) with examples (e.g., "max_position_size = 2 blocks positions of exactly 2 or larger"). Architect reviews AC before dev starts.
**Why:** 5.2's `>=` semantics surprised the code reviewer at review time, not at story creation.
**Owner:** Story creator + Winston (AC review).
**Success:** Epic 6 stories have explicit comparison semantics on every threshold; no semantic-drift findings at code review.
**Deadline:** Apply when creating Epic 6 stories.

**P-5. Claude usage-limit operational pattern.**
Plan epic-completion bursts to start early in the cycle window. For long story-dev sessions, prefer Sonnet for routine dev and reserve Opus for review/retro. Memory files keep cross-session context cheap.
**Why:** Hitting the limit mid-sprint cost time and context.
**Owner:** Michael (project lead).
**Success:** Epic 6 burst doesn't hit the limit, or has a documented graceful-continuation pattern.
**Deadline:** Apply during Epic 6 work.

## Closure

Epic 5 delivered the canonical risk framework: 10 breaker/gate types, gate-vs-breaker type-enforcement, panic-mode escalation surface, infrastructure breakers, event-aware trading windows, and operator alerting — all in a single day's burst of six stories. The framework is structurally sound, hot-path-clean, and well-tested.

The integration story is incomplete by design (the architecture maps live-system wiring to Story 8.2) but partially incomplete by accident — the worktree-merge-at-end pattern produced rebased code where task checkboxes survived but call-site wiring did not. The 5.3 BLOCKING items are the visible failure mode; the action items above (P-1 framework-first merge order; P-2 live-trading exit gates checklist; P-3 task checkbox contract) address it systemically.

Carry forward into Epic 6: tight component discipline, slipping integration discipline, plus a now-explicit checklist mechanism to keep the latter from compounding silently.
