## Acceptance Criteria Audit

- **AC1** — `max_position_size`: trade would exceed ⇒ gate blocks; auto-clears when within limits: **MET.**
  - `check_position_size(current_position: i32, ts) -> Option<CircuitBreakerEvent>` (`circuit_breakers.rs:650-684`). `abs(current_position) >= max_position_size_limit` ⇒ trip via `trip_breaker(BreakerType::MaxPositionSize, …)`. Below the limit and previously tripped ⇒ `clear_gate(MaxPositionSize, ts)` returns the canonical `Cleared` event with duration logging.
  - Tests: `max_position_size_gate_trips_and_auto_clears`, `max_position_size_uses_absolute_value`, `max_position_size_clears_via_update_gate_conditions`, `position_size_re_trip_is_idempotent`. Coverage of the trip / auto-clear / idempotency / short-side / pre-existing API path. Solid.
  - Minor: AC1 says "trade would exceed" but the implementation trips at "reach OR exceed" (`>=`). Story task 1.3 explicitly says `>=`, so the implementation matches the task. The architecture spec line 478 says "Position exceeds limit" — also ambiguous. The test uses `max_position_size = 2` and trips at `current = 2`, i.e., AT the limit, not past it. **This is a deliberate semantic choice but the AC text is loose.** See N-1.

- **AC2** — `max_daily_loss_ticks`: cumulative loss exceeds threshold ⇒ daily-loss breaker activates, session over, manual reset, persists across reconnections: **MET.**
  - `update_daily_loss(realized_pnl: i64, unrealized_pnl: i64, ts)` (`circuit_breakers.rs:703-733`). `total = realized.saturating_add(unrealized)`; trip when `total <= -daily_loss_limit`. Once `Tripped`, future calls update only the cached `daily_loss_current` and never re-evaluate the trip — so recovery does not auto-clear. Manual reset is via `reset_breaker` (= process restart in V1). Persistence across reconnections: state lives on the in-memory `CircuitBreakers` struct, which is reconstructed from the journal on restart by the post-5.1 framework — the breaker stays tripped because the trip event is in the journal. (Persistence is a journal-replay concern, not a 5.2 concern; the journal write is on the 5.1 side.)
  - Tests: `daily_loss_breaker_trips_at_cap`, `daily_loss_trips_at_exact_threshold`, `daily_loss_does_not_auto_clear_on_recovery`, `daily_loss_aggregates_realized_and_unrealized`, `update_daily_loss_saturates`. The "stays tripped on recovery" assertion is the load-bearing one and is explicit. Solid.

- **AC3** — `max_consecutive_losses`: N consecutive losses ⇒ breaker activates; counter resets on winning trade (NOT on reset); requires manual reset: **MET.**
  - `record_trade_result(is_winner: bool, ts)` (`circuit_breakers.rs:751-782`). On winner: counter unconditionally resets to 0 (regardless of breaker state). On loser: `saturating_add(1)`; if not already tripped and counter `>= consecutive_loss_limit`, trip. Once tripped, the counter continues to increment via `saturating_add` for forensic context (no re-trip).
  - The architecture-spec invariant "manual reset does NOT zero the counter — winning trade does" is asserted by two complementary tests: `manual_reset_does_not_zero_consecutive_counter` (operator resets at counter=3 with no intervening winner ⇒ next loss takes counter to 4 ⇒ immediate re-trip) and `consecutive_loss_counter_resets_on_win_even_when_tripped` (winner zeroes counter even when breaker is currently `Tripped`; subsequent reset+loss starts fresh from 1). This is the architecturally load-bearing piece and the dev nailed it.
  - Tests also cover the standard threshold trip (`consecutive_loss_breaker_trips_at_limit` — 3rd loss when limit=3) and increment-then-win (`consecutive_loss_counter_increments_then_resets_on_win`).

- **AC4** — `max_trades_per_day`: count exceeded ⇒ breaker activates, requires manual reset: **MET.**
  - `record_trade(ts)` (`circuit_breakers.rs:793-812`). `trade_count.saturating_add(1)`; if not already tripped and `trade_count > max_trades_limit`, trip. The "exceeds" condition is strict greater-than: `max_trades_per_day = 30` allows 30 trades; the 31st trips. Manual reset preserves the count (dev's intent: a manual reset clears only the breaker state; count is by-design a session-life counter).
  - Tests: `max_trades_breaker_trips_when_exceeded` (counts to 30 without tripping, 31st trips), `max_trades_count_persists_across_manual_reset` (reset → count preserved → next trade re-trips). Solid.

## Adversarial Findings

### SHOULD-FIX

#### S-1 — Inconsistent trip semantics across the four breakers (`>=` vs `>` vs `<= -limit`)

**Source:** Edge Case Hunter + Acceptance Auditor
**Location:** `circuit_breakers.rs:663` (MaxPositionSize, `abs_pos >= limit`), `:720` (DailyLoss, `total <= -daily_loss_limit`), `:770` (ConsecutiveLosses, `count >= limit`), `:800` (MaxTrades, `trade_count > max_trades_limit`)

The four breakers use three different comparison flavors:

| Breaker            | Trip condition                  | "Exceeds" interpretation |
|--------------------|---------------------------------|--------------------------|
| MaxPositionSize    | `abs_pos >= limit`              | Reach-or-exceed          |
| DailyLoss          | `total <= -daily_loss_limit`    | Reach-or-exceed (loss)   |
| ConsecutiveLosses  | `consec_count >= limit`         | Reach-or-exceed          |
| MaxTrades          | `trade_count > max_trades_limit`| Strictly greater         |

This means: if `max_position_size = 2`, position 2 trips; if `max_consecutive_losses = 5`, the 5th loss trips; if `max_trades_per_day = 30`, the 31st trade trips (not the 30th). The story tasks 1.3 / 3.3 / 4.3 explicitly call for `>=` on position-size and consecutive-losses but `>` on max-trades — so the inconsistency is faithfully implemented per the tasks. But the AC text is the same for all four ("when count exceeds", "when N consecutive losses"). An operator reading "max 30 trades" in their TOML will reasonably expect either (a) 30 trips OR (b) 31 trips — and won't expect the same word "exceeds" to mean different things across siblings.

The exact-threshold test `daily_loss_trips_at_exact_threshold` documents the daily-loss boundary explicitly. The position-size and consecutive-losses tests document `>=`. The max-trades test documents `>`. So coverage is honest about the semantics — but the inconsistency is not called out in the docstrings or in the user-facing config docs.

**Fix options** (pick one):
1. Pick a single semantic ("trips on the Nth event that would breach", i.e., `>` for all four) and refactor the three `>=` sites. Matches the AC's "exceeds" reading. **Breaking change** for any downstream consumer that already relies on the `>=` semantic — but no such consumer exists yet, so the cost is just refactoring tests.
2. Document the per-breaker semantics in the docstrings AND in `crates/core/src/config/trading.rs` so an operator can read "max_position_size = N means N trips" vs "max_trades_per_day = N means the (N+1)th trade trips" without surprise.

I'd take option 2 today (less risk of churn during a partial-deliverable epic) and put option 1 on the epic-5 retro list. Either way, this needs to be acknowledged before the trade-loop integration ships.

#### S-2 — `update_daily_loss` doc does not warn that the args are session-cumulative, not deltas

**Source:** Blind Hunter + Edge Case Hunter
**Location:** `circuit_breakers.rs:686-732`

The implementation overwrites `self.daily_loss_current` with each call:

```rust
let total = realized_pnl.saturating_add(unrealized_pnl);
self.daily_loss_current = total;
```

This is correct ONLY if the caller passes session-cumulative P&L. If a future event-loop wiring (story 8-2) mistakenly passes the per-fill _delta_ instead, the breaker will silently never trip — every call simply overwrites `daily_loss_current` with the most recent fill's P&L. The doc says:

> `daily_loss_current = realized_pnl + unrealized_pnl`. The breaker trips when the current loss is at or beyond the configured cap

It does not explicitly say "callers MUST pass cumulative session P&L, not deltas". The architecture spec uses "cumulative loss" but the docstring doesn't mirror that contract.

The test `daily_loss_breaker_trips_at_cap` shadows the issue: it calls `update_daily_loss(-999, …)` then `update_daily_loss(-1001, …)`. -1001 already implies a cumulative step from -999 to -1001 — but if a wiring engineer wrote `update_daily_loss(-2, 0, ts)` thinking "I lost 2 ticks on this fill", that single call would set `daily_loss_current = -2` and the cumulative -1001 would be silently lost.

**Fix:** rewrite the docstring to start with **"Pass the session-cumulative realised + unrealised P&L, NOT the per-fill delta."** Add an example: `breakers.update_daily_loss(session_realised_pnl, current_unrealised_pnl, now);`. Optionally rename the parameters from `realized_pnl, unrealized_pnl` to `session_realised_pnl, current_unrealised_pnl` to make the contract self-documenting.

### NICE-TO-HAVE

#### N-1 — AC text says "would exceed", impl trips at "reach OR exceed"; minor doc drift

**Source:** Acceptance Auditor
**Location:** `5-2-position-and-loss-limit-breakers.md:14` (AC1) vs `circuit_breakers.rs:663`

AC1 in the story file: "When trade would exceed Then max position size gate blocks". The implementation trips when the position REACHES the limit (`abs_pos >= limit`). For `max_position_size = 2`, a position of exactly 2 trips. So "would exceed" in the AC is shorthand for "would reach or exceed". This is fine for a trader's mental model — most operators read "max 2" as "2 is allowed but not 3" — but the literal AC text says "exceed" (i.e., 3+, with 2 still permitted). See S-1 for the cross-breaker semantic question.

**Fix:** none required for 5.2; the story task 1.3 explicitly demands `>=`. Capture the AC vs task text drift so the next story-spec authoring pass tightens the AC.

#### N-2 — `record_trade_result` increments the counter even when already tripped, with no upper bound on observable count

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:760-768`

When the breaker is already `Tripped` and a loss arrives, the counter is incremented (`saturating_add(1)`) and the early-return path is taken. The implementation comment says:

> Already tripped — keep counting for the contextual field but don't re-trip.

Two questions:
1. Why keep counting? The DenialReason context (`current: u32`) is unbounded above the limit. An operator seeing "27 consecutive losses, limit 5" learns nothing more than "27 ≥ 5 ⇒ breaker tripped"; the magnitude past the limit is forensic noise. Forensic data should live in the journal records, not a runtime counter that can grow to `u32::MAX`.
2. The `saturating_add` will clamp at `u32::MAX` after ~4 billion losses, which is pleasantly impossible — but the framework would also accept a winning trade after that and snap the counter back to zero, then re-evaluate. This is fine but is not tested.

**Fix:** none required. Behavior is harmless. Captured for the epic-5 retro: the "keep counting after trip" semantic could be tightened by simply not incrementing past the trip point — the journal events are the durable record.

#### N-3 — `check_position_size` updates `requested_position_size` even on the no-op path; the field name is misleading

**Source:** Blind Hunter
**Location:** `circuit_breakers.rs:655-658`

The field is named `requested_position_size` (implying "size of the order being requested"), but `check_position_size` writes the _current_ position into it on every call:

```rust
let abs_pos = current_position.unsigned_abs();
self.requested_position_size = abs_pos;
```

Then the DenialReason rendering in `collect_denials` says:

```rust
DenialReason::MaxPositionSizeExceeded { requested: self.requested_position_size, limit: ... }
```

So the operator-facing message would read: `"requested position size 4 exceeds max 2"` — but `4` is the CURRENT position, not what the operator requested. The semantic is wrong. The original 5.1 review (S-1) noted `requested_position_size` as one of the un-settable numeric-context fields; 5.2 sets it but conflates "current" with "requested".

**Fix options:**
1. Rename the field and the `DenialReason` variant context: `requested` → `current` (in the breaker, the variant, and the display string).
2. Have `check_position_size` accept TWO arguments: current position AND requested order size, and set the field to the would-be position after the requested order. This is closer to the original semantic but adds a parameter to the API.

Option 1 is correct — at gate-trip time, the size that matters is the current net position (the trip is on the position itself, not on a particular order). The `requested` framing is a leftover from the AC text "trade would exceed". Captured for the epic-5 retro / 5.1 carryover cleanup.

#### N-4 — `daily_loss_breaker_trips_at_cap` test name vs assertion mismatch

**Source:** Blind Hunter
**Location:** `circuit_breakers.rs:1239-1255`

The test name says "trips at cap" but uses `-1001` (i.e., 1 tick past the cap of 1000). The "trips at cap" boundary case is in a separate test (`daily_loss_trips_at_exact_threshold`) using `-1000`. The first test should probably be renamed `daily_loss_breaker_trips_when_loss_exceeds_cap` to be precise.

**Fix:** trivial rename. No behavioral change.

#### N-5 — No test for `record_trade` re-trip idempotency

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:793-812`

The `record_trade` early-return on `matches!(self.max_trades_state, BreakerState::Tripped)` is symmetric to the `record_trade_result` and `update_daily_loss` once-tripped paths. There's a test `position_size_re_trip_is_idempotent` covering the position-size case, but no equivalent for `record_trade` (or `update_daily_loss` or `record_trade_result`). The `max_trades_count_persists_across_manual_reset` test indirectly exercises it (records 31 trades, all subsequent calls hit the early-return), but no explicit "no duplicate journal event on already-tripped re-trigger" assertion.

**Fix:** add three small tests (one per breaker) asserting that `try_recv()` on the journal channel is `Err` after a no-state-change re-call. Cheap.

#### N-6 — `record_trade_result` accepts `bool` — no flat-trade ("scratch") handling

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:751-755`

`record_trade_result(is_winner: bool, ts)` only models two outcomes: winner (counter resets) or loser (counter increments). What about a flat trade (`pnl == 0`) — a scratch where the position was opened and closed at the same price? The current API would force the caller to choose: treat scratches as winners (reset the counter, possibly hiding a losing streak that bracketed the scratch) or as losers (increment, falsely tightening the streak counter).

The architecture spec is silent on scratches. Most prop-trading rule sets count a scratch as "neither" — it shouldn't increment the loss counter, but it also shouldn't reset it. The current API can't express this nuance.

**Fix:** none required for 5.2 — this is a spec-level question. Capture for the epic-5 retro: should `record_trade_result` accept `enum TradeOutcome { Win, Loss, Scratch }`?

## Other Observations

- **Story 5.1 carryover S-1 is naturally resolved by 5.2.** S-1 asked for per-breaker numeric-context setters so `DenialReason` reports truth, not zero. The four 5.2 entry points (`update_daily_loss`, `record_trade_result`, `record_trade`, `check_position_size`) ARE those setters — each updates its cached field BEFORE the trip check. The `DenialReason` context will now reflect the most recent observed value at trip time. Good outcome from a partial-deliverable epic structure.
- **Story 5.1 carryover S-2 (GateConditions::default footgun) is NOT addressed.** The `max_position_size_clears_via_update_gate_conditions` test in 5.2 actually relies on `..Default::default()` for the other gate fields — exactly the pattern S-2 flagged as a footgun. As long as 5.5 (event-aware trading windows) and the eventual production wiring don't construct `GateConditions::default()`, this remains a latent risk. Capture it on the epic-5 retro list as carryover.
- **Story 5.1 carryover S-3 (`reset_breaker` does not journal, does not zero numeric context) is NOT addressed.** The 5.2 manual-reset tests (`manual_reset_does_not_zero_consecutive_counter`, `max_trades_count_persists_across_manual_reset`) actually _depend_ on `reset_breaker` not zeroing the counter — by design, per the architecture spec. So S-3's "should zero numeric context" recommendation is partially obsoleted: for `consecutive_loss_count` and `trade_count`, the non-zeroing is now load-bearing. For `daily_loss_current` and `requested_position_size`, S-3's concern still stands.
- **Hot-path discipline preserved.** All four entry points are integer-only. No heap allocation on the no-state-change path: `format!` is invoked only on the cold trip-event-construction path. The `#[deny(unsafe_code)]` declaration at the top of `circuit_breakers.rs` is preserved. The `#[inline]` annotation on `permits_trading` (5.1) remains.
- **Tests use `i64` literals like `-1001` directly without quarter-tick conversion** — fine for unit tests since `max_daily_loss_ticks` is itself a quarter-tick `i64` and the test config sets it to `1000`. Just noting this for clarity.
- **`#[doc(hidden)] pub` accessors** for the three numeric counters (`daily_loss_current`, `consecutive_loss_count`, `trade_count`) follow the existing 5.1 precedent (`set_buffer_occupancy`). Acceptable encapsulation pattern.
- **Sprint status updated correctly** — 5-2 → `review`. No drift.
