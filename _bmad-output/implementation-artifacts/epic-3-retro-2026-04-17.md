## Successes & Strengths

1. **Learning velocity within the epic** — review findings dropped from 6 (3.1) to 5 (3.2) to 0 (3.3, 3.4, 3.5). Story 3.3 dev notes explicitly cite learnings applied from 3.1 (snapshot timestamps, early-return state clearing). Lessons weren't just documented — they were baked into subsequent implementations.

2. **Consistent defensive pattern established** — every signal follows: is_tradeable gate -> NaN/Inf guard -> zero-denomination check -> early-return clears state. Became the canonical template by Story 3.3.

3. **Zero-allocation hot-path discipline** — all signal computation runs on fixed-size arrays with no heap allocation. Concrete types in SignalPipeline with zero vtable overhead. Integer arithmetic in FeeGate for fee computation.

4. **Testkit investment from Epic 1 paid off** — SimClock and OrderBookBuilder made test setup trivial across all 5 stories. 56 new tests added with minimal boilerplate.

5. **Critical overflow catch in VPIN (3.2)** — review caught u64 subtraction underflow in bucket trimming and bucket_size=0 infinite loop. Both could cause real production failures.

6. **Causality tracing foundation** — monotonic decision_id from CompositeEvaluator enables full trace from signal values through composite score to order events. Ready for Epic 4's order execution.

7. **Spec quality drove zero-debug implementation** — exact formulas, epsilon tolerances, and enumerated edge cases in acceptance criteria eliminated ambiguity. All 5 stories' tests passed on first run.

## Challenges & Growth Areas

### SignalSnapshot Debt Accumulation (Deferred in Stories 3.1 and 3.2)

SignalSnapshot struct (defined in core during Epic 1) doesn't capture full signal state:
- Story 3.1: `update_count` not in snapshot, needed for full determinism
- Story 3.2: bucket state not in snapshot, needed for replay verification

**Impact:** Not blocking for Epics 4-6. Becomes critical path for Epic 7 (Historical Replay & Paper Trading) where determinism verification depends on comparing full signal state at every step.

**Root cause:** SignalSnapshot was designed as a minimal struct during Epic 1 before the full range of signal state was understood. Cross-cutting change deferred because it affects all signal implementations.

### Constructor Spec Deviation (Story 3.1)

ObiSignal constructor added max_spread parameter not in the original spec. Necessary because OrderBook::is_tradeable() requires it. The dependency was implicit in the architecture doc.

**Root cause:** Architecture didn't fully specify constructor signatures for Signal implementations. Worth watching in future stories.

### Skipped Epic 2 Retrospective

No retro was run after Epic 2 (Market Data Ingestion & Recording). Lessons from Rithmic integration, SPSC buffer patterns, Parquet recording, and gap detection are undocumented.

## Key Insights & Learnings

1. **Compounding improvement is real and measurable** — 11 review findings in the first 2 stories, 0 in the last 3. Explicit forward-application of review learnings (cited in dev notes) drives this improvement.

2. **Canonical patterns reduce cognitive load** — once the defensive signal pattern was established, new signals were implemented cleanly without re-discovering edge cases.

3. **Spec precision eliminates debug time** — exact formulas with epsilon tolerances meant zero ambiguity. Every story's tests passed on first run.

4. **Deferred items accumulate silently** — SignalSnapshot was deferred twice. Without tagging the epic where it becomes critical (Epic 7), it could be forgotten until it's too late.
