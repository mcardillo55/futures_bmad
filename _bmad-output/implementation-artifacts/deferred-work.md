## Deferred from: code review of story-4.5 (2026-04-30)

- S-1 (carryforward into 8-2): In-flight-fill / broker-query race during periodic reconciliation produces false-positive `SideOrQuantity` mismatches. The current `reconcile(&[BrokerPosition])` API has no as-of-T0 sequence number or quiesce hook. Startup-trigger case is safe; **periodic during live trading WILL trip on this race during any active fill burst**. Resolution: extend the tracker API (or add a wrapper in the lifecycle module) with either (a) an as-of timestamp tag on the broker view, retry-rather-than-trip when local has fills since T0 touching a mismatched symbol, or (b) a `quiesce()` hook the lifecycle drives before each periodic reconciliation. Must land alongside the periodic-reconciliation wire-up in 8-2. [`crates/engine/src/order_manager/tracker.rs:255-316`]
- S-2 (epic-4 exit gate): `trading_halted` is set on the tracker after a mismatch, but `OrderManager::submit_order` never consults it. AC3's "trading halted" effect is structurally unwired today — a grep finds the flag referenced only in `tracker.rs`. Resolution: either wire `submit_order` to consult `tracker.trading_halted()` (shared-Arc<AtomicBool> or explicit parameter), or route the mismatch through the same `submissions_paused` gate the Uncertain path already uses. Must land before 8-2 enables live trading. [`crates/engine/src/order_manager/mod.rs:306-399`, `crates/engine/src/order_manager/tracker.rs:380-381`]
- S-3 (epic-4 exit gate; deferral CONTESTED — see 4-3 S-2 / 4-4 S-2 history): Partial-entry SL/TP oversizing in `BracketManager::on_entry_fill` remains a warn-log only. On a Buy-5 entry partial-filled-3, SL fires at 5 → flatten 3 + naked short 2 — NFR16 violation. Dev defers as "BracketManager scope, not PositionTracker scope"; reviewer pushes back: 4-5 was the live-trading gate per the original 4-3 review. ~30 LoC panic-mode escalation in the partial-entry branch (fire breaker, engage flatten on the partial). Must land before live trading is enabled, OR the engine init code must explicitly disable live trading until partial-entry handling lands. [`crates/engine/src/order_manager/bracket.rs:325-334`]
- S-4 (carryforward into 8-2 periodic): Reconciliation `avg_entry_price != b.avg_entry_price` is strict integer equality with no rounding tolerance. Local weighted-avg uses truncating integer division (`total_cost / new_qty`); broker may use a different rounding policy. After several additions to a position the two avg-entry values can drift by 1 quarter-tick without any actual error → false PhantomLocal-style mismatch → trading halted. Resolution: configurable tolerance band on `avg_entry_price` comparison (default 1 quarter-tick per fill, capped) OR document that local weighted-avg matches broker's exactly (requires sourcing broker algorithm). Side and quantity must remain strict. [`crates/engine/src/order_manager/tracker.rs:296-306`, `crates/core/src/types/position.rs:75-81`]
- N-1: `last_mid_prices` map in `PositionTracker` is never pruned. Flat-and-pruned positions leave stale mid-price entries. Trivial leak across long-running session with many one-shot symbols. Optional fix: drop the entry when pruning a flat-with-no-history position. [`crates/engine/src/order_manager/tracker.rs:163, 233`]
- N-2: Synthetic `InvalidTransition { from: OrderState::Idle, trigger: OrderTransition::Reject }` is fabricated to fire the breaker callback on reconciliation mismatch — the from/trigger values have nothing to do with the actual mismatch. Operators reading breaker diagnostics see misleading info. Recommendation: widen the breaker callback signature to accept a typed `BreakerCause` enum (`InvalidTransition(...)` | `PositionMismatch(...)`). [`crates/engine/src/order_manager/tracker.rs:384-388`]
