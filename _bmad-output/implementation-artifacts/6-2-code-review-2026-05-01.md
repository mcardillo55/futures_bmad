### SHOULD-FIX

#### S-1 — `last_transition_time` not updated on conservative collapse; cooldown window anchors to wrong epoch

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/orchestrator.rs:165-188`

When `conservative_on_oscillation = true` and an oscillation proposes a more-conservative regime (e.g. Trending at t=10s → Volatile during cooldown at t=20s), the orchestrator collapses to Volatile (updates `self.current_regime` and calls `apply_strategy_permissions`). However, `last_transition_time` remains `Some(ts_secs(10))` — it is only updated in the non-cooldown path at line 192. This creates a behavioral consequence: subsequent cooldown checks continue to compute the elapsed window from t=10s, not from t=20s (the time of the conservative collapse). This means:

1. If the operator-configured cooldown is, say, 300s and the original transition was at t=10s, the cooldown expires at t=310s. But if the conservative collapse at t=20s represents a new "acted-upon" state change (the orchestrator DID change `current_regime` and strategy permissions), callers may expect the next transition at t=320s (300s after the collapse) rather than t=310s. The window is shorter than expected.

2. More subtly: a subsequent oscillation that proposes an even-more-conservative regime (e.g. Volatile→Unknown at t=25s) would also be checked against the t=10s anchor — still within the 300s window — and would collapse to Unknown. `last_transition_time` still stays at t=10s. The cooldown epoch is pinned at the original anchor forever until a genuine non-cooldown transition occurs.

Test 7.11 passes because it only checks that the conservative regime is retained, not that the cooldown anchor was updated correctly. The scenario "conservative collapse + subsequent oscillation within the residual window + then cooldown-after-collapse elapses" is not tested.

Whether `last_transition_time` should be updated on a conservative collapse is a design decision, but the current behavior is undocumented and the `last_transition_time` field doc comment ("Last timestamp at which a transition was acted upon") implies it SHOULD update since the orchestrator does act on a conservative collapse (it updates `current_regime` and `enabled_strategies`).

**Fix:** Either (a) update `last_transition_time = Some(timestamp)` inside the `if safer != self.current_regime` branch alongside the `current_regime` update, or (b) explicitly document in the field comment that conservative collapses do not reset the anchor and explain why the single-anchor-from-first-acted-transition semantics is intentional.

*(Auto-converted from `decision-needed` to `defer` per non-interactive policy — sensible default is option (b): document the semantics explicitly. The current behavior is a reasonable "short cooldown window" interpretation. Deferred.)*

#### S-2 — `pending_transitions` grows without bound across the orchestrator's lifetime

**Source:** Blind Hunter + Edge Case Hunter
**Location:** `crates/engine/src/regime/orchestrator.rs:77, 93, 169`

`pending_transitions: Vec<RegimeTransition>` accumulates every cooldown-suppressed oscillation but is never cleared. The only read access is the test-helper `pending_transitions()` accessor; no production code consumes or drains it. In a live-trading session running for hours with an oscillating classifier near a threshold boundary, `pending_transitions` can grow to hundreds or thousands of entries, leaking memory indefinitely.

The dev note says "Bounded only by cooldown windows; in practice this stays small because the detector runs at bar cadence." That reasoning is partially correct — at 1-minute bar cadence and a 5-minute cooldown, the rate of oscillations is bounded by bar cadence, so the growth is ~1 entry per bar during persistent oscillation. Over an 8-hour session that is up to ~480 entries — negligible memory, but semantically the vector is dead state that serves no production function (transitions are logged and journaled on arrival; there is no dequeue step).

**Fix options:**
1. Remove `pending_transitions` entirely. Its only consumer is a test-assertion in 7.6 (`assert_eq!(orch.pending_transitions().len(), 1)`); that test assertion is redundant given the returned `Some(transition)` already verifies the event was surfaced. If diagnostics of "how many oscillations occurred during this cooldown window" are desired, replace with a counter (`oscillation_count: u32`) that resets on cooldown expiry.
2. Cap the vector with `pending_transitions.truncate(N)` after push (ring-buffer semantics).
3. Clear `pending_transitions` each time the cooldown expires (non-cooldown path, line 190+).

*(Auto-converted from `decision-needed` per non-interactive policy — option 3 is least disruptive and preserves diagnostics during a cooldown window. Deferred; no production crash risk since memory growth is bounded by bar cadence.)*

### NICE-TO-HAVE

#### N-1 — `transition` struct in conservative-collapse path logs/journals the original proposed `from→to`, not the actual collapse target

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/orchestrator.rs:158-171`

The `transition` struct is built with `from: self.current_regime` and `to: new_regime` (lines 158-162) before the conservative-collapse check (lines 173-186). When `conservative_on_oscillation = true` and the collapse changes `current_regime` to `safer` (which may differ from `new_regime`), the `transition` forwarded to the journal and the structured log still reads `from: Trending, to: Volatile` (the proposed pair) — but the actual acted-upon change was `from: Trending, to: Volatile` in this specific case (since `safer == new_regime == Volatile`). They happen to be the same. However, if a three-way case ever occurs (e.g., A→B where B is more conservative than A, but an intermediate state C is even more conservative than B due to a previous unconsumed collapse), the logged `to` might not match what `current_regime` actually became.

This is low-risk today because `is_more_conservative` operates on exactly two inputs (`current` and `new_regime`), so the `safer` value is always either `current` (no-op) or `new_regime` (in which case the log is accurate). Flag for future awareness if the oscillation logic is extended to compare against a stack of pending regime proposals.

#### N-2 — `is_more_conservative` is `pub(crate)` rather than private; cleaner alternative exists

**Source:** Acceptance Auditor (scope decision 4)
**Location:** `crates/engine/src/regime/orchestrator.rs:199-207`

The dev marked this `pub(crate)` to allow the `conservative_ordering_is_correct` test to call it directly. Since the tests live in the same file inside `#[cfg(test)] mod tests { use super::*; }`, a `pub(super)` visibility on `is_more_conservative` would suffice. Alternatively, `conservative_rank` (a free `const fn`) could be made `pub(crate)` instead, and the test could compare ranks directly without calling the wrapper. Neither changes correctness. The current choice is defensible; it's a minor hygiene concern.

#### N-3 — No test for missing `regime_strategy_map` key; `apply_strategy_permissions` silently disables all strategies

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/orchestrator.rs:218-225`

```rust
fn apply_strategy_permissions(&mut self, regime: RegimeState) {
    self.enabled_strategies.clear();
    if let Some(allowed) = self.config.regime_strategy_map.get(&regime) {
        self.enabled_strategies.extend(allowed.iter().cloned());
    }
}
```

If `regime_strategy_map` does not contain a key for the new regime (e.g., an operator TOML that only enumerates `Trending` and `Rotational` but omits `Volatile` and `Unknown`), the `if let Some` branch is skipped and `enabled_strategies` is left empty — effectively disabling all strategies. This is the "safe default" behavior (fail-closed), which is correct for a regime like `Unknown`. However, it is not tested and not documented. An operator who misconfigures the map by omitting a regime key will silently see all strategies disabled with no warning log.

**Fix:** Add a `tracing::warn!` in the `None` branch ("no strategy mapping found for regime {regime}; disabling all strategies") and add a test that constructs an orchestrator with an incomplete map and verifies both the silent-disable behavior and the warn emission.

#### N-4 — `cooldown_zero_disables_suppression` test sends same timestamp for two consecutive transitions

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/regime/orchestrator.rs:523-528`

```rust
fn cooldown_zero_disables_suppression() {
    let mut orch = RegimeOrchestrator::new(test_config(0, false));
    orch.on_regime_update(RegimeState::Trending, ts_secs(1), &clock);
    orch.on_regime_update(RegimeState::Rotational, ts_secs(1), &clock);  // same ts
    assert_eq!(orch.current_regime(), RegimeState::Rotational);
}
```

Both calls use `ts_secs(1)`. The cooldown calculation is `now_nanos <= last_nanos` → in-cooldown (`true`), but `cooldown_nanos == 0` so `is_in_cooldown` returns `false`. This means the test incidentally passes through the `cooldown_nanos == 0` early-return path, not the "timestamps are the same" path. The test is correct and the behavior it exercises is real. The concern is that a future reader might think equal-timestamp is the key invariant being tested. A one-line comment or a distinct second-timestamp would clarify intent.

## Scope Decision Verdicts

### 1. `conservative_on_oscillation` semantics deviation

**Dev claim:** Spec literal reading ("do NOT update current_regime during oscillation") would leave the orchestrator in Trending when Volatile is proposed during cooldown — i.e., the spec text produces a less-conservative outcome than the AC's safety intent ("system remains in the more conservative state"). Dev implemented "collapse to the more conservative of {current, proposed}" so oscillation can only ever degrade to a safer state.

**Verdict: DEFENSIBLE SAFETY IMPROVEMENT — accept the deviation, update spec wording.**

The spec's task description (Task 4.1, bullet 3) says "if `conservative_on_oscillation`, do NOT update current_regime — remain in current (more conservative) state." This wording assumed "current is always the more conservative one," which is only true when initial transitions are always from less-conservative to more-conservative (e.g., Unknown→Trending as the first transition). The AC text ("system remains in the more conservative state") unambiguously states the desired end state, not the mechanism. The dev's resolution is the correct one: collapse to `max(conservative_rank(current), conservative_rank(proposed))`. Test 7.11 exercises the Trending→Volatile case (safer) and the subsequent Volatile→Rotational case (no degrade), which together verify both sides of the invariant. The spec task wording should be updated to match the implementation, but the code should not be reverted.

### 2. `RegimeTransition` moved from `events/lifecycle.rs` to `events/risk.rs`

**Verdict: CLEAN — no orphan references in the main codebase.**

The move is correct. `events/mod.rs` re-exports `RegimeTransition` from `risk` instead of `lifecycle`, and `core/src/lib.rs` re-exports `RegimeTransition` from `events`. The public API path (`futures_bmad_core::RegimeTransition`) is unchanged. Grepping all non-worktree `.rs` files: `lifecycle.rs` no longer contains or imports `RegimeTransition`; the only references are in the worktrees which are at pre-commit snapshots and not part of the main codebase. `EngineEvent::Regime(RegimeTransition)` in `events/engine.rs` continues to compile because the re-export path is unchanged. The journal's `EngineEvent::RegimeTransition(RegimeTransitionRecord)` is a distinct type (the journal-side variant); no cross-contamination. Consumers of `futures_bmad_core::RegimeTransition` do not need to change their imports.

One minor note: the doc comment on `RegimeTransition` (`risk.rs:31`) still says "Carried both as an in-memory event on the engine bus (`crate::events::EngineEvent::Regime`)" — this correctly references the core-side `EngineEvent::Regime(RegimeTransition)` variant which exists. Accurate.

### 3. `RegimeState` gained `Hash` + `serde::Deserialize` + `as_str()`

**Verdict: CORRECT — all three additions are internally consistent.**

- **`Hash` consistency with `Eq`:** `RegimeState` derives both `PartialEq` and `Eq` (previously) and now adds `Hash`. Rust's derive macro generates `Hash` implementations that are consistent with `PartialEq` for `enum`s with no fields (each variant hashes to its discriminant, and equal variants have equal discriminants). No custom `PartialEq` exists. The `Hash` derive is correct and consistent.

- **`serde::Deserialize` round-trip with TOML keys:** The TOML inline-table `[regime.regime_strategy_map]` uses `Trending`, `Rotational`, `Volatile`, `Unknown` as string keys. With `#[derive(Deserialize)]` on a unit enum, serde uses the variant name strings exactly. The `deserialises_from_toml` test in `config/regime.rs` covers this round-trip end-to-end and passes. Variant names are PascalCase; TOML keys are case-sensitive. The config doc comment explicitly warns operators about case-sensitivity. Correct.

- **`as_str()` consistency:** The `as_str()` match arms return exactly the same strings as the variant names used by serde (`"Trending"`, `"Rotational"`, `"Volatile"`, `"Unknown"`). The `regime_transition_to_record_round_trip` test in the orchestrator suite verifies that the `From<RegimeTransition>` conversion correctly maps via `as_str()` to the expected journal strings. Consistent.

### 4. `is_more_conservative` is `pub(crate)` rather than private

**Verdict: ACCEPTABLE — see N-2 for cleaner alternatives.**

The `pub(crate)` visibility is not wrong; it is somewhat wider than necessary. Since the tests live in `#[cfg(test)] mod tests { use super::*; }` within the same file, `pub(super)` would be sufficient. Alternatively, moving the `conservative_rank` free function to `pub(crate)` and testing ranks directly avoids needing the wrapper to be public at all. The current choice is pragmatic and poses no correctness risk (the function is `const`-friendly and has no side effects). Tracked as N-2.
