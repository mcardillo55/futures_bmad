## Top-Line Verdict: **APPROVE-WITH-CHANGES**

**Findings:** 0 BLOCKING / 4 SHOULD-FIX / 5 NICE-TO-HAVE

The implementation is structurally sound, well-documented, and the trait-injection wiring is exactly the right shape for the AC's "no replay-specific branching" intent. All 28 tests (22 unit + 6 integration) pass, clippy is clean, and the deliberate fidelity-boundary documentation is excellent. However, three findings warrant attention before this story is considered production-ready: (1) AC1 mis-claims that `EventLoop` is wired through replay — the orchestrator's `run()` runs its own private loop (`apply_market_event` only) and entirely bypasses the live `EventLoop`'s data-quality gate, circuit breakers, event windows, and stale-data detection; (2) AC4 says "all events written to journal" but the production binary in `main.rs` never calls `attach_journal()` — only `replay_start`/`replay_complete` system events would be journaled even if a journal were attached, because no `TradeEvent` records flow through; (3) `ReplayDriver` is `pub`-exported in `mod.rs` but is dead code relative to the orchestrator's own loop — it duplicates the producer-push logic without the monotonicity guard or fill simulation. None of these block marking the story `done` per the dev's scope decisions, but each should be deferred and tracked because they are baseline assumptions Story 7.2 (deterministic replay verification) and Story 7.3 (paper trading mode) will need to act on.

### SHOULD-FIX

#### S-1 — Orchestrator's `run()` bypasses the production `EventLoop` entirely

**Source:** Acceptance Auditor + Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:362-467`

The module docstring (`mod.rs:11-12, 19-22` and `orchestrator.rs:11-13`) repeatedly claims that replay flows through "the same engine `EventLoop` (no replay-specific branching)" via trait injection. In reality, `run()` runs a hand-rolled loop:

```
loop {
    event = source.next_event()
    sim_clock.set_time(event.ts)
    market_producer.try_push(event)
    drain_market_consumer()         // calls apply_market_event ONLY
    drain_orders_and_simulate_fills()
    drain_fills()                   // → trade_stats only
}
```

`drain_market_consumer` calls `apply_market_event(&mut self.book, &ev)` which updates the order book but skips ALL of `EventLoop`'s gates: `BufferMonitor` (pressure tracking), `StaleDataDetector` (stale-quote watchdog — though documented as gated by `is_market_open`), `SequenceGapDetector` (gap detection), `DataQualityGate`, `EventWindowManager`, `CircuitBreakers` (anomaly checks, position-tracker plumbing), `PositionTracker`, and journal-side `TradeEvent` emission.

Why this matters for Story 7.1's stated purpose:
1. **AC1 verification.** "Event loop code identical" is literally true (`event_loop.rs` is untouched), but the orchestrator does not actually use it. The trait-injection design in the architecture doc anticipates `EventLoop<SimClock>::new(consumer, ...)` being constructed by the orchestrator and given the consumer. That construction is missing.
2. **Story 7.2 (deterministic replay verification) cannot exercise the real engine path.** Determinism is a property of the WHOLE event-loop pipeline (signals + risk + breakers + regime). If 7.2 wires up the real `EventLoop` later, the orchestrator's current `run()` will need to be partially rewritten — the FIFO trade aggregator competes with `PositionTracker` for ownership of fill events.
3. **Risk-side gates are silently disabled in replay.** A backtest that wants to validate "would the circuit breaker have fired on this bad data?" cannot answer that question with the current orchestrator because the breaker is never wired in.
4. **The cfg(test) accessors `market_consumer_mut`, `order_producer_mut`, `fill_consumer_mut`** (orchestrator.rs:329-342) imply the orchestrator was originally designed to hand these out to a downstream `EventLoop`; the comment on `market_consumer_mut` says "Hand the engine-side market event consumer to a downstream owner (e.g. an [`crate::EventLoop`]) BEFORE calling [`Self::run`]". `Self::run` then drains the consumer itself, so the cfg(test) hand-off path conflicts with the production drain path — only the test path can use it.

**Fix:** Either (a) restructure `run()` to construct an `EventLoop<SimClock>` and call its drive method (preferred, matches the architecture doc), or (b) update the module docstrings to explicitly describe the orchestrator as a "minimal-pipeline replay harness for V1, with full-pipeline replay tracked under Story 7.2/7.3" — i.e. change the ASPIRATIONAL claim into an HONEST scope statement so future readers don't act on a false premise.

*(Auto-converted to defer — sensible default per non-interactive policy is option (b): document the V1 scope honestly, then expand the wiring under Story 7.2 / 7.3 when the integration is needed. Deferred; no production crash risk because no operator can yet run live trading.)*

#### S-2 — Production binary in `main.rs` never attaches a journal; only `replay_start`/`replay_complete` system events are written even with one attached

**Source:** Acceptance Auditor + Blind Hunter
**Location:** `crates/engine/src/main.rs:45-105` and `crates/engine/src/replay/orchestrator.rs:283-285, 490-500`

AC4 requires "all events written to journal." The current state:
1. `main.rs` constructs a `ReplayOrchestrator` and calls `orch.run()`. It never calls `orch.attach_journal()`. There is no code path in production that journals replay output.
2. Even when `attach_journal()` IS called (by `end_to_end_replay_journals_start_and_complete`), only `JournalEvent::SystemEvent { category: "replay_start" | "replay_complete" }` records are emitted. No `TradeEvent`, no `OrderStateChange`, no `RegimeTransition`, no `CircuitBreakerEvent`. The integration test passes because it only asserts on the system-event categories.
3. `journal_system_event` (orchestrator.rs:490-500) uses `self.sim_clock.now()` for the `replay_start` event — but the clock is at 0 at that point (the first event timestamp hasn't been processed yet). So the journaled `replay_start` record carries a `timestamp = 0` which is misleading from a forensic standpoint (the journal worker writes it at the actual wall-clock time of the call, but the embedded `timestamp` field is 0).

**Fix options:**
1. **Minimum:** Update `main.rs` to construct an `EventJournal` worker and call `orch.attach_journal(sender)` before `orch.run()`. The integration test pattern (`tests/replay_orchestrator.rs:115-117`) shows the API is in place.
2. **Correct AC4:** Convert each `FillEvent` consumed by `drain_fills` into a `TradeEventRecord` (if a journal is attached) and push it via `journal.send(JournalEvent::TradeEvent(...))`. This requires looking up the symbol_id → symbol-string mapping or just deferring on `symbol` for now.
3. **Document the gap:** If AC4 is intentionally scoped to "the journal IS WIRED, downstream events are added in Story 7.3 paper-trading mode", say so in both the spec and the orchestrator docstring.
4. **Use the actual event timestamp for `replay_start`:** push `replay_start` AFTER the first event is consumed, so `sim_clock.now()` is non-zero. Or compute the source's first timestamp before pushing.

*(Auto-converted to defer per non-interactive policy — sensible default is option (1) + option (3): wire the journal in `main.rs` and document the V1 scope. Trade-event journaling is genuinely a Story 7.3 (paper trading mode) concern. Deferred.)*

#### S-3 — `ReplayDriver` is `pub`-exported but is dead code relative to the orchestrator's own loop

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/driver.rs` (entire file) and `crates/engine/src/replay/mod.rs:24, 29`

`ReplayDriver` is a separate type that drains a `DataSource` and pushes to a `MarketEventProducer`. It is `pub`-exported via `mod.rs:24, 29` but is NEVER called by the orchestrator. The orchestrator owns its own `market_producer` and pushes directly (orchestrator.rs:394-405). `ReplayDriver` has its own monotonicity-free push logic (no `set_time` call, no skip-on-non-monotonic), and a different drop policy: it warns every 1000 dropped events and silently drops the rest, whereas the orchestrator drains-and-retries-once then warns + skips.

Two issues:
1. **Two divergent mechanisms.** A future contributor wiring a new replay scenario can reach for `ReplayDriver` (it is `pub`!) and silently get the WRONG behavior — no clock advancement, no monotonicity guard, no fill simulation. This is a footgun.
2. **Tests in `driver.rs:117-163` exercise the dead code**, lending it false credibility. The driver is genuinely tested in isolation but is never integrated into the production replay path.

**Fix options:**
1. **Delete `ReplayDriver` entirely** if the orchestrator's inline loop is the canonical mechanism. Remove `pub mod driver;` from `mod.rs:24` and the `pub use driver::ReplayDriver;` re-export at line 29. This is the cleanest option.
2. **Refactor the orchestrator's push logic to USE `ReplayDriver`,** moving the monotonicity check and the drain-and-retry policy into the driver. Single source of truth.
3. **Mark `ReplayDriver` with `#[doc(hidden)]` and a deprecation warning** explaining "use `ReplayOrchestrator::run` instead." Preserves the test history but signals it's dead.

*(Auto-converted to defer per non-interactive policy — sensible default is option (1): delete it. The orchestrator's loop is the production path; the driver is leftover scaffolding from an earlier design. Deferred to a code-cleanup pass.)*

### NICE-TO-HAVE

#### N-1 — `TradeStats::apply_fill` win-rate denominator excludes break-even trades but does NOT count them as `total_trades`-only-wins

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:165-170`

```rust
self.total_trades += 1;
if leg_pnl_qt > 0 {
    self.wins += 1;
} else if leg_pnl_qt < 0 {
    self.losses += 1;
}
```

A scratch trade (`leg_pnl_qt == 0` — opens at price X and closes at price X) is counted in `total_trades` but is NOT counted in `wins` or `losses`. Then `win_rate_pct = wins / (wins + losses)` excludes scratches from BOTH numerator and denominator. So with 1 win + 1 loss + 8 scratches, the report shows `total_trades = 10, win_rate = 50%`. Operators reading the summary will reasonably expect `win_rate * total_trades = wins + ties` or similar invariant; the current breakdown does not satisfy any obvious invariant.

This is a design choice, not a bug — backtesting tools are split on whether scratches count as wins, losses, or are excluded. The point is it should be DOCUMENTED in the `ReplaySummary::win_rate_pct` doc comment ("`win_rate_pct` excludes scratch trades from both numerator and denominator; `total_trades - wins - losses` is the count of break-even trades").

**Fix:** Add a one-line doc comment to `ReplaySummary` describing the denominator semantics.

#### N-4 — `SimClock::set_time` is called BEFORE `try_push`; if the push fails (capacity full), the clock is in an inconsistent state with the SPSC

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:374-405`

The flow is:
1. `set_time(event_ts)` — clock now reads `event_ts`
2. `try_push(event)` — fails if SPSC full
3. Drain (releases capacity)
4. `try_push(event)` again — fails again ⇒ `continue` (skip event)

If step 4 fails the event is dropped but the clock has already been advanced to its timestamp. Subsequent events with timestamps `< event_ts` would be classified as non-monotonic (correctly) but the dropped event's intervening "this happened" record is lost. The summary reports `events_consumed` (which excludes dropped) but the clock pretends the event was processed.

In the current single-threaded loop the second push failure is genuinely "should never happen" (we drained immediately before retrying — there is no contention because the consumer is owned by the same thread). But the warn message at line 401 acknowledges it as a potential outcome. A defensive fix: revert the clock to `last_event_ts` if the second push fails, OR move the `set_time` call to AFTER a successful push.

Low risk in practice (single-thread topology means second push cannot fail in V1) but a real consistency bug if the orchestrator is ever extended to a multi-threaded model where the consumer runs on a separate thread.

**Fix:** Move the `set_time` / `last_event_ts = Some(event_ts)` updates to a block that runs only after `try_push` succeeds. Or document that the clock-advance-then-push order is safe under the single-thread invariant.

#### N-5 — `validate_schema` does NOT verify nullability or extra columns; a Parquet file with a NULL `timestamp` column would pass schema validation and panic at iteration

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/data_source.rs:214-235`

`validate_schema` checks (a) the column exists, and (b) the column's `data_type()` matches. It does NOT check `field.is_nullable()`, so a Parquet file written with `Field::new("timestamp", DataType::Int64, true /* nullable */)` and actual NULLs in the column would pass schema validation. The downstream `ParquetDataSource::next_event` (in `crates/engine/src/data/parquet_source.rs`, not shown but called from data_source.rs:159) is what actually reads rows — depending on its implementation, NULL values in `timestamp` may panic, produce a default `UnixNanos::new(0)`, or skip the row.

The schema constants in `EXPECTED_COLUMNS` (line 36-43) all carry `DataType::Int64` etc. but the `Field::new(name, ty, nullable)` triple — `nullable` is unverified. The "open the file fast and fail clean" promise is correct for the column-presence and type checks but not for nullability.

**Fix:** Add `if field.is_nullable() { return Err(...) }` for the strict-non-null columns, OR explicitly document the nullability policy ("nullability is permissive on open; the lower-level reader silently skips NULL rows" or similar). At minimum, add a test that constructs a Parquet file with nullable timestamp column and asserts the behavior.

## Scope Decision Verdicts

### 1. `MockFillSimulator` lives outside `MockBrokerAdapter`; orders flow through SPSC, not through `broker.submit_order`

**Dev claim (Completion Notes):** "Fill model isolated from broker adapter. `MockBrokerAdapter` (testkit) records submitted orders but does not emit fills directly — `MockFillSimulator` (new, `replay/fill_sim.rs`) reads the engine→broker SPSC queue and pushes synthesized fills onto the broker→engine SPSC queue. This keeps the testkit mock minimal and keeps the fill model swappable for future fidelity tiers (slippage, queue position, partial fills)."

**Verdict: ACCEPTABLE — but document the consequence clearly.**

The decoupling is sound: the `MockBrokerAdapter` lives in `testkit` and stays minimal (test fixtures should not own simulation logic), and `FillModel` becoming an enum in `replay/fill_sim.rs` lets future fidelity variants slot in without touching the testkit. The trade-off is that the `MockBrokerAdapter` instance held by the orchestrator is never actually hit by the order flow — only its trait surface (`subscribe`/`cancel`/`query_positions`) is exposed. See N-2 for the documentation request. The decision itself is the right one.

### 2. `engine` binary added; running without `--replay` returns `ExitCode::from(2)`

**Verdict: CORRECT — but the no-mode-selected error is silent without a tracing subscriber (see N-3).**

The exit-code-2 convention is appropriate for "user error in invocation" (as opposed to exit-code-1 for "internal failure"). The clap definition is clean and the `--config` arg is parsed but ignored with an honest doc comment. The only UX gap is that `tracing_subscriber_init` is a no-op, so the `error!` log line is dropped. See N-3.

### 3. `futures_bmad_testkit` promoted from `[dev-dependencies]` to `[dependencies]`

**Verdict: CORRECT — necessary for the SimClock + MockBrokerAdapter wiring.**

The promotion is required because the orchestrator uses these types at runtime. The risk to flag is that `testkit` is named `_bmad_testkit` and the convention is "test fixtures only" — so a downstream consumer of `futures_bmad_engine` who pulls in the engine crate now also pulls in `testkit`. This is acceptable for V1 (replay IS a test/dev mode and the testkit dep makes that lineage explicit), but Story 7.3 (paper trading mode) and especially Epic 8 (live mode) should consider whether `SimClock` and `MockBrokerAdapter` should be promoted into a non-test crate (e.g. moved into `core` or a new `futures_bmad_replay` crate) so production builds don't carry a dep named `_testkit`. ## Other Observations

- **`#![deny(unsafe_code)]` is NOT set** on `replay/mod.rs`, `orchestrator.rs`, `data_source.rs`, `fill_sim.rs`, or `main.rs`. The story-6.2 convention used `#![deny(unsafe_code)]` on the regime module files. For consistency and defense-in-depth, add it to all four new replay files. (This would be N-6 if classified.)
- **`SimClock::new(0)` initial value:** The orchestrator constructs `SimClock::new(0)` (orchestrator.rs:240). On the first event, `set_time(event_ts)` jumps the clock from 0 to event_ts — which can be a jump of ~1.7 trillion (i.e. 56 years for a 2026-04-18 timestamp). Any time-dependent code reading the clock between `new` and the first event sees `now() == 0`, which is 1970-01-01. The `journal_system_event("replay_start", ...)` call at `orchestrator.rs:366-368` reads `sim_clock.now()` for the timestamp ⇒ the journaled record carries `timestamp = 0`. See S-2 point 3.
- **`OrderEvent::order_type` is read but unused under V1's `ImmediateAtMarket`:** The `let _ = order.order_type;` at `fill_sim.rs:154` correctly suppresses the unused-binding lint. Future fidelity tiers (Limit / Stop) can branch here without touching the call sites.
- **`saturating_add` everywhere:** `cumulative_pnl_qt = self.cumulative_pnl_qt.saturating_add(leg_pnl_qt)` (orchestrator.rs:164), `next_fill_id = self.next_fill_id.saturating_add(1)` (fill_sim.rs:75). Correct overflow handling — replay over very long periods or extreme prices won't panic.
- **`win_rate_pct() == 0.0` when `decided == 0`:** Sensible default for "no completed trades" rather than NaN or panic. Documented in code via the explicit branch.
- **`drain_market_consumer` calls `apply_market_event` per event** — for Trade events this is a no-op (the order book doesn't change), so trade events ARE counted in `events_consumed` but do not affect `book.best_bid`/`book.best_ask`. Good — the count is over events, not over book changes.
- **`market_event_capacity: Option<usize>`** — defaults to `MARKET_EVENT_QUEUE_CAPACITY` (131,072). Test `fill_model_immediate_at_market_explicit` uses 64 to exercise the override path. Correct API design.
- **No `Send + Sync` annotation tests on the orchestrator.** The orchestrator holds `MockBrokerAdapter` (synchronously mutable), `MockFillSimulator`, and SPSC consumer/producer types. If a future story needs to drive replay from a separate task (tokio spawn), this could become a problem. Worth a `_assert_send::<ReplayOrchestrator>()` compile-time check.
