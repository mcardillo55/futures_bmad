## Top-Line Verdict: **APPROVE-WITH-CHANGES**

**Findings:** 0 BLOCKING / 4 SHOULD-FIX / 7 NICE-TO-HAVE

The framework is structurally sound. `permits_trading()` is `#[inline]`, the happy path collapses to a fixed boolean reduction with zero allocation, the gate-vs-breaker distinction is enforced at the type level via `clear_gate -> Result<_, GateClearError>`, the stop-loss preservation invariant is encoded directly in `orders_to_cancel`, and journal emission is `try_send` (non-blocking on `Full`/`Disconnected`). The 12-test surface covers idempotency, the multi-denial aggregation, the gate-vs-breaker boundary, and a smoke-test hot-path budget.

The four SHOULD-FIX items all concern the **gap between this story's API surface and the per-breaker logic landing in 5.2-5.6**: there is no public path to update the cached denial-context fields (`daily_loss_current`, `consecutive_loss_count`, `requested_position_size`, `fee_days_stale`, `buffer_occupancy_pct`) — only `set_buffer_occupancy` is exposed. So `DenialReason::DailyLossExceeded { current: 0, limit: X }` is the only thing operators will ever see if they trip a breaker via `trip_breaker(BreakerType::DailyLoss, "loss too big", ts)`. The reason text in `trigger_reason` carries the correct human-readable info, but the structured `DenialReason::*` will publish stale zeros until story 5.2 retrofits setters. None of this prevents 5.1 itself from being correct against AC1-4, but the next story will be unable to wire up proper denial reasons without a follow-up to this surface — so this branch is a partial deliverable by design.

**Story can be marked `done`** with the SHOULD-FIX items captured as carryovers into stories 5.2-5.6.

## Acceptance Criteria Audit

The spec lists 4 ACs (BDD form):

- **AC1** — single source of truth for "can we trade?", contains state for all gate/breaker types, distinguishes gates (auto-clear) from breakers (manual reset): **MET.**
  - All 10 breaker/gate states live on `CircuitBreakers` (`circuit_breakers.rs:55-103`).
  - Gate vs breaker distinction is encoded in `BreakerType::category()` (`events/risk.rs:62-75`) and enforced at runtime by `clear_gate -> Result<_, GateClearError::NotAGate>` (`circuit_breakers.rs:312-319`). Test `clear_gate_rejects_breaker_types` is the machine-checked invariant.

- **AC2** — `permits_trading()` synchronous, returns `Result` with specific denial reasons, completes within one tick (NFR10): **MET.**
  - Synchronous, no `async`/no awaits anywhere on the path. `#[inline]` on `permits_trading`, `#[inline]` on `all_active`. Happy path is a 10-AND boolean reduction over `BreakerState::Active` (`circuit_breakers.rs:155-178`), zero allocation. Failure-side `collect_denials` builds a `Vec<DenialReason>` capped at 10 — only fires when at least one gate/breaker is non-`Active`.
  - NFR10 ("checking must complete within one tick"): the smoke-test `permits_trading_is_fast_on_happy_path` (`circuit_breakers.rs:723-736`) measures 100k iterations in <100ms (i.e. <1µs/call). The dev correctly notes this is a smoke test, not the actual SLA — the real budget is enforced by the event-loop timing harness in story 8-2, which does not exist yet. Test is fine as a regression tripwire but does not prove the NFR.

- **AC3** — breaker tripped -> entry/limit cancelled, stop-loss preserved, CircuitBreakerEvent emitted to journal: **PARTIAL.**
  - `orders_to_cancel(&self, &[OrderEvent]) -> Vec<u64>` (`circuit_breakers.rs:415-424`) filters out `OrderType::Stop { .. }` and returns IDs of `Market` and `Limit { .. }` orders. Test `orders_to_cancel_preserves_stops` (3 input orders, 2 returned, stops dropped). MET in isolation.
  - `trip_breaker` emits a `CircuitBreakerEvent` via `try_send` on the crossbeam channel (`circuit_breakers.rs:295-303` -> `emit_event` at 460-478). Test `trip_breaker_emits_event_to_journal` asserts the receive side. MET.
  - **Critical gap:** `trip_breaker` does **NOT** itself call `orders_to_cancel`. It simply transitions the state and emits the event. The actual cancellation is left to the caller — which doesn't exist yet (the event-loop wiring is story 8-2). For the framework alone this is a defensible separation of concerns (cancellation needs a broker handle), but the AC text says "When activates Then entry/limit orders cancelled" — that side-effect is not delivered in 5.1. **PARTIAL until 8-2 wires `trip_breaker` -> `orders_to_cancel` -> broker submitter.**

- **AC4** — gate condition resolves -> auto-clears, trading resumes, clearance logged: **MET.**
  - `update_gate_conditions` re-evaluates the 3-gate table each tick (`circuit_breakers.rs:433-455`). When `still_tripped == false` and the gate is currently `Tripped`, it calls `clear_gate`, which flips state -> `Active`, emits `info!` log with `tripped_for_ms`, and pushes a `CircuitBreakerEvent` with `new_state: Cleared` to the journal. Test `gate_auto_clear_transitions_tripped_to_active`.

**AC summary: 3 met / 1 partial / 0 not-met.** AC3's partial is on the side-effect (cancellation), not the framework itself; the framework provides the helper `orders_to_cancel` and the event emission. The wiring will land in story 8-2.

## Adversarial Findings

### SHOULD-FIX

#### S-1 — `trip_breaker` cannot populate denial-context numeric fields

**Source:** Edge Case Hunter + Acceptance Auditor
**Location:** `engine/src/risk/circuit_breakers.rs:79-88, 247-304`

`CircuitBreakers` carries cached numeric context: `daily_loss_current`, `daily_loss_limit`, `consecutive_loss_count`, `consecutive_loss_limit`, `trade_count`, `max_trades_limit`, `requested_position_size`, `max_position_size_limit`, `fee_days_stale`, `buffer_occupancy_pct`. The `_limit` fields are seeded from `TradingConfig` at construction time. The `current` / `count` / `requested` / `pct` fields are seeded to `0` and **never updated by anything in this branch**.

`trip_breaker(breaker_type, reason: String, timestamp)` accepts only a free-form reason string. It updates the `*_detail` String fields for `AnomalousPosition`, `ConnectionFailure`, `DataQuality`, `MalformedMessages` (lines 266-280), but for the numeric breakers (`DailyLoss`, `ConsecutiveLosses`, `MaxTrades`, `MaxPositionSize`, `FeeStaleness`, `BufferOverflow`) there is no path to surface the actual current value. Consequence:

```rust
// caller in story 5.2 (daily-loss breaker):
breakers.trip_breaker(BreakerType::DailyLoss, "session pnl -1200 ticks <= -1000 cap", ts);
// later:
let denied = breakers.permits_trading().unwrap_err();
// denied.reasons[0] = DailyLossExceeded { current: 0, limit: 1000 }
//   ...even though the actual current is -1200.
```

The `trigger_reason` carries the human-readable truth for the journal record. But `DenialReason::Display` and structured-log consumers will see `"daily loss 0 ticks reached limit 1000"` — a lie. `set_buffer_occupancy` exists (`circuit_breakers.rs:483-485`) — the only public setter. The other 9 fields have no analog.

**Fix options** (pick one):
1. Add per-breaker setters: `set_daily_loss(i64)`, `set_consecutive_loss_count(u32)`, `set_trade_count(u32)`, `set_requested_position_size(u32)`, `set_fee_days_stale(i64)`. Mark `#[doc(hidden) pub` for now (matches `set_buffer_occupancy` precedent) until a more disciplined surface is decided.
2. Reshape `trip_breaker` to take a typed `TripContext` enum carrying the relevant numeric for each breaker variant, so that `trip_breaker(BreakerType::DailyLoss, TripContext::DailyLoss { current: -1200 }, ...)` is the only way to call it. This is the most type-safe option but reshapes the API across stories 5.2-5.6.

The second option is the right end-state. The first is the migration path. **Either way this needs to land before story 5.2 (daily-loss breaker) implementation, otherwise 5.2 will ship a broken `DenialReason`.**

#### S-2 — `update_gate_conditions(&GateConditions::default(), ts)` silently auto-clears every tripped gate

**Source:** Edge Case Hunter
**Location:** `engine/src/risk/circuit_breakers.rs:433-455`, `GateConditions: Default` at line 494

`GateConditions` derives `Default` and every `*_tripped` field defaults to `false`. The loop at line 444 calls `clear_gate` whenever `!still_tripped && state == Tripped`. So:

```rust
// Story 5.5 (max-position-size gate) trips the gate:
breakers.trip_breaker(BreakerType::MaxPositionSize, "would-be 5 > cap 2", ts1);
// Some time later, an unrelated story 5.4 wants to update only DataQuality:
breakers.update_gate_conditions(&GateConditions::default(), ts2);
// All three gates are now reset to Active. MaxPositionSize "auto-cleared"
// even though the position cap is still in violation.
```

The test `breaker_manual_reset_required` (line 647-656) accidentally relies on this behavior: it passes `GateConditions::default()` to verify a *breaker* doesn't auto-clear, but it would *also* auto-clear any gate that was tripped — which the test happens not to exercise. So this is a footgun, not a tested-and-broken contract.

**Fix:** require callers to evaluate every gate every tick (the architecture spec actually says exactly this: "re-evaluates the underlying condition each tick"). Two options:
1. Remove `Default` from `GateConditions` so callers must construct every field explicitly. This is the strict fix and cheap to do.
2. Document the contract in the doc-comment: callers MUST pass the full snapshot every tick; partial updates are not supported. This is the soft fix.

I'd take option 1. The struct is small enough that defaulting it adds little ergonomics and the failure mode is silent gate erasure.

#### S-3 — `reset_breaker` does not journal the reset and does not clear cached numeric context

**Source:** Blind Hunter
**Location:** `engine/src/risk/circuit_breakers.rs:364-374`

`reset_breaker` flips the state field back to `Active` and clears the gate-trip `Instant` for gates, but:
1. It emits **no** `CircuitBreakerEvent`. Manual-reset events are silent in the journal.
2. It does **not** zero `daily_loss_current`, `consecutive_loss_count`, etc. So a hypothetical "operator resets at runtime" path (not V1, but planned) leaves stale denial-context values in place.

The architecture spec says "manual reset (= process restart in V1)". On restart everything is reconstructed from journal, so (1) is moot for V1 — but if the framework is to be the single source of truth for "every breaker transition", then a reset is a transition and merits an event with `previous_state: Tripped, new_state: Active`. `clear_gate` already emits with `new_state: Cleared`; the equivalent for breakers would be a `new_state: Active` (or a new `BreakerState::Reset` variant — but `Active` is fine since the audit trail captures the transition).

**Fix:** make `reset_breaker` accept a `timestamp: UnixNanos` and a reason string, emit a `CircuitBreakerEvent`, and zero the numeric context fields associated with the breaker.

#### S-4 — `permits_trading` does not enforce that a *fresh* `CircuitBreakers` is in the expected initial state if `TradingConfig` has weird limits

**Source:** Edge Case Hunter + Blind Hunter
**Location:** `engine/src/risk/circuit_breakers.rs:110-146`, `core/src/config/validation.rs:55-118`

`CircuitBreakers::new` accepts a `&TradingConfig` and copies the limit fields verbatim. `validate_trading_config` rejects `max_position_size == 0`, `max_consecutive_losses == 0`, `max_trades_per_day == 0`, `max_daily_loss_ticks < 0`. But `validation.rs` is **not called** by `CircuitBreakers::new`. If a caller constructs a `TradingConfig` directly (skipping TOML loading + validation — which test fixtures do, including the one in `circuit_breakers.rs:517-530`), they can produce a `CircuitBreakers` with `max_trades_limit: 0`. The `MaxTrades` breaker would then need to trip whenever `trade_count > 0` — but this branch ships nothing that updates `trade_count`, so the question is moot **today**. It will become a real problem once story 5.2+ wires the trade-count update.

The code's defensible position is "validation is the caller's job". That's fine, but `CircuitBreakers::new(&config, ...)` should at minimum `debug_assert!` invariants matching `validate_trading_config`. Otherwise integration tests will silently construct nonsensical state.

**Fix:** add a `debug_assert!` at the top of `new` checking the config is non-degenerate (`max_position_size > 0`, `max_consecutive_losses > 0`, `max_trades_per_day > 0`, `max_daily_loss_ticks >= 0`), and document that callers must pre-validate.

### NICE-TO-HAVE

#### N-1 — Doc-comment is internally inconsistent and contradicts the spec

**Source:** Blind Hunter
**Location:** `crates/core/src/events/risk.rs:5-9` and line 129

Module-level doc says "10 total — 4 gates that auto-clear and 6 breakers that require manual reset". The test at line 130 says "(3 gates / 7 breakers)". The architecture spec (`architecture.md:472-482`) says 7 breakers / 3 gates. The actual `category()` mapping confirms 3 gates / 7 breakers. The module doc is just wrong. Trivial fix.

#### N-2 — `pub malformed_window: VecDeque<Instant>` is a public mutable field on the framework struct

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:93`

The field is `pub` so future stories (5.4, malformed-message breaker) can populate it. But `pub` on a struct field bypasses the framework's invariants entirely — anyone with `&mut CircuitBreakers` can clear, drain, or stuff the window arbitrarily. Compare with `set_buffer_occupancy(&mut self, pct: f32)` which is `#[doc(hidden) pub fn`. Inconsistent encapsulation.

**Fix:** wrap `malformed_window` behind a `#[doc(hidden)]` setter / `record_malformed(&mut self, instant: Instant)` API. (Or accept the inconsistency and document it loudly.)

#### N-3 — `clear_gate` on an already-`Active` gate returns a synthetic event with `trigger_reason: "already-active"`

**Source:** Blind Hunter
**Location:** `circuit_breakers.rs:319-330`

The idempotent no-op path returns `Ok(CircuitBreakerEvent { ... trigger_reason: "already-active", previous_state: Active, new_state: Active })`. No journal record is sent (good), but the returned event is a synthetic placeholder that callers might log or persist. The truthful representation would be `Result<Option<CircuitBreakerEvent>, GateClearError>` so `Ok(None)` indicates the no-op cleanly. Symmetric concern in `trip_breaker` line 254-263: returns a synthetic event with `previous_state == new_state == Tripped` and the caller's `reason` repeated — also confusing.

**Fix:** change return types to `Result<Option<CircuitBreakerEvent>, GateClearError>` and `Option<CircuitBreakerEvent>` respectively, with `None` meaning "no-op, no transition occurred". Updates one line in each caller.

#### N-4 — `reset_breaker` accepts gate types and silently bypasses the journal

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:364-374`

The doc says `clear_gate()` is the "preferred path" for gates because it logs duration and emits the event. But `reset_breaker(BreakerType::DataQuality)` is also accepted and just flips state silently. There is nothing preventing a misguided caller from using `reset_breaker` on a gate. Symmetric to the `clear_gate` rejection of breaker types — `reset_breaker` should reject gate types (or at least emit the same audit-trail event as `clear_gate`).

**Fix:** make `reset_breaker` mirror `clear_gate`: return `Result<(), ResetBreakerError::NotABreaker(BreakerType)>` or, more pragmatically, dispatch internally to `clear_gate` when the type is a gate.

#### N-5 — `permits_trading_is_fast_on_happy_path` may flake on shared CI

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:722-736`

100k iterations in <100ms is a wall-clock test. On a contended CI runner (especially with sanitizers / coverage instrumentation enabled) the budget can be blown by an unrelated GC / scheduler hiccup. Dev acknowledged this in the comment. Either (a) widen the budget to 500ms (still 5µs/call which is generous), or (b) gate behind `#[cfg(not(debug_assertions))]` so it only runs in release builds, or (c) accept the flake risk. I'd take (a).

#### N-6 — `emit_event` always clones the entire `CircuitBreakerEvent` (including the `trigger_reason: String`) even on `Disconnected`

**Source:** Blind Hunter
**Location:** `circuit_breakers.rs:460-478`

`try_send` consumes its argument, so a clone is unavoidable on the success path. But on `Disconnected` (channel receiver dropped — i.e. journal worker has gone away — terminal-state from the framework's POV) the clone is wasted because the event will never be received. Not on the hot path so not a perf concern, but it does mean a tripped breaker on a disconnected journal channel allocates one `String` copy per trip event. Minor.

**Fix:** none required. Note for future optimization passes.

#### N-7 — `update_gate_conditions` table is a hardcoded length-3 array; adding an 11th breaker/gate type requires touching this method

**Source:** Edge Case Hunter
**Location:** `circuit_breakers.rs:436-454`

Today there are 3 gates. The table at line 437-443 hardcodes them. If story 5.X adds a fourth gate (e.g. "self-trade prevention gate"), the developer has to remember to:
1. Add the variant to `BreakerType`.
2. Update `BreakerType::category()`.
3. Add a state field, a tripped-at field, a context field to `CircuitBreakers`.
4. Update `all_active`, `collect_denials`, `state`, `set_state`.
5. Update the `update_gate_conditions` table.
6. Update the gate-trip-Instant arms in `trip_breaker`, `clear_gate`, `reset_breaker`.
7. Add a field to `GateConditions`.

That's six places, all easy to forget. The state machine lends itself to a `[BreakerState; 10]` indexed by `BreakerType as usize`, with the gate-condition evaluation a parallel `[bool; 3]` indexed by gate-position. That refactor is out-of-scope for 5.1 but should be on the table when 5.2-5.6 are integrated.

**Fix:** none required for 5.1. Captured for the epic 5 retro / refactor.
