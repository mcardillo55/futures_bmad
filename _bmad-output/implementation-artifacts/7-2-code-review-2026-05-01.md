### SHOULD-FIX

#### S-1 — `ReplayResult::compare()` silently ignores vector length divergence; `is_deterministic` can be `true` even when run lengths differ

**Source:** Acceptance Auditor + Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:243-283` (`compare_with_epsilon`), `crates/engine/src/replay/orchestrator.rs:290-330` (`compare_signal_series`)

The compare method iterates `n = a.len().min(b.len())` for `trade_pnls_qt`, `obi_values`, `vpin_values`, `microprice_values`, `regime_states`, and `snapshots`. If two runs produce vectors of differing length (e.g. one has 1000 obi_values and the other has 999), the divergence in tail items is silently dropped and the prefix comparison alone determines `is_deterministic`. This is the determinism-leak failure mode the spec is designed to detect.

The `total_trades` and `net_pnl_qt` scalars ARE compared (`orchestrator.rs:229-242`, with sentinel indices `usize::MAX` and `usize::MAX - 1`), but the underlying vector lengths are not. `total_events` is not compared at all (it's a captured field but not iterated in `compare`).

This is unlikely to bite under the current test fixture (the data source is rewound and the same source is iterated, so lengths should always match), BUT the entire purpose of `compare()` is to detect determinism violations in the wild — including ones that produce length divergence (a dropped event, an early-exit branch, an off-by-one in snapshot capture). The current implementation of `compare` would report `is_deterministic = true` for "first 999 of 1000 events match, run B dropped the last one".

The `snapshot_interval_comparison` test (`tests/replay_determinism.rs:352-358`) does its own out-of-band length check (`assert_eq!(result_a.snapshots.len(), result_b.snapshots.len(), ...)`), tacitly acknowledging that `compare()` doesn't catch this. The other three tests don't, so length divergence in `obi_values`, `vpin_values`, `microprice_values`, `regime_states`, or `trade_pnls_qt` would slip through silently.

**Fix:** In each `compare_*` helper and in `compare_with_epsilon`, when `a.len() != b.len()`, push a sentinel mismatch (analogous to the `total_trades` / `net_pnl_qt` length-divergence handling at `orchestrator.rs:229-242`). Suggested:

```rust
if a.len() != b.len() {
    report.tier1_mismatches.push(FixedPriceMismatch {
        index: usize::MAX - 2,           // distinct from net_pnl_qt sentinel
        expected_raw: a.len() as i64,
        actual_raw: b.len() as i64,
    });
}
```

(Or add a dedicated `LengthMismatch` variant and a `length_mismatches: Vec<LengthMismatch>` field on `DeterminismReport` — slightly more surface change but the sentinel-index approach matches the existing convention.) Apply to all five vector pairs. Also surface `total_events` divergence the same way.

#### S-2 — `assert_initial_pipeline_state` only checks `SignalSnapshot` fields (name/value/valid/timestamp), not the underlying signal state (buckets, counters, etc.)

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:1076-1104`

The post-reset assertion's docstring claims "verifies signal state is at initial values after reset" (Story 7.2 Task 3.2). In reality it constructs a reference pipeline, takes snapshots from both, and compares the four fields exposed by `SignalSnapshot`: `name`, `value`, `valid`, `timestamp`. None of these include the internal accumulators that determinism relies on:

- `VpinSignal::buckets` (`VecDeque<VolumeBucket>`)
- `VpinSignal::current_bucket` (`VolumeBucket`)
- `VpinSignal::last_trade_price` (`Option<FixedPrice>`)
- `VpinSignal::last_classification` (`Option<Side>`)
- `ObiSignal::update_count` (`u64`)

A buggy `reset()` implementation that forgets to clear, say, `VpinSignal::buckets` would NOT trigger this assertion because all four snapshot fields would still match a freshly-constructed signal (post-reset `last_vpin = None`, `valid = false`, `last_update_ts = default()`, `name = "vpin"` — all match the fresh constructor). The leftover buckets would only manifest as divergence when subsequent events flow through.

The `signal_reset_required` test catches this class of bug indirectly (scenario A's "without reset, runs differ" assertion fires when leftover state changes the output), but only when the input data exercises the leftover state. A test fixture that happens not to exercise the bucket leak — or a future signal that stores state which `reset()` clears but only matters under certain inputs — would silently slip past.

This makes the assertion's promise narrower than its docstring suggests. Two possible directions:
1. **Document the actual scope.** Update the docstring to "verifies the public `SignalSnapshot` view post-reset matches a fresh pipeline; full state cleanliness is verified only indirectly via `signal_reset_required`." Cheap, honest.
2. **Strengthen the check.** Add a `SignalPipeline::reset_invariants_ok()` method that returns `bool` (or panics with details) by introspecting each signal's internal state directly. This pulls reset-correctness invariants into the signals themselves, which is where they belong.

**Fix:** Option (1) is sufficient for V1. Option (2) is the right long-term shape but is more work and out of scope for this story.

### NICE-TO-HAVE

#### N-1 — Test parquet fixture is regenerated-if-missing; a stale or generator-divergent regeneration would silently re-baseline

**Source:** Edge Case Hunter
**Location:** `crates/engine/tests/replay_determinism.rs:149-168` (`ensure_dataset`)

`ensure_dataset()` checks for the committed parquet at `tests/data/market/DETERMINISM/2026-04-18.parquet`; if missing, it regenerates from `generate_test_events()`. The intent is "self-contained — works in a fresh worktree". The risk:

1. A developer accidentally `rm`s or git-skips the parquet file → next `cargo test` regenerates it from `generate_test_events`. If `generate_test_events` has DRIFTED from the version that produced the committed snapshot, the regenerated parquet won't match the snapshot and `replay_determinism_snapshot` will fail. That's actually the correct outcome.
2. BUT: a developer who CHANGES `generate_test_events` (e.g. tweaks the trade size formula at line 119) and ALSO deletes the parquet will get a regenerated parquet from the new generator, the snapshot will fail, they'll run `cargo insta review`, accept the new snapshot, and commit BOTH the new snapshot AND the new parquet — re-baselining the determinism gate with no record of the old baseline. The story's regression mechanism (snapshot file is the gate) only works if the parquet file IS treated as immutable input, not regenerated input.

This is a footgun rather than a defect. Two mitigations:
1. **Commit a hash of the parquet** alongside it, and `ensure_dataset` asserts the regenerated parquet matches the committed hash (via SHA-256 or similar). Forces the developer to consciously update both.
2. **Remove the regeneration path entirely** and require the parquet file to be present (panic with a clear "run `cargo run --bin gen_test_data` to regenerate" message if missing). Simpler; the cost is that fresh worktrees need an explicit regen step.

**Fix:** Option (1) is the ideal — keeps "fresh worktree works" while preventing accidental re-baseline. Option (2) is acceptable for V1 if the regen complexity outweighs the risk.

#### N-2 — `compare_signal_series` records `(Some, None)` and `(None, Some)` divergence with `actual = NaN` — but `NaN` is itself a "tier 2 mismatch" signal in the same code path, conflating two failure modes

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:312-328`

```rust
(Some(x), None) => {
    report.tier2_mismatches.push(SignalMismatch {
        index: i,
        expected: x,
        actual: f64::NAN,
        abs_diff: f64::INFINITY,
    });
}
```

The `actual: f64::NAN` here is a SENTINEL meaning "the other run had `None`". Elsewhere in the same compare path, an actual NaN value in one of the runs (which the signal code is supposed to guard against, but a future signal might accidentally emit) would also produce a mismatch via the `nan_either` check at line 302. Both cases produce a `SignalMismatch` whose `actual` field is NaN — distinguishable only by the `expected` field's NaN-ness (sentinel: expected non-NaN, actual NaN; real NaN: expected NaN OR actual NaN OR both).

The `Display` impl at `determinism.rs:222-232` prints `actual={act}` which renders as `NaN` in either case, so the human-readable failure report can't distinguish "the other run had `None`" from "the signal code emitted NaN". For a regression report, that's a real diagnostic gap.

**Fix:** Add a separate `SignalMismatch` variant or a discriminator field (e.g. `kind: enum { ValueMismatch, NoneVsSome, NaN }`) so the Display output can clearly say "run A had Some(0.5), run B had None" vs "signal emitted NaN at index 42". Alternatively, use `f64::INFINITY` as the sentinel for "other side was None" (still a sentinel, but distinguishable from any real signal value since OBI/microprice/VPIN are bounded).

#### N-4 — `BarBuilder::flush` is called unconditionally at end-of-replay; no test verifies the final partial bar is included in `regime_states`

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:1000-1007` and `crates/engine/src/replay/orchestrator.rs:1207-1217`

`run_with_capture_internal` flushes the final partial bar after the event loop ends, pushing one extra `regime_states` entry if a regime detector is configured AND a partial bar exists. This adds one extra regime state per run. The committed snapshot has 17 regime states for a 1000-event run with `bar_interval_nanos = 60_000_000_000` (1 minute) — events span 1000 seconds = 16 minutes 40 seconds = 16 complete bars + 1 partial. The `+1` from the flush is consistent across runs (deterministic), so it doesn't break the determinism gate, BUT:

1. If the final event aligns exactly on a bar boundary, the partial bar is empty AND `bar_builder.flush()` returns `Some(...)` based on whatever `current` is — actually, `flush()` returns `self.current.take()` so if `current` was already finalised by `consume()` returning `Some(closed_bar)` (which then SET `self.current = Some(BarInProgress { ...new bucket })`), the flush would return that NEW (potentially empty-volume) bar. Let me re-read the consume logic at `orchestrator.rs:1180-1201` — yes, when a new bucket starts, `current` is replaced with a fresh `BarInProgress` containing the triggering trade event's data. So `flush` always returns the in-progress bar with at least one trade in it (volume > 0). OK, that's safe.
2. The behavior IS deterministic but it's untested. No test asserts that the regime_states length includes the flush. If a future change to `run_with_capture_internal` accidentally drops the flush (e.g. a refactor extracts the loop body), the snapshot would change by one entry and surface the regression — but only if the test fixture's last event is mid-bar. Worth adding a unit test for `BarBuilder::flush` directly.

**Fix:** Add a unit test for `BarBuilder` that constructs an instance, feeds a few trades within one bucket boundary, calls `flush`, and asserts the returned `Bar` has the expected OHLCV fields. Lives in `orchestrator.rs`'s existing `mod tests` block.

#### N-5 — `PipelineSnapshotSerde::PartialEq` is derived (uses `f64::PartialEq` which is NaN-asymmetric); the `snapshot_interval_comparison` defence-in-depth assert would NOT catch a NaN snapshot value

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:171-176` and `crates/engine/tests/replay_determinism.rs:373-376`

`PipelineSnapshotSerde` derives `PartialEq` (and `Debug`, `Clone`, `Serialize`, `Deserialize`). The derived `PartialEq` uses `f64::PartialEq`, which returns `false` for `NaN == NaN`. The test at line 373 (`assert_eq!(result_a.snapshots, result_b.snapshots, ...)`) is described as "defence in depth: catches a future bug where `compare` accidentally short-circuits without iterating".

But: if a signal ever emits NaN (the code guards against this with `is_finite` checks, but a future signal might miss the guard), both runs would produce identical NaN snapshots, and the `assert_eq!` would FAIL because NaN != NaN. The test would report "snapshot vectors must be PartialEq across deterministic runs" — falsely, because the runs ARE deterministic, just produce NaN. The dev's `compare_signal_series` already handles this case correctly (NaN ⇒ mismatch); the defence-in-depth check is the weaker assertion in this scenario.

Low practical risk because the signal code currently guards against NaN. Worth a one-line note in the test docstring explaining the NaN edge.

**Fix:** Either (a) add a comment to `snapshot_interval_comparison` noting the NaN limitation of derived `PartialEq`, or (b) implement `PartialEq` manually on `SignalSnapshotSerde` to treat NaN-NaN as equal (matches the determinism notion better than f64's IEEE semantics, but introduces a non-standard equality).

## Scope Decision Verdicts

### 1. Tier 1 stored as raw `i64` quarter-ticks instead of via `FixedPrice`'s serde representation

**Dev claim (Completion Notes):** "All `FixedPrice`-derived values are stored as raw quarter-tick `i64` so Tier 1 comparisons remain bit-identical without depending on the (lossy, f64-based) `FixedPrice` serde representation."

**Verdict: CORRECT and well-motivated.**

`FixedPrice`'s default serde (if it round-trips through `to_f64`/`from_f64`) would lose precision for values that don't have an exact f64 representation. Capturing `raw()` as `i64` sidesteps the issue entirely. The conversion happens at capture time (`trade_pnls_qt: Vec<i64>` populated via `std::mem::take` from `trade_stats.per_trade_pnls_qt`) and the `compare` method does pure integer equality. Tier 1 is now ACTUALLY bit-identical, not "approximately bit-identical modulo f64 quirks". This is the right architectural call.

### 2. `SignalSnapshotSerde` and `PipelineSnapshotSerde` wrappers (own `String` for `name` instead of `&'static str`)

**Dev claim (Completion Notes):** "`SignalSnapshot` carries `name: &'static str` which serde can't own through a `Deserialize` round-trip. The wrapper holds an owned `String` instead so insta YAML round-trips cleanly without changing the core trait surface."

**Verdict: CORRECT.**

Keeps the core trait minimal while solving the serde ownership problem at the replay-result boundary. The `From<&SignalSnapshot> for SignalSnapshotSerde` impl is straightforward and the conversion is explicit (only happens during snapshot capture). No production performance impact (snapshot capture is rare and only during replay capture mode).

### 3. `RegimeState` in `core` gained a `Serialize` derive

**Dev claim (Completion Notes):** "`RegimeState` gained a `Serialize` derive (it already had `Deserialize`). This is the minimum surface change in `core` required for the new `ReplayResult`'s insta snapshot."

**Verdict: CORRECT.**

Minimum-blast-radius change. `RegimeState` is a small enum with 4 variants and `as_str` already provides the canonical string form. The `Serialize` derive is consistent with the existing `Deserialize` derive. Updated docstring on `RegimeState` (`crates/core/src/traits/regime.rs:11-14`) explains the rationale clearly.

### 4. Pipeline + regime detector OWNED across `run_with_capture*` calls (not constructed per-call)

**Dev claim:** "The orchestrator now owns the pipeline and detector across calls so the no-reset test path (`run_with_capture_no_reset`) actually observes residual state."

**Verdict: CORRECT and required.**

Without this, `run_with_capture_no_reset` would effectively behave the same as `run_with_capture` (each call would get a fresh pipeline and there would be no state to leak). The owned-pipeline pattern is what makes the AC2 verification ("no residual state affects results") testable. The `Option<SignalPipeline>` + `take()` + restore pattern is a clean way to give the inner loop unfettered access to the pipeline while keeping it owned by `self`.

One small wart: the regime detector's "rebuild on reset" handling at `orchestrator.rs:877-883` is asymmetric with the pipeline's "always exists, call `reset()`" pattern. The reason is documented in the inline comment ("the trait has no reset method; rebuilding is the canonical clean-slate operation"). Acceptable for V1; if a future story adds `reset()` to `RegimeDetector` trait, harmonise.

### 5. Test parquet fixture is regenerated-if-missing rather than required-present

**Verdict: ACCEPTABLE BUT NOTE THE FOOTGUN — see N-1.**

The "fresh worktree works" property is valuable. The risk is that a generator change + parquet deletion silently re-baselines the determinism gate. N-1 proposes a hash check as the fix.

### 6. `unknown_blocks_trading` exposed in TOML (and thus `ThresholdRegimeConfig` plumbed through the test config) but NOT obviously used by the determinism check

**Verdict: ACCEPTABLE — the field is consumed by the regime classifier's logic and its value affects the resulting `RegimeState` series. ## Other Observations

- **`Cargo.lock` shows insta + console + similar transitive deps.** Total +37 lines of lock additions for the insta dev-dep. Reasonable.
- **`#![deny(unsafe_code)]` is NOT set on `replay/determinism.rs`.** The convention used elsewhere (e.g. story-6.x regime files) includes this lint at the file head. For consistency, add it to `determinism.rs`. Minor.
- **`insta` version 1.46.3 declared in workspace** (`crates/engine/Cargo.toml` adds it), but the resolved version is 1.47.2 (per `Cargo.lock`). Cargo's caret operator allows minor-version upgrades, so this is expected. No issue.
- **`compare_with_epsilon` is `pub`** but `compare` is the documented entry point. Nice to have an escape hatch for tests that want to assert a tightened epsilon, though no test currently uses it.
- **`DeterminismReport::finalize()` is idempotent** and updates `is_deterministic` from `total_mismatches() == 0`. Correct, but the field is `pub` — a caller could mutate the mismatch vectors and forget to call `finalize`. Consider making the field `pub(crate)` and exposing only a getter, or move to a builder pattern. Low priority.
- **`BarBuilder` is private to the orchestrator module.** Good — no need to expose it as a public API surface.
- **`SignalInstrumentationConfig::default()` uses `obi_warmup = 10, max_spread_qt = 16, vpin_bucket_size = 100, vpin_num_buckets = 10`.** The TOML test config overrides all four (warmup=5, max_spread=16, bucket_size=50, num_buckets=5). The `main.rs` uses the default. No determinism issue (the default is only used by the binary, which doesn't capture).
- **`RegimeInstrumentationConfig::default()` sets `bar_interval_nanos = 60_000_000_000` (1 minute).** Reasonable default for futures intraday data.
- **No `Send + Sync` annotation tests on the new `ReplayResult` / `DeterminismReport`.** Both are `Serialize + Deserialize`; if a future story wants to send them across threads (e.g. ship a determinism report from a worker thread to a main reporter), this could become a problem. Add `_assert_send::<ReplayResult>()` if multi-threaded usage is anticipated.
- **`ReplayResult::compare` returns by value** rather than mutating an existing report. Correct API choice — comparison is a pure function.
- **The journal is NOT exercised in `run_with_capture`.** `attach_journal` is a method on the orchestrator from Story 7.1; `run_with_capture_internal` does not call `journal_system_event` on start/complete (search shows the calls only exist in `run`, not in `run_with_capture_internal`). This is a deliberate scope decision — the determinism check doesn't need journal output — but the dev should document it. A test that attaches a journal then calls `run_with_capture` would silently produce no journal entries, which could surprise.
