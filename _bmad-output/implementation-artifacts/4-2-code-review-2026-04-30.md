## Findings Detail

### Should-Fix

**S-1 â€” Over-fill / fill_size > remaining_quantity is silently accepted.**
`apply_fill` does not validate that incoming `fill.fill_size` is consistent with `tracked.remaining_quantity` or `original_quantity`. A `FillType::Partial { remaining: 100 }` for an order originally sized 5 will happily set `tracked.remaining_quantity = 100`. A `FillType::Full` with `fill_size = 999` for a qty-1 order is also accepted. Real-world cause: malformed exchange message, broken wire protocol decoder, replay of a stale message. Effect: corrupt position arithmetic via `Position::apply_fill` downstream.
**Fix:** in `crates/engine/src/order_manager/mod.rs:243-251`, sanity-check `fill.fill_size <= tracked.remaining_quantity` and `Partial { remaining }` < `tracked.remaining_quantity`. Reject with `FillOutcome::InvalidTransition` (or a new `FillOutcome::InvalidFillSize`) and log; leave `tracked` untouched.
**Location:** `crates/engine/src/order_manager/mod.rs:151-285`.

**S-2 â€” Zero-size non-rejected fills are accepted as terminal.**
A `FillType::Full` with `fill_size = 0` (or `Partial { remaining = X, fill_size = 0 }`) currently transitions the order to `Filled` with no diagnostic. The reject-fill path *intentionally* uses `fill_size = 0` for rejections, but a non-Rejected zero-size fill is almost certainly a protocol bug. Without a guard the order is dropped from tracking and the position is unchanged (because `Position::apply_fill` early-returns on `fill_size == 0`), leaving a phantom "filled" order with no fill in the position state.
**Fix:** in `apply_fill`, treat `fill_size == 0 && !matches!(fill_type, FillType::Rejected { .. })` as `FillOutcome::InvalidTransition` with a warn log.
**Location:** `crates/engine/src/order_manager/mod.rs:182-251`.

**S-3 â€” Partial-then-Rejected fills strand the order in `PartialFill`.**
The state machine (`crates/core/src/types/order.rs:36-49`) has no `PartialFill -> Rejected` arc. A real-world sequence (broker accepts entry, fills 1-of-3, exchange halts and rejects the remainder) currently hits the InvalidTransition branch (`order_manager/mod.rs:222-239`), the warn fires, and the order remains in `PartialFill` indefinitely with `remaining_quantity > 0` â€” no journal record, no terminal cleanup.
**Fix:** either (a) add `PartialFill -> Cancelled` to `OrderState::can_transition_to` and reroute partial-then-reject through Cancelled, or (b) add `PartialFill -> Rejected`. Either way, expose a path to terminate the partial. (b) preserves the operator-facing semantics ("the rest got rejected") more clearly.
**Location:** `crates/core/src/types/order.rs:28-50`; `crates/engine/src/order_manager/mod.rs:222-239`.

**S-4 â€” Synthetic reject-fill can be silently dropped on full FillQueue.**
In `route_pending_orders` (`crates/broker/src/order_routing.rs:347`), the synthetic rejection fill is written via `fill_producer.try_push(reject)` whose return value is ignored. If the FillQueue is full at that moment (engine stalled), the rejection is logged-and-dropped at the producer level; the engine never sees the rejection, the order stays tracked indefinitely, and the audit trail is incomplete.
**Fix:** at minimum, log the dropped rejection at `error` with order_id + decision_id + reason (the producer logs already, but the routing layer should explicitly note the consequence â€” order stranded). Better: a small bounded retry (e.g., back off + retry once) before logging the strand. Long-term resolution is the broker reconciliation in story 4.5; the comment in code should make that handoff explicit.
**Location:** `crates/broker/src/order_routing.rs:347`.

**S-5 â€” `SubmissionError::Timeout` maps to `RejectReason::Unknown` and emits a Rejected fill.**
A submission timeout means we don't know whether the exchange received and accepted the order. Treating it as Rejected is unsound: if the order *did* hit the exchange, the engine moves on as if it didn't, then a real fill arrives later and surfaces as an Orphan. The order state machine already has `Submitted -> Uncertain` precisely for this case (`crates/core/src/types/order.rs:34`).
**Fix:** introduce `FillType::Uncertain { reason }` (or have the routing loop NOT synthesize a fill on `Timeout`, leaving the order in `Submitted` for reconciliation in 4.5). At minimum, document that timeouts are mis-classified pending 4.5.
**Location:** `crates/broker/src/order_routing.rs:269-280`.

### Nice-to-Have

**N-1 â€” `OrderSubmitter` vs `BrokerAdapter::submit_order` is a real second source of truth.**
The dev's deviation note is correct that the rename ripple is out of scope, but the resulting code has two ways to "submit an order": `BrokerAdapter::submit_order(OrderParams) -> Result<u64, BrokerError>` (still in `core::traits`, exercised by `RithmicAdapter` and `MockBrokerAdapter` and the testkit) and `OrderSubmitter::submit_order(&OrderEvent) -> Result<(), SubmissionError>` (the routing-loop path). Different signatures, different error types, different ownership semantics. Future reviewers will ask which is canonical. Recommend either consolidating in story 4.4 (when the OrderState machine lands) or adding a doc comment on `BrokerAdapter::submit_order` explaining that the OrderEvent-shaped routing path supersedes it for the production order pipeline.
**Location:** `crates/core/src/traits.rs` (BrokerAdapter) vs `crates/broker/src/order_routing.rs:200-206` (OrderSubmitter).

**N-2 â€” `route_pending_orders` drains in a tight loop with no `tokio::task::yield_now()`.**
Under burst load (engine pushes a flurry of orders, mock submitter is sub-microsecond), the `while let Some(event) = consumer.try_pop()` loop won't yield to other Tokio tasks until the queue is empty. Bracket-order bursts (story 4.3) plus a fast OrderPlant could reasonably push 64+ events between scheduler ticks. Add a `tokio::task::yield_now().await` every N events (e.g., every 32) or after a wall-clock budget.
**Location:** `crates/broker/src/order_routing.rs:306-350`.

**N-3 â€” `DecisionIdMap` is unbounded for non-terminal partials.**
Entries are removed only on `is_terminal()` reports (`order_routing.rs:533-535`). An order that gets a partial fill and never a terminal (broker disconnects mid-stream) leaves the entry forever. Story 4.5 reconciliation will eventually clean up, but until then, a long-running session has a slow leak. Consider a TTL or a bounded LRU as defense in depth, or at least a warn log when the map exceeds a threshold (e.g., 1024 entries).
**Location:** `crates/broker/src/order_routing.rs:370-405`.

**N-4 â€” Auto-upgrade `Submitted -> Confirmed` on first non-rejection fill skips the journal record for that arc.**
`apply_fill` (`order_manager/mod.rs:175-179`) silently upgrades `Submitted -> Confirmed` and then journals only the `Confirmed -> Filled` (or `Confirmed -> PartialFill`) transition. Forensic replay of the audit trail will see Confirmed appear without an explicit `Submitted -> Confirmed` record. Either emit two journal entries (Submittedâ†’Confirmed, then the fill), or document this in the schema notes as an intentional shortcut.
**Location:** `crates/engine/src/order_manager/mod.rs:175-262`.

**N-5 â€” Orphan fills journal `decision_id = Some(0)` rather than `None`.**
`apply_fill` always wraps `fill.decision_id` in `Some(...)` (`order_manager/mod.rs:258`), so an orphan fill (where `decision_id` is the 0 sentinel from `publish_execution_report`) is journaled as `Some(0)`. This is ambiguous with a legitimate `decision_id = 0` and breaks the schema's `decision_id` nullability semantics. Map `0 -> None` at the journal boundary, or change the sentinel to a distinct discriminator.
**Location:** `crates/engine/src/order_manager/mod.rs:258`; cross-ref `crates/broker/src/order_routing.rs:474-482`.

**N-6 â€” `publish_execution_report` does not validate `side` against the original submission.**
The exchange echo's `side` is taken at face value. A garbage exchange payload that flips Buyâ†’Sell would propagate to `Position::apply_fill` and corrupt the position. Defensive: look up the `OrderEvent` (the routing loop has it on the success path) and reject the report if sides disagree. Cheap and useful given how cheap this is to mistake on the wire.
**Location:** `crates/broker/src/order_routing.rs:466-537`.

## Verdict on the Three Flagged Deviations

1. **`OrderSubmitter` shipped instead of `BrokerAdapter::submit_order` (Task 3.1).** **JUSTIFIED but worth addressing in story 4.4.** The blast-radius argument is real (testkit, MockBrokerAdapter, RithmicAdapter all depend on the existing `BrokerAdapter::submit_order(OrderParams)`). However, the result is two parallel "submit" trait surfaces, which will confuse future readers (see N-1). Recommend a doc comment now and consolidation in 4.4 when the WAL-write-before-submit invariant lands.

2. **`RithmicSubmitter` is a placeholder.** **JUSTIFIED.** The seam is clean: `OrderSubmitter` is a single async trait method with a clear error type. The placeholder's docstring mappings (`OrderEvent -> RequestNewOrder` field-by-field at `:215-223`) are the right shape; swapping in the real `rithmic-rs` `OrderPlantHandle::send_new_order(...)` should be a localized change. The reject-fill pathway is end-to-end exercised against the stub. Wiring the real OrderPlant is straightforward; no architectural debt introduced.

3. **Live fill listener task deferred (Task 4.1).** **JUSTIFIED.** `publish_execution_report` is the right seam: pure function, takes an `ExecutionReport`, mutates the `DecisionIdMap`, and pushes a `FillEvent`. Tests drive it directly (`fill_listener_emits_partial_then_full_with_decision_id`, `fill_listener_logs_orphan_with_zero_decision_id`). The integration concern that *isn't* yet covered is end-to-end: `rithmic-rs` execution payload â†’ `ExecutionReport` decoding. That decoder lives in the deferred listener and should be unit-tested when it lands.
