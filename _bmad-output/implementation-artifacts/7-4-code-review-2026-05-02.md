## Top-Line Verdict: **APPROVE-WITH-CHANGES**

**Findings:** 1 BLOCKING / 5 SHOULD-FIX / 7 NICE-TO-HAVE

The source-tag stamping mechanism is well-engineered: a single `source_override` field on `JournalSender` rewrites the `source` column on every trade / order-state record at send time, and paper / replay orchestrators install the override exactly once via `attach_journal`. Schema migration is idempotent (PRAGMA table_info guard around `ALTER TABLE`), the column index pair (`idx_trade_events_source`, `idx_order_states_source`) supports the per-source `WHERE` filter, and the FIFO P&L pairing is consistent with the convention `replay::TradeStats` already uses. The 9 end-to-end integration tests exercise the channel + worker batching path (not just `write_event`), so the production hot-path is covered. Default `TradeSource::Live` and `'live'` SQL DEFAULT give belt-and-suspenders fail-closed semantics: a record that escapes without explicit tagging records as live, which is the conservative choice for backwards compatibility but also means **paper records that escape the override are silently miscoded as live** â€” see B-1.

One blocking concern outweighs the otherwise solid implementation:

1. **The "single seam" framing in the dev notes is incomplete: `JournalSender::with_source` only rewrites events that flow THROUGH the paper orchestrator's stored sender, but `OrderManager` and `BracketManager` â€” the only production code that emits `TradeEvent` records â€” receive their `JournalSender` independently at construction time and there is NO production wiring path that hands them a paper-tagged clone (B-1).** This is provable by `git grep "OrderManager::new\|BracketManager::new"` returning matches ONLY in test code; neither manager is constructed by `main.rs`, `startup.rs`, `event_loop.rs`, or the paper / replay orchestrators. Production paper mode today: zero `TradeEvent` records would land in the journal at all (entry fills are never journaled even in live; only `bracket.rs:599` exit fills are), and any state-machine transitions emitted by `OrderManager` use a sender whose `source_override` is `None`, so they record as `live`. The integration tests that "pass" mask this entirely by clone-and-stamping the sender into the test harness manually (the `record_fill` helper at `paper_trade_recording.rs:63-77`) and by calling `paper_sender.send(...)` directly. **The seam exists; it is not connected to anything in production.**

The remaining concerns are softer:

- **The orchestrator-level stamping deviates from the spec's "BrokerAdapter level" wording, but the dev's rationale for the deviation is correct: orders DON'T flow through `BrokerAdapter::submit_order` in paper mode** (S-1, also see Story 7.3 review S-1). The actual fill-recording path is `MockBrokerAdapter â†’ MockFillSimulator â†’ FillQueue â†’ OrderManager â†’ JournalSender`, and stamping at the adapter would not naturally reach the journal. So the dev's design choice is technically sound; the *spec* is the wrong layer for this responsibility. But the dev notes oversell ("a single seam") â€” see B-1.
- **The 500-trade `ReadinessReport` test confirms the algorithm scales without panicking, but the methodology is very thin for a "go/no-go decision for live deployment"** (S-2). The Sharpe estimate is non-annualised, the profit-factor edge cases are partially documented but partially incoherent (`0.0` when both wins and losses are zero â€” acceptable; `f64::INFINITY` when losses sum to exactly zero â€” acceptable; but no guard against `sum_losses_qt` overflowing on extreme datasets), there is no statistical-significance gate, no out-of-sample cross-validation hook, and the consecutive-loss-max counter is across the entire run with no rolling window. For a true go/no-go gate this is informational-only at best. The dev acknowledges this in the rustdoc ("informational; the deployment decision is human-made"), which is the right framing â€” but it should be stated in the AC verification too.
- **The migration is genuinely idempotent for the schema, but a real pre-7.4 journal that contains rows with `decision_id = 0` (the writer's "missing" sentinel) will round-trip those as `decision_id = None` even though the original record had no `decision_id` at all and the existing `error!` log line claimed it was an NFR17 violation** (S-3). This isn't a 7.4 regression but 7.4's new `JournalQuery` exposes the issue: `trace_decision(0)` returns every NFR17-violating row across all of history, scrambled together. Document the sentinel collision or use `NULL` for "missing" decision_id on the trade_events column too.
- **Per-symbol FIFO pairing is correct, but the `pnl_summary` and `paper_readiness_report` ignore the `kind = "submit"` / `"cancel"` rows entirely** (S-4). This is correct accounting, but means the reports are computed only over rows where `kind` is one of the five hard-coded fill variants. If a future story emits a fill with a different `kind` string (typo, new bracket leg type), it gets silently dropped from P&L. Better: compute P&L from rows with `size > 0` and a non-empty `decision_id`, or make the fill-kind list configurable.
- **The Replay path now writes `source = "replay"` records into the SAME database the paper / live runs use, but there's no enforcement that an operator pointing replay at a production journal won't pollute the paper-readiness report** (S-5). The `paper_readiness_report` correctly filters `WHERE source = 'paper'` so live and replay rows are excluded from the gate, but operator UX risk: a replay run against `data/journal.db` (the production default) silently mingles audit trails. Suggest defaulting replay to a separate file (`data/replay.db`) or requiring an explicit `--journal=` flag.

Probe answers (per review brief):

The actual fill-recording path in V1 is `MockBrokerAdapter â†’ MockFillSimulator â†’ FillQueue â†’ (in production: `OrderManager`) â†’ `JournalSender``. - **"Is the schema migration genuinely idempotent and won't break on existing journals from prior epics?"** PARTIALLY. The migration is idempotent for the column-add itself (`migrate_add_source_column` checks `column_exists` via `PRAGMA table_info` before issuing `ALTER TABLE`). The tests are honest about this in the inline comment ("mirror the resulting fill events into the journal as the order manager would in the production wiring"), but no production wiring exists yet. **A real `engine --paper` run today writes ZERO `trade_events` rows tagged `source = "paper"` to the journal.** The recording path's *machinery* works (verified by 9 tests); the *wiring* to drive it from the orchestrator's actual fill flow does not exist.

#### S-2 â€” `ReadinessReport` methodology is too thin to justify a "go/no-go decision for live deployment" â€” informational at best

**Source:** Edge Case Hunter + Acceptance Auditor
**Location:** `crates/engine/src/persistence/query.rs:120-185` (`ReadinessReport` struct, doc comments), `:401-471` (`compute_readiness_report`)

The report's metrics:
- `total_trades` â€” round-trip pair count
- `positive_expectancy` â€” net P&L > 0 (boolean)
- `win_rate` â€” wins / (wins + losses)
- `profit_factor` â€” sum of winning P&L / |sum of losing P&L|
- `max_drawdown` â€” largest peak-to-trough decline
- `sharpe_estimate` â€” mean / sample std-dev (NOT annualised)
- `consecutive_loss_max` â€” longest losing run
- `net_pnl` â€” cumulative

The dev correctly labels this informational ("the deployment decision is human-made"). For a real go/no-go gate, missing pieces include:

- **Statistical-significance test.** A 500-trade win rate of 52% is not statistically distinguishable from 50% at any reasonable confidence level. Without an n-required-for-confidence calculation, "positive expectancy over 500 trades" is a weaker signal than the AC implies.
- **Regime stratification.** Per-regime P&L (trending vs choppy vs unknown) is necessary because a strategy that won 80% in a trending regime over 400 trades and lost 35% in a chop regime over 100 trades shows positive expectancy overall, but is a false positive in a chop-dominant future. The journal has the `regime_transitions` table; the report doesn't join against it.
- **Drawdown-recovery analysis.** Max drawdown alone misses "drawdown N% recovered in K trades vs drawdown N% never recovered". The latter is a kill criterion; the former is normal.
- **Fee-aware accounting.** Acknowledged by dev as deferred to live wiring story. For a paper-vs-live comparison this matters because paper P&L without fees overstates expectancy by a meaningful margin (1-2 ticks per round trip on retail futures, in some markets).
- **Consecutive-loss in absolute time, not just count.** Five consecutive losses over 30 minutes is different from five consecutive losses over 30 days.

Edge cases in the current code:
- `profit_factor`: when `sum_losses_qt == 0 && sum_wins_qt == 0` â†’ returns `0.0`. When `sum_losses_qt == 0 && sum_wins_qt > 0` â†’ returns `f64::INFINITY`. Documented. Acceptable.
- `profit_factor` when `sum_losses_qt` overflows `i64` via `saturating_add(p)` for `p < 0`: at saturation the value clamps to `i64::MIN` and `unsigned_abs()` becomes `i64::MAX as u64 + 1` (i.e. `2^63`). The ratio then collapses meaning. Realistically requires ~`i64::MAX / 2` accumulated quarter-tick losses, which is `~4.6 * 10^18` quarter-ticks â€” not reachable in any realistic dataset, but worth a `debug_assert!` or comment.
- `sharpe_estimate` uses sample std-dev (`n - 1` divisor) but pairs it with raw mean â€” the ratio is dimensionally fine but isn't comparable to industry "Sharpe ratio" without annualisation. Document this divergence prominently.
- `consecutive_loss_max` resets on any non-loss including scratches (P&L = 0). Unusual â€” most analytics treat scratches as either "neither win nor loss" (counter pauses) or explicitly count them. Document the chosen convention.

**Fix options:**
1. **Sharpen the AC.** Change AC3 from "sufficient data for go/no-go decision" to "sufficient data for human reviewer to inform a go/no-go decision; quantitative gate is operator-defined."
2. **Document the metric semantics.** Add a `ReadinessReport` rustdoc section that calls out: not annualised, sample std-dev, scratches reset consecutive-loss counter, no statistical-significance test, no regime stratification, no fees.
3. **Add a `JournalQuery::regime_stratified_pnl` companion** (defer-able). Returns `BTreeMap<RegimeState, PnlSummary>`. The data is there; the join is a one-liner against `regime_transitions`.

Option 2 is cheap. Option 1 is dev-notes-only. Option 3 is genuinely useful but defer-able to the regime-aware-deployment story.

#### S-3 â€” `decision_id = 0` sentinel-as-`None` collision means `trace_decision(0)` returns every NFR17-violating row across all sources

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/persistence/journal.rs:643-644` (writer's sentinel), `crates/engine/src/persistence/query.rs:330-334` (reader's sentinel mapping)

Writer (`journal.rs`):
```rust
// Persist decision_id as 0 sentinel when missing â€” keeps NOT NULL.
r.decision_id.unwrap_or(0) as i64,
```

Reader (`query.rs`):
```rust
decision_id: if decision_id == 0 {
    None
} else {
    Some(decision_id as u64)
},
```

Effects:
1. A `TradeEventRecord` with `decision_id = None` (NFR17 violation, surfaced as `error!` log at `journal.rs:629`) lands in the DB as `decision_id = 0`.
2. `JournalQuery::trace_decision(0)` returns EVERY such row across the entire journal â€” paper, live, replay all mixed. Not what a reviewer expects.
3. A legitimate `decision_id = 0` (the very first decision after a fresh signal-id-generator startup, if the generator starts at 0) would round-trip as `None`, scrambling the audit trail.
4. The `order_states` table uses `decision_id INTEGER` (nullable) and the writer correctly uses `r.decision_id.map(|id| id as i64)` for the order-states write â€” so order-state rows with `decision_id = NULL` are distinguishable. Only `trade_events` has the sentinel collision.

The asymmetry between `trade_events` (NOT NULL with sentinel-as-None) and `order_states` (NULL allowed) is itself surprising and undocumented.

**Fix options:**
1. **Migrate `trade_events.decision_id` to NULLable** (matches `order_states`). Breaking the NOT NULL constraint requires SQLite ALTER TABLE rewrite (12-step recipe) or an idempotent `ALTER TABLE â€¦ DROP NOT NULL` (SQLite supports it as of 3.35). Either is a real schema migration; defer-able.
2. **Make signal-id generation start at 1** (or any non-zero value). Eliminates the legitimate-zero collision. Easy, non-breaking.
3. **In the writer, log `error!` AND skip the row** (rather than persisting with sentinel). The current behavior persists invalid data; logging-and-dropping would surface the NFR17 violation more cleanly. Risk: silent data loss for malformed events.
4. **In `trace_decision`, refuse `0`** â€” return an error. Operators tracing decision 0 either typo'd or are looking for NFR17-violating rows; both are operator error.

Option 4 is the cheapest defensive fix. Option 2 is the cleanest (one-liner in the signal-id generator). Option 1 is the architecturally correct long-term answer. Document whichever you pick.

#### S-4 â€” `pnl_summary` and `paper_readiness_report` silently drop any fill row whose `kind` doesn't match the hard-coded list of five strings

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/persistence/query.rs:549-554` (`is_fill_kind`)

```rust
fn is_fill_kind(kind: &str) -> bool {
    matches!(
        kind,
        "fill" | "partial_fill" | "tp_fill" | "sl_fill" | "entry_fill"
    )
}
```

Any row with `kind` outside that set is excluded from P&L, win rate, drawdown, etc. Today the production `BracketManager` only emits `"tp_fill"` and `"sl_fill"`, and tests use `"fill"`. So the current set is adequate.

But:
- Story 4.5's `OrderManager::on_fill` does NOT emit `TradeEvent` at all â€” only `OrderStateChange` (verified via grep: zero `JournalEvent::TradeEvent` in `order_manager/mod.rs`). So entry fills are never recorded as trade events in production. The integration tests use `"fill"` to simulate what entry-fill recording WOULD look like.
- A future change that adds a new fill kind ("liq_fill", "manual_close", "entry_fill", "modify_fill") will silently drop those rows from P&L â€” the test suite passes because it uses `"fill"`.
- The string match is order-sensitive: a typo (`"tpfill"` vs `"tp_fill"`) silently excludes the row. No log line.

Better: identify fill rows structurally rather than by string match. A row is a fill if `size > 0` AND the `kind` is not one of a known non-fill set (`"submit"`, `"cancel"`, `"reject"`). Or: emit `kind` as an enum with a `From<&str>` parser and pattern-match on the variant.

**Fix options:**
1. **Document the contract.** Add a rustdoc section listing the five recognized kinds and explicitly call out that any other kind is silently excluded. Reduces footgun visibility but doesn't fix the root cause.
2. **Invert the predicate.** `is_fill_kind` checks `!matches!(kind, "submit" | "cancel" | "reject")` instead. Future-proof against new fill kinds. Risk: a typo in the writer ("tpp_fill") would now BE counted as a fill, possibly with incorrect P&L semantics.
3. **Make the kind an enum.** Refactor `TradeEventRecord::kind: String` â†’ `kind: TradeKind` (enum). Forces every emit site to choose from a known set; new kinds require a code change in the enum + `is_fill_kind` together.

Option 3 is the structurally correct fix. Option 1 is the cheap honesty fix. Option 2 is risk-symmetric (different but not strictly better).

#### S-5 â€” Replay tagging now coexists with paper / live in the same database; no operator-UX guard against a replay run polluting paper-readiness data

**Source:** Edge Case Hunter
**Location:** `crates/engine/src/replay/orchestrator.rs:626-628` (`attach_journal` re-tags with `Replay`); `crates/engine/src/persistence/journal.rs:40` (`DEFAULT_DB_PATH = "data/journal.db"`)

The replay orchestrator now writes `source = "replay"` records into the journal. Before 7.4, the journal had no source distinction and replay records were indistinguishable from live records (also a real issue, but one 7.4 fixes). Now:

- `JournalQuery::paper_readiness_report` correctly filters `WHERE source = 'paper'`, so replay rows do not pollute the paper gate.
- `JournalQuery::trade_count(Replay)` correctly returns only replay rows.
- BUT: an operator running `engine --replay --parquet=2026-04-18.parquet` against the production `data/journal.db` (the default for `EventJournal::open_default()`) silently mingles replay audit trails into the production database. The journal grows unboundedly with each replay; the `paper_readiness_report` is correctly isolated, but `trace_decision` (which queries by `decision_id` regardless of source) returns rows from BOTH the replay and any production paper / live runs that happened to use the same decision_id.
- `decision_id` collision risk: replay reuses the recorded decision_ids from the original session; paper / live generate new ones at runtime. If the same id existed in the original recording AND was generated by a later production run, `trace_decision(N)` returns rows from BOTH, mixed across sources.

**Impact:**
- Operator reading a `trace_decision` result has to manually filter by source to disambiguate.
- The journal file grows without bound across replay runs (each replay adds full event history). For a regression test running 50 replays a day, this is meaningful.
- No `JournalQuery::trace_decision_by_source(decision_id, source)` variant exists for the operator who wants the production-only trace.

**Fix options:**
1. **Default replay to a separate database.** `ReplayOrchestrator::new` constructs its own `EventJournal::new(Path::new("data/replay.db"))`. Requires plumbing the path into the config; cheap.
2. **Add `JournalQuery::trace_decision_by_source(decision_id, source)`.** Filter at the SQL level so operators don't have to post-filter in Rust.
3. **Document the convention.** Add a rustdoc on `ReplayOrchestrator::attach_journal` that says: "the same journal can be shared with paper / live runs; use `JournalQuery::trade_count(Replay)` and friends to disambiguate, and consider passing a separate journal for regression testing to avoid bloating the production audit trail."

Option 1 is the operationally correct fix. Option 3 is the cheapest. Option 2 is a useful read-side companion to either.

#### N-5 â€” `JournalQuery::trades_by_source` always reads every column even when callers only need the count or the source filter; no `count_by_source` companion at the SQL level

## Scope Decision Verdicts

### 1. Source tag stamped at `JournalSender` instead of `BrokerAdapter` (spec deviation)

**Dev claim (Completion Notes):** "Per Story 7.4 Task 2, the source tag is set 'at the BrokerAdapter level' â€” pragmatically, this lands as a single override on `JournalSender`."

**Verdict: CORRECT DESIGN, MISLEADING DEV NOTES.** The dev's design choice is correct (Story 7.3 S-1: orders bypass `BrokerAdapter::submit_order` in V1, so adapter-level stamping wouldn't reach the journal). But the dev notes treat the deviation as merely "pragmatic" rather than as a substantive deviation from the spec. The honest framing: the spec is wrong because the architecture it presumes (adapter is on the order path) doesn't exist in V1. Update the dev notes to acknowledge this explicitly. Update the spec to match. **See S-1.**

### 2. Idempotent migration via `column_exists` + `CREATE INDEX IF NOT EXISTS`

**Dev claim:** "Idempotent â€” guarded by a runtime schema check so a fresh DB (already has the column) skips the ALTER."

**Verdict: CORRECT.** The `migrate_add_source_column` function uses `PRAGMA table_info` to check column existence before issuing `ALTER TABLE`. The index creation uses `CREATE INDEX IF NOT EXISTS`. The migration is genuinely idempotent and the `migration_adds_source_column_to_pre_story_7_4_journal` test exercises the upgrade path explicitly. One real edge: the migration assumes the default is `'live'`; an operator who manually added the column with a different default would not be normalized. Document this assumption.

### 3. `TradeSource::Live` as the default to fail closed

**Dev claim (Completion Notes):** "Default is [`TradeSource::Live`] â€” see module docs. This default is chosen so callers that have not migrated to source-aware sender construction continue to record under the live tag, never silently landing under paper or replay."

**Verdict: CORRECT but with B-1 as the negation.** The default IS conservative for the schema-migration case (existing journals backfill to live). But for the paper-orchestrator case it's the opposite of fail-closed: an `OrderManager` constructed without going through `paper_orchestrator.journal_sender()` writes records that look identical to live data. The comment reads "never silently landing under paper or replay" â€” true; but the converse is not enforced ("never silently landing under live when actually paper" â€” currently violated, see B-1).

### 4. `MockBrokerAdapter` carries a `source` field that is informational only

**Dev claim (Completion Notes):** "The mock adapter now carries a `source: TradeSource` constructed via `MockBrokerAdapter::with_source(behavior, TradeSource)`. This is informational â€” the mock does not write to the journal directly â€” but it surfaces the per-mode tag at the wiring level so a code reviewer can verify by inspection that paper passes `Paper` and replay passes `Replay`."

**Verdict: CORRECT and well-motivated.** The honest version of "stamping at the adapter level" â€” exposing the source as a field that a reviewer can grep for. No `submit_order` plumbing depends on it. No risk of silent miswiring because no production code reads it. Inspection-only.

### 5. Paper P&L excludes fees ("downstream story (live execution wiring)")

**Dev claim (Completion Notes):** "Story 7.4 Task 3.3 mentions fee-aware P&L, but no fee model is wired into the journal write path today; the comment in the readiness report acknowledges this."

**Verdict: ACKNOWLEDGED DEFER.** Acceptable for V1 because the goal is paper / replay parity (both currently exclude fees), not paper / live parity (which requires the live fee path to exist first). When the live fee model lands, both paper and replay should be re-evaluated against it.

### 6. `emit_readiness_summary` requires the caller to supply a separate `Connection`

**Dev claim:** "The connection is parameterised because the orchestrator does not own the journal worker's `Connection` (single-thread pin invariant)."

**Verdict: CORRECT.** The `Connection` is `!Send`. The orchestrator can't own the writer's connection (the worker thread does). Forcing the caller to pass one is the cleanest answer. The integration test `emit_readiness_summary_journals_session_summary` demonstrates the pattern: drop the worker, open a fresh read-only connection, call `emit_readiness_summary`. Slightly inelegant but correct.

### 7. Replay sender re-tags in `attach_journal` (was previously untagged)

**Verdict: CORRECT and CONSISTENT.** Same pattern as paper: `journal.with_source(TradeSource::Replay)`. Replay used to write records that looked like live data (a real defect that 7.4 fixes). Now they're cleanly distinguishable. **But see S-5** for the operator-UX risk of mingling replay and live data in the same database.

### 8. `TradeSource::parse` not implemented as `FromStr`

**Dev claim:** "The natural API name `from_str` collides with `std::str::FromStr::from_str` (clippy `wrong_self_convention`-style warning). Renamed to `parse` to keep the 'unknown variant â‡’ None' semantic without committing to the `FromStr` trait's `Err` machinery."

**Verdict: CORRECT.** `FromStr` requires an `Err` type and the dev wanted `Option`-style failure. Renaming to `parse` is the conventional escape hatch. Documented explicitly in the rustdoc. Fine.
