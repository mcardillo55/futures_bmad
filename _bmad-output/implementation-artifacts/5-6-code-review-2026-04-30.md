## 2. Headline conclusions

- **Hot-path non-blocking guarantee holds.** Every code path that emits an alert does so via `AlertSender::send`, which uses `try_send` against a bounded crossbeam channel. The producer never blocks on disk, channel, or process I/O. The channel cap (1024) is high enough that healthy operation never trips overflow but low enough that a stuck consumer is detected promptly via dropped sends logged at `warn`. **No blocking-on-hot-path violation found.**
- **fsync-per-write is the right call for this workload.** The throughput concern raised in the brief is not load-bearing in practice: V1 breakers are non-resetting (per `BreakerType::category()` — only 7 reset-on-restart breaker types) and the `idempotent_trip_does_not_double_alert` test confirms re-trips don't re-emit. Realistic per-session ceiling is ≈8 alerts (7 breaker types + panic mode), so the per-write `sync_all()` cost is amortised over hours, not microseconds. The concern *would* matter if alerting was ever extended to per-tick events; flagged below as a forward-looking caveat, not a blocker.
- **Script-invocation thread model is sound but has two sharp edges** that are worth surfacing — they are not blockers for V1, both are mitigated by upstream rate limits, and both are documented below as `defer` or `decision-needed`.

Total findings: **8** raised, **1** `decision-needed`, **3** `patch`, **3** `defer`, **1** `dismiss`.

## 3. Findings

### 3.1 decision-needed

#### D1. Per-alert script thread spawn is unbounded above the channel cap [crates/engine/src/risk/alerting.rs:355]

**Sources:** Edge Case Hunter, Blind Hunter (merged)

`AlertManager::process_alert` spawns a *new* OS thread for every script invocation:

```rust
let _ = thread::Builder::new()
    .name("alert-script".to_string())
    .spawn(move || invoke_notification_script(&script, &payload));
```

Each thread sleeps in 50 ms increments up to `script.timeout` (default 10 s) waiting on the child. Under burst load, the AlertManager drains the bounded channel sequentially (one fsync per alert, ≈1–5 ms on SSD), so in the worst case ~1000 alert-script threads can exist simultaneously, each holding a child process and a watchdog stack. In a healthy V1 system this is a non-event (≤8 alerts/session); under a pathological breaker-storm or future expansion of alert sources it becomes an OS-resource pressure point.

Two viable shapes:
1. **Status quo (accept).** Document the upper bound (≤ALERT_CHANNEL_CAPACITY concurrent script invocations) and rely on V1's natural rate limits.
2. **Single dedicated script-runner thread.** Replace the per-alert `thread::spawn` with a second bounded channel (`Receiver<String>`) consumed by one persistent worker. Trade-off: a slow/hung script delays subsequent alerts' script invocation (but **not** alert log persistence — that already happens before the spawn).

This is not a code defect under the AC — script invocation is best-effort and non-blocking by design. The choice is whether to harden the resource ceiling now or defer.

**Recommendation:** Discuss with operator/ops persona — the answer depends on whether the deployed runbook already has a script-process supervisor (e.g., systemd unit limits) or relies on the engine to bound its own behaviour.

### 3.2 patch (unambiguous fixes)

#### P1. Silent failure to spawn alert-script thread [crates/engine/src/risk/alerting.rs:355]

**Source:** Blind Hunter

```rust
let _ = thread::Builder::new()
    .name("alert-script".to_string())
    .spawn(move || invoke_notification_script(&script, &payload));
```

The `let _ =` discards the `io::Result` from `Builder::spawn`. In the (rare) case of OS thread-limit exhaustion or memory pressure, the script invocation is silently skipped — the alert is logged to disk but the operator is never notified, with no `warn!` trail. Every other failure mode in this module logs at `warn` (channel full, channel disconnected, script spawn fail, script non-zero exit, script timeout). Add a `.unwrap_or_else(|e| warn!(...))` for symmetry.

Suggested:

```rust
if let Err(e) = thread::Builder::new()
    .name("alert-script".to_string())
    .spawn(move || invoke_notification_script(&script, &payload))
{
    warn!(target: "alerting", error = %e, "failed to spawn alert-script thread");
}
```

#### P2. Timeout test does not actually assert the child was killed [crates/engine/src/risk/alerting.rs:706 (test)]

**Source:** Acceptance Auditor

`notification_script_timeout_is_enforced` writes a `sleep 30` script with a 200 ms timeout, calls `process_alert`, sleeps 800 ms, then asserts the *test wall-clock* elapsed < 5 s. Because `process_alert` is fire-and-forget, that assertion is satisfied even if the watchdog never fired — the script thread is detached and the assertion only proves `process_alert` returned promptly. Task 7.7 says "script timeout is enforced" — the test should observe a side-effect proving the kill (e.g., have the script `touch` a file *after* the sleep, then assert the file is absent after 800 ms; or capture the script PID and assert it is no longer alive).

Concrete shape:

```sh
#!/bin/sh
sleep 30
touch "$DONE_MARKER"   # never reached if watchdog kills us
```

Then in the test, assert `!done_marker.exists()` after the 800 ms sleep.

#### P3. Redundant clone of Option<PositionSnapshot> [crates/engine/src/risk/circuit_breakers.rs:391]

**Source:** Edge Case Hunter

```rust
let snapshot = position.clone().unwrap_or_else(PositionSnapshot::flat_unknown);
```

`position` is the owned function parameter `Option<PositionSnapshot>`; the `.clone()` is unnecessary — `position.unwrap_or_else(...)` consumes the option directly. Trivial allocator/copy save on the alert path; preferred for clarity.

### 3.3 defer (real but not blocking, or pre-existing)

#### W1. fsync-per-write throughput ceiling is invisible to future maintainers

**Source:** user-supplied spotlight + Edge Case Hunter

The fsync-per-write design is correct for V1 (≤8 alerts/session). It would *not* survive being repurposed for higher-rate event streams — e.g. if a future story routes per-order rejection alerts through `AlertSender`. The module docs in `alerting.rs` document the durability contract but not the implicit rate ceiling. A short comment near `write_alert_line` ("expected event rate: O(1)/session — see Story 5.6 §1; if extended to per-tick or per-order events, switch to a buffered+grouped fsync strategy") would prevent future regressions. **Defer to a docs follow-up; not blocking.**

#### W2. `BreakerKind::PanicMode` is constructible only via the panic alert path

**Source:** Blind Hunter

`BreakerKind` is a `pub enum` with two variants but the `for_panic` constructor is the only path that yields `BreakerKind::PanicMode`. The variant is therefore effectively private-by-construction. This is fine but exposing both variants publicly means a downstream caller could in principle build an `Alert` with `breaker_type = BreakerKind::PanicMode` via a struct literal (the `Alert` fields are `pub`). Not exploited today, no security implication, but a future caller could short-circuit the `for_panic` factory and skip the severity-Critical invariant. **Defer**: tighten by making `Alert` field-private with a builder if and only if a downstream caller proves the need for it. Not justified now.

#### W3. `child.kill()` does not kill the script's children

**Source:** Edge Case Hunter

On Unix, `Child::kill()` sends SIGKILL only to the immediate child. A shell script that exec's `curl` (the canonical V1 example) hands off the file descriptor and will be reaped, but a script that *forks* a long-running subprocess will leak it past the watchdog deadline. Hardening would involve `setsid` + sending the signal to the process group, which is platform-specific and not in `std::process`. **Defer**: V1 expectation per the task list is bash-script-shaped notification — sufficient for `curl-to-Slack`. Document the contract in the alert script's example file when it lands.

### 3.4 dismiss (noise / handled elsewhere)

#### N1. `PositionSnapshot::flat_unknown()` is a "lying" placeholder

**Source:** Blind Hunter

The `flat_unknown` constructor returns a snapshot indistinguishable from a real flat position. A reviewer could argue this conflates "no data available" with "really flat at zero". Dismissed because (a) the docstring is explicit about the semantics, (b) production paths in `anomaly_handler.rs` always supply a real snapshot, and (c) the placeholder is only reachable when `with_alerts` is wired in but the caller invokes `trip_breaker` (the no-context shim) — i.e. tests and incremental wiring. The story explicitly accepts this trade-off via the dual-API `(activate, activate_with_context)` pattern.

## 4. Spotlight: script-invocation thread model

The brief asked for explicit scrutiny on the thread model and the "never blocks the hot path" rule. Walking the path end-to-end:

1. **Producer (`CircuitBreakers::trip_breaker_with_context` or `PanicMode::activate_with_context`)** calls `AlertSender::send(alert)`.
2. `AlertSender::send` invokes `tx.try_send(alert)` against a `crossbeam_channel::bounded(1024)`. `try_send` is **wait-free**: it returns `Full` or `Disconnected` immediately, never parks. ✅ No hot-path block possible.
3. The AlertManager thread (a single dedicated OS thread, named `alert-manager`) calls `recv()` (blocking), `serde_json::to_string`, `OpenOptions::new().append`, `write_all`, `sync_all`, then `thread::Builder::new().spawn(move || invoke_notification_script(...))`. All of this happens on the manager thread, never on a producer thread. ✅
4. The spawned `alert-script` thread writes the JSON to the child's stdin (broken-pipe-tolerant via `let _ =`), then polls `child.try_wait()` in 50 ms increments up to the timeout. On timeout it issues `child.kill()` + `child.wait()`. ✅ Isolated; manager thread is free to drain the next alert.

**Verdict:** the decoupling is clean. The single residual concern (D1) is *resource-bounded* concurrency, not blocking concurrency.

## 5. Spotlight: fsync-per-write throughput

`write_alert_line` performs `open → write_all(json) → write_all('\n') → sync_all → close` per alert. Cost per call is ~one disk fsync (typically 1–5 ms on a consumer SSD, 50–500 µs on an NVMe with battery-backed cache, much higher on spinning rust). For V1's ≤8 alerts/session this is the right trade-off:
- Crash-survival is the explicit AC (Task 2.4: "alerts must not be lost in a crash").
- Reopening per write makes external `logrotate` safe (Task 2.3).
- Throughput per session is bounded by the breaker model itself (manual reset, no auto-clear loop).

The model would *not* hold up at hundreds-per-second sustained — that's a ceiling worth recording in the module docs (W1).
