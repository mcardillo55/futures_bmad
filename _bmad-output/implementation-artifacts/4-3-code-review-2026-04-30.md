## Top-Line Verdict

**APPROVE-WITH-CHANGES.**

The architecture this story delivers is exactly what the spec asked for: a bracket lifecycle state machine with linear forward transitions plus a `Flattening` escape hatch from any non-terminal state, a broker-side `FlattenRetry` orchestrator (3 attempts × 1s default, override-able for tests), and a panic-mode controller that uses an `AtomicBool` for lock-free "is trading enabled?" checks and refuses to expose a `deactivate()` method. The seam between the policy controller and the engine's order-cancel implementation (`OrderCancellation` trait) is correct: cancellation is the engine's job, the trait's contract makes "never cancel a stop" a hard precondition, and the deferred wiring of the actual engine-side iterator (story 4-4 / 4-5 / epic-5) is honest.

`decision_id` (NFR17) is propagated cleanly through `BracketOrder` → every leg `OrderEvent` (via `build_order_event`) → `FlattenOutcome` → `TradeEventRecord` on exit. The test `tp_fill_computes_positive_pnl_and_clears_bracket` asserts `decision_id == 7` is preserved end-to-end. Realised P&L is computed in raw quarter-ticks via saturating arithmetic, and the dollar conversion is correctly deferred to display/reporting layers.

There are however three correctness concerns in `BracketManager::on_entry_fill` that the test suite does not exercise — they will become live defects the moment the engine event loop wires this onto a real Rithmic fill stream. None are catastrophic on the trading-disabled bring-up path, but they should be tightened before story 4-5 reconciliation (which depends on `on_entry_fill` being safe under partial / rejected / out-of-order entry reports).

The five flagged dev deviations are individually defensible. The most important is #3 (`FlattenOutcome::Success { order_id, attempts }` instead of `Success(FillEvent)`) — the dev's argument about avoiding two sources of truth is correct: the actual fill arrives via the FillQueue. This was the right call.

## Verdict on the Five Flagged Deviations

1. **`BracketOrder.entry/take_profit/stop_loss` stayed as `OrderParams` (not `OrderEvent`).** **JUSTIFIED.**
   - The architecture document (`_bmad-output/planning-artifacts/architecture.md:447-451`) shows the canonical struct as `OrderParams` — the spec text was the deviation, not the implementation.
   - `OrderEvent` carries `order_id` and `timestamp`, both of which are routing-layer concerns allocated at submission time, not at construction. Stuffing those into `BracketOrder` at construction would either require a bogus sentinel (`order_id = 0`) or premature allocation.
   - The translation seam `BracketManager::build_order_event` (`bracket.rs:178-202`) is a clean, single-purpose function that takes `&OrderParams` + `order_id` + `decision_id` + `now` and produces an `OrderEvent`. It is called exactly four times (entry, SL, TP, and the pre-allocated `OrderEvent` in `FlattenRetry::flatten_with_retry`). Single source of truth for the bracket's logical shape; routing fields attached at submit time. This is the right factoring.

2. **Renamed constructor to `from_decision`; kept legacy 5-arg `new(...)` for compat.** **JUSTIFIED with one caveat.**
   - Two constructors with different signatures (`new(bracket_id, decision_id, entry: OrderParams, tp: OrderParams, sl: OrderParams)` vs `from_decision(bracket_id, decision_id, symbol_id, side, qty, tp_price, sl_price)`) is normal Rust idiom — `from_decision` is the ergonomic factory; `new` is the validated underlying ctor. Both call into the same validation path (`from_decision` delegates to `new` at line 295).
   - **Caveat:** the doc comment on `new` (`order.rs:213-216`) calls it "the legacy bracket API" but doesn't tell the reader when to prefer it. Future callers will be unsure which to call. Minor — captured below as N-1.

3. **`FlattenRetry::flatten_with_retry` returns `FlattenOutcome::Success { order_id, attempts }` instead of `Success(FillEvent)`.** **JUSTIFIED — strongly.**
   - The spec text mixed up the submission ack and the fill — they are distinct events. The submission ack is "broker accepted the OrderEvent". The fill is what arrives later on the FillQueue when the exchange reports execution. Embedding a `FillEvent` in `Success` would require synthesizing one, and that synthesized event would compete with the real one downstream — exactly the "two sources of truth" pathology the dev called out.
   - The wiring is correctly deferred: the engine event loop drains the FillQueue regardless of how the order was submitted. There is no special "flatten fill" path. This is the architecturally correct shape.
   - Note: if the engine event loop wants to know "the flatten succeeded **and** filled", it must observe both `FlattenOutcome::Success` from the retry orchestrator AND a subsequent `FillEvent` for `order_id`. That's the right factoring.

4. **OCO cancellation relies on Rithmic CME native OCO; no local emulation.** **JUSTIFIED — but the deferral target needs to be re-confirmed.**
   - The spec at Task 2.5 explicitly says: "story 4.5 reconciliation will verify the cancel actually arrived. Local OCO emulation is a follow-up if the live OrderPlant turns out not to support native OCO." This deferral is honest.
   - **However**: the architecture doc only says "TP and SL resting at exchange as OCO" without committing to native vs emulated. The dev's note implies Rithmic CME does support it. **The reviewer cannot verify this from the diff alone** — there is no integration test or recorded session demonstrating that an OCO pair behaves as a pair on the live wire. This is the right deferral target (story 4.5), but the reviewer flags this as a known unknown and recommends that 4.5 carry an explicit test against a recorded session showing the counterpart-cancel arriving. Captured below as N-2.

5. **`OrderCancellation` trait surface is ready; engine-side iterator deferred.** **JUSTIFIED.**
   - The trait is testable: the `MockCancellation` in `panic_mode.rs:171-186` exercises the contract shape. The contract — "never cancel a stop" — is enforced by docstring rather than by the trait signature itself. That's defensible because the trait can't enforce it (it has no notion of stops); only the implementer can. The docstring at `panic_mode.rs:40-46` is unambiguous.
   - The deferral of the engine-side implementation to "subsequent stories that integrate the event loop" is honest. The trait is set up to compose with whatever the engine builds.
   - **Minor concern:** the trait method returns `usize` (count of cancellations issued), not a `Result<usize, _>`. If the engine-side cancel issuer encounters an error on one order out of N, it has no way to surface it. This is captured as N-3.

## Findings Detail

### Should-Fix

**S-1 — `BracketManager::on_entry_fill` does not handle `FillType::Rejected` on the entry leg.**

If the entry leg's fill arrives with `FillType::Rejected { .. }` (i.e., the broker echoed a rejection report for the entry order), `on_entry_fill` currently:
1. Skips the `entry_fill_price` capture (correctly — the `matches!` guard at lines 271-273 only captures Full/Partial),
2. Transitions the bracket to `EntryOnly` regardless of fill type,
3. Submits a stop-loss leg.

A rejected entry means **there is no position**. Submitting an SL when there's no position creates a naked stop — if the stop ever triggers (price walks through it), it creates an unprotected (wrong-direction) position.

This is mitigated in practice today only because:
- Nothing in production code calls `on_entry_fill` yet (the engine event-loop wiring is deferred).
- `OrderManager::apply_fill` (story 4.2) handles the rejection on the order tracker side and never re-routes the rejection into `BracketManager`.

But the function's contract is broken on its own terms. The fix is small: at the top of `on_entry_fill`, branch on `FillType::Rejected` and surface a different `FlattenContext` (e.g., a new `EntryRejected` variant) that tells the caller "no position created — drop the bracket; do NOT engage flatten". The bracket should be removed from `brackets` map and `leg_to_bracket` cleaned for `entry_order_id`.

**Fix:** in `crates/engine/src/order_manager/bracket.rs:253-275`, add a Rejected branch that journals the entry rejection, calls `drop_bracket(bracket_id)` (or inlines the cleanup), and returns a new `FlattenContext::EntryRejected { ... }` variant or a clearly named alternative.
**Location:** `crates/engine/src/order_manager/bracket.rs:253-275`.

**S-2 — `on_entry_fill` ignores `FillType::Partial { remaining }` size, oversizing the SL/TP legs.**

`on_entry_fill` captures `entry.entry_fill_price = Some(fill.fill_price)` for both `Full` and `Partial`, but the SL/TP submission uses `entry.bracket.stop_loss.quantity` and `entry.bracket.take_profit.quantity` — the original bracket-construction quantity. If the entry partials (e.g., 1-of-3 fills, then later a separate fill for the remaining 2), the SL submitted at first partial is sized for 3, not 1. If the second partial arrives, `on_entry_fill` re-runs, the `transition(EntryOnly)` errors (already in EntryOnly, can't re-enter), and the manager warn-logs. The over-sized SL stays.

In a CME futures context, **Market entry fills are typically atomic** — partials on liquid instruments are rare. The architecture doc does not commit either way. But the AC text "given entry fills" doesn't restrict to "fully fills", and the engine event loop will eventually feed every `FillEvent` into this function.

**Fix options:**
- (a) Document at the top of `on_entry_fill` that the function **assumes Market entry fills are atomic** and that partial-fill handling is out of scope for story 4.3, with an explicit `// TODO(story-X)` reference.
- (b) Implement partial entry handling: track cumulative fill size on the bracket, only submit SL/TP after the entry is fully filled, OR submit SL/TP sized for the cumulative position so far and re-size on subsequent partials.

(a) is acceptable for 4.3 if it lands as a hard `debug_assert!` plus a doc note; (b) is the long-term fix and probably belongs in story 4-5 alongside position reconciliation.

**Location:** `crates/engine/src/order_manager/bracket.rs:253-335`.

**S-3 — `engage_flatten` silently swallows the `transition(Flattening)` error.**

At line 585: `let _ = entry.bracket.transition(BracketState::Flattening);`. The `_ =` discards the error. If the bracket is *already* in `Flattening` (because a previous `engage_flatten` call ran and didn't get cleaned up), the transition fails silently, but the function continues and surfaces another `FlattenContext::EntryFlatten { .. }` to the caller. The caller will then engage a **second** `flatten_with_retry`, potentially submitting a second flatten market order with a fresh `order_id`.

This compounds the retry-storm risk: under flaky network where the engine event loop double-fires the entry-fill path (e.g., the broker echoes the same fill twice), this code will engage flatten twice for the same bracket.

**Fix:** treat `transition(Flattening)` returning `Err` as "we are already flattening — do not engage again". Return `None` or a new `FlattenContext::AlreadyFlattening` variant. Do not surface a fresh `EntryFlatten` to the caller.

**Location:** `crates/engine/src/order_manager/bracket.rs:579-616` (specifically line 585).

**S-4 — `FlattenRetry` reuses the same `order_id` across all 3 retries.**

`flatten_with_retry` builds the `OrderEvent` once at lines 128-136 and submits it on every attempt. The `order_id = req.order_id` is stable across attempts. This is a deliberate idempotency design — if attempt 1 reached the exchange but the ack was lost on the network, attempt 2 should be a duplicate that the broker dedupes by `user_tag` / `order_id`. **However**:
- Rithmic OrderPlant's dedup behavior for replayed `RequestNewOrder` with the same `user_tag` is not verified in the diff.
- If the broker does NOT dedupe, attempt 1 + attempt 2 + attempt 3 = three flatten orders submitted; one fills, two end up rejected by the exchange (or worse, all fill, double-flattening into a wrong-direction position).
- The `FlattenOutcome::Success { attempts: 2 }` would correctly report "we got an ack on attempt 2", but the engine has no way to know whether attempt 1 *also* succeeded silently.

**Fix options:**
- (a) Document the assumption: "this code assumes the broker dedupes `RequestNewOrder` by `order_id` (`user_tag`); if the live broker does not, the retry path will produce duplicate orders". Add a `// CRITICAL` comment block at lines 121-126.
- (b) Allocate a fresh `order_id` per attempt and rely on broker-side reconciliation to detect duplicates. This costs more but is robust to a non-deduping broker.

(a) is acceptable for 4.3 and aligns with the deferred reconciliation work in 4.5; (b) is a follow-up if the live OrderPlant turns out not to dedupe.

**Location:** `crates/broker/src/position_flatten.rs:127-194`.

### Nice-to-Have

**N-1 — `BracketOrder::new` doc comment is unclear about when to prefer `new` vs `from_decision`.**

`order.rs:213-216` calls it "the legacy bracket API" but doesn't tell the reader who should use it. New code calling sites will be unsure which constructor is canonical. Recommend: "Prefer `from_decision` for normal callers. Use `new` when the caller has already constructed validated `OrderParams` legs (e.g., signal evaluators with custom params)."

**Location:** `crates/core/src/types/order.rs:213-216`.

**N-2 — OCO native-vs-emulated assumption is unverified in the diff.**

The dev's note that "Rithmic CME supports native OCO" is correct per Rithmic documentation, but there is no integration test or recorded-session test in the diff that proves the engine sees the counterpart cancel arrive. Story 4.5 reconciliation owns this verification. Recommend that 4.5 carry an explicit test against a recorded session showing the counterpart-cancel arriving on TP/SL fill.

**Location:** `crates/engine/src/order_manager/bracket.rs:30-33` (module docstring); future story 4.5.

**N-3 — `OrderCancellation::cancel_entries_and_limits` returns `usize`, not `Result<usize, _>`.**

If the engine-side cancel issuer encounters an error on one order out of N (e.g., the broker rejects the cancel for one order), there is no way to surface a partial failure to `PanicMode::activate`. The activation will record `cancelled_entries_and_limits: N-1` without any indication that one cancel failed. Long-term, the trait method should return a `Result<usize, CancellationError>` or a richer outcome struct. Not blocking for 4.3 since the implementation is deferred.

**Location:** `crates/engine/src/risk/panic_mode.rs:40-46`.

**N-4 — `FlattenRetry` does not enforce a precondition that a position exists.**

`flatten_with_retry` blindly submits a market order on the requested side. If the engine event loop engages flatten without a position (e.g., a stale bracket entry that was already exited), the broker will create one in the wrong direction. The orchestrator can't verify position state — that's the engine's job — but a debug assertion or a docstring hard-warning would be useful.

**Location:** `crates/broker/src/position_flatten.rs:121-130`.

**N-5 — `BracketManager::on_stop_confirmed` does not handle `transition(EntryAndStop)` failure path beyond a warn.**

At lines 351-361: if the bracket isn't in `EntryOnly` (e.g., a stop confirmation arrived twice), the function warn-logs and bails. That's fine for an idempotent call. But the same path — an out-of-order stop confirmation arriving when the bracket has already been flattened — would also hit this branch. There is no terminal guard distinguishing "I'm already past EntryAndStop because we're Flattening" from "I'm getting a duplicate ack". A `match self.state` in `transition` returning the offending arc would let the caller distinguish — that's already what `BracketStateError` carries, but the caller currently just `?err`s into the warn log without acting. Consider differentiating in the caller.

**Location:** `crates/engine/src/order_manager/bracket.rs:344-361`.
